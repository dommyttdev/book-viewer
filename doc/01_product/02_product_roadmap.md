# プロダクトロードマップ

## 目的

このドキュメントは、自炊本閲覧Webアプリケーションの開発段階ごとの到達目標を整理するためのロードマップである。

ロードマップは固定された納期計画ではなく、プロダクト価値、技術的リスク、運用準備の優先順位をそろえるための見通しとして扱う。詳細なユーザーストーリー、受入条件、API仕様、画面仕様、ADRは、各バックログおよび設計ドキュメントで継続的に具体化する。

## 前提

- MVPでは、自炊本を登録し、変換し、探し、読むための最小限の流れを成立させる。
- 一般ユーザは書籍を保持せず、書籍アップロードも行わない。
- 書籍アップロードは管理ユーザのみ可能とする。
- PostgreSQLを正本とし、Elasticsearchは検索用の再構築可能な派生データとして扱う。
- アップロードされた原本ファイルは保存し続ける。
- 画像変換は変換ワーカーの非同期ジョブとして実行する。
- 初期の本番運用は、単一Linuxホスト上のDocker Compose構成を前提とする。
- バックアップは行わない方針とし、リスクと許容範囲は運用ドキュメントで明記する。

## ロードマップ概要

| 区分 | 主な到達目標 | 判断基準 |
| --- | --- | --- |
| MVP | 自炊本を登録、変換、検索、閲覧できる最小構成を成立させる。 | 管理ユーザがアップロードした本を、一般ユーザが検索してビューアで読める。 |
| Beta | MVPの運用性、管理性、閲覧体験を強化し、継続利用できる状態に近づける。 | 失敗時の確認、再実行、絞り込み、主要なユーザ管理が実用に耐える。 |
| v1.0 | 単一ホスト運用で安定して使える正式版として、品質、権限、運用手順を整える。 | 主要機能、権限、テスト、Runbookがそろい、既知リスクが文書化されている。 |
| 将来対応 | 利用状況に応じて、検索、取り込み、閲覧、運用の拡張を検討する。 | MVPからv1.0までの利用実績をもとに、投資価値が確認できている。 |

## MVP

MVPでは、機能の広さよりも「登録した自炊本を安定して探して読めること」を優先する。

### プロダクト機能

- 一般ユーザの会員登録、メール認証、ログイン、ログアウトを提供する。
- 一般ユーザのパスワードリセットを提供する。
- 管理ユーザが管理画面へログインできるようにする。
- 管理ユーザがzip / rar / 7zip形式の自炊本アーカイブをアップロードできるようにする。
- アップロード後に原本ファイルを保存し、変換ジョブを作成する。
- 変換ワーカーでアーカイブ展開、WebP変換、サムネイル生成を実行する。
- WebPの既定品質値は80とし、application propertiesで設定可能にする。
- 管理ユーザが変換ジョブの状態を確認できるようにする。
- 管理ユーザがタイトル、著者、タグ、シリーズ、種別の基本メタ情報を編集できるようにする。
- 一般ユーザが本一覧から書籍を探せるようにする。
- 一般ユーザがタイトル、著者、タグ、シリーズで基本検索できるようにする。
- 一般ユーザがビューアでページを表示し、次ページ、前ページへ移動できるようにする。
- 一般ユーザの読みかけ位置を保存する。
- 一般ユーザがお気に入り登録、解除、お気に入り一覧からの再閲覧をできるようにする。

### 技術と設計

- Next.jsフロントエンド、Spring BootバックエンドAPI、Spring Boot変換ワーカーの基本構成を用意する。
- PostgreSQLにメタ情報、ユーザ、権限、ジョブ状態を保存する。
- Elasticsearchとanalysis-kuromojiを利用し、タイトル、著者、タグ、シリーズの基本検索インデックスを作成する。
- ElasticsearchはPostgreSQLから再構築可能な派生データとして扱う。
- rar / 7zipの展開は、変換ワーカーから7-Zip for Linux コンソール版を外部プロセスとして呼び出す。
- 変換ジョブは専用キューで扱い、API処理と重い画像処理を分離する。
- API、ワーカー、ミドルウェアを将来分離できるよう、モジュール境界を明確にする。

### 運用と品質

- 単一Linuxホスト上のDocker Composeで主要コンポーネントを起動できるようにする。
- 変換ジョブの成功、失敗、処理中状態を確認できるようにする。
- Elasticsearchの全件再インデックスと書籍単位再インデックスの見通しを用意する。
- 管理ユーザのみが書籍アップロードできる権限制御を実装する。
- MVPの主要フローについて受入条件と基本テストを用意する。

### MVP完了の目安

- 管理ユーザが自炊本アーカイブをアップロードできる。
- 変換ジョブが作成され、WebP画像とサムネイルが生成される。
- 一般ユーザが本一覧または検索から本を見つけられる。
- 一般ユーザがビューアで本を読み、読みかけ位置から再開できる。
- 一般ユーザが管理操作を実行できない。
- 主要な異常系が確認でき、失敗状態が追跡できる。

## Beta

Betaでは、MVPで成立した基本フローを、日常利用しやすい状態へ改善する。

### プロダクト機能

- 一般ユーザが会員情報を編集できるようにする。
- 一般ユーザが退会できるようにする。
- 管理ユーザがアップロード失敗理由を確認できるようにする。
- 管理ユーザが失敗した変換ジョブを再実行できるようにする。
- 管理ユーザが変換状態で絞り込みできるようにする。
- 管理ユーザが変換結果のページ順を確認できるようにする。
- 管理ユーザがタグや著者の表記揺れを整理できるようにする。
- 管理ユーザがシリーズ概要を編集できるようにする。
- 一般ユーザが種別や複数条件で絞り込みできるようにする。
- 一般ユーザが検索結果を並び替えできるようにする。
- 一般ユーザが見開き表示、ページ番号指定、拡大縮小を利用できるようにする。
- キーボード操作やタッチ操作による閲覧体験を改善する。
- 管理ユーザが一般ユーザの停止、管理ユーザ管理、ロール管理を実用的に行えるようにする。

### 技術と設計

- 変換失敗理由、再実行、キャンセル、タイムアウトの仕様を具体化する。
- 画像ファイル判定、ページ順序、表紙決定の仕様を具体化する。
- 検索インデックス更新失敗時の再試行方式を具体化する。
- 権限マトリクスを作成し、管理操作の可否を明確にする。
- UIフローと画面メモを更新し、主要画面の遷移と操作をそろえる。

### 運用と品質

- 変換ジョブ滞留、変換失敗数、画像変換処理時間を確認できるようにする。
- 障害ログの記録形式を用意する。
- 主要機能の受入テストと回帰テストを増やす。
- Elasticsearch再インデックス手順をRunbookに反映する。

## v1.0

v1.0では、単一Linuxホスト上で安定運用できる正式版として、機能、品質、運用手順を整える。

### プロダクト機能

- 自炊本管理、画像変換、検索、閲覧、お気に入り、アカウント管理、管理機能の主要ユーザーストーリーを完了する。
- 管理ユーザが書籍削除と変換ジョブ再実行を安全に実行できるようにする。
- 一般ユーザがスマートフォンでも読みやすいビューア体験を得られるようにする。
- お気に入り、閲覧履歴、検索結果からの再閲覧を安定して利用できるようにする。

### 技術と設計

- API契約、データモデル、検索設計、ファイル保存設計、画像変換設計、権限設計を整備する。
- Spring Boot採用、モジュラーモノリス採用、PostgreSQL採用、Elasticsearch採用、WebP変換、アーカイブ展開、非同期変換ジョブ方式のADRを整備する。
- PostgreSQLとElasticsearchの整合性回復手順を確立する。
- 変換ワーカーの同時実行数10、1ジョブ30分タイムアウトを既定値として扱い、設定変更できるようにする。

### 運用と品質

- Docker Composeによる起動、停止、再起動、ログ確認、障害時確認手順をRunbookにまとめる。
- バックアップなし運用のリスクと許容範囲を文書化する。
- 監視方針を整備する。
- リリースノート形式と障害ログ形式を用意する。
- MVPからv1.0までの主要機能に対する受入テスト、回帰テスト、基本的な異常系テストを整備する。
- Definition of Doneに沿って、関連ドキュメント更新、権限確認、ログ、エラー処理を確認する。

## 将来対応

将来対応は、MVPからv1.0までの利用実績をもとに必要性を判断する。初期段階では、実装を前提にしすぎず、設計上の拡張余地を残す。

### 機能拡張候補

- 一般ユーザによる書籍アップロード。
- 一般ユーザ同士の書籍共有。
- 複数冊のまとめアップロード。
- 外部ストレージからの取り込み。
- 外部書誌データベースからのメタ情報補完。
- OCRによる本文や目次の抽出。
- 本文全文検索。
- 高度な検索ランキング調整。
- お気に入りの共有、メモ、分類。
- 読書進捗の詳細分析。
- オフライン閲覧。
- ネイティブモバイルアプリ。

### 技術拡張候補

- API、変換ワーカー、ミドルウェアの別ホスト分離。
- 変換ワーカーの水平スケール。
- 検索インデックス設計の高度化。
- 本ごとの変換設定。
- 一部ページのみの再変換。
- バックアップ方針の見直し。
- クラウドストレージやマネージドサービスの利用検討。

## ロードマップの更新方針

- ユーザーストーリー、受入条件、設計、ADRが具体化されたら、このロードマップの区分も見直す。
- MVP、Beta、v1.0の境界を変更する場合は、理由を関連ドキュメントに残す。
- 将来対応候補を実装対象に移す場合は、先にプロダクト価値、運用負荷、セキュリティ、データモデルへの影響を確認する。
- 重要な仕様判断は、このロードマップだけでなくADRまたは設計ドキュメントに記録する。

# ファイル保存設計初版

## 目的

このドキュメントは、自炊本閲覧Webアプリケーションにおける原本ファイル、変換済みWebP、サムネイルの保存方針を定義する。

保存領域の物理構成、命名規則、削除時の扱い、バックアップなし運用との関係を明確にし、バックエンドAPI、変換ワーカー、PostgreSQLの責務を分ける。

## 前提

- PostgreSQLをファイル管理情報の正本とする。
- ファイル実体は書籍ファイル保存領域に保存する。
- 原本ファイルは通常運用では保存し続ける。
- 変換済みWebPとサムネイルは原本ファイルと変換条件から再生成可能な派生ファイルとして扱う。
- ElasticsearchはPostgreSQLから再構築可能な派生データとして扱う。
- バックアップは行わない。PostgreSQL、Elasticsearch、原本ファイル、変換済みWebP、サムネイルはいずれもバックアップ対象外とする。
- 内部物理パス、コンテナパス、ホストパスはAPIレスポンスへ直接露出しない。

## 保存対象

| 種別 | 位置づけ | 保存主体 | 主な用途 | 再生成 |
| --- | --- | --- | --- | --- |
| 原本ファイル | 永続ファイル | バックエンドAPI | 再変換、障害時確認、将来の変換条件変更 | 不可 |
| 変換済みWebP | 派生ファイル | 変換ワーカー | ビューアでのページ表示 | 原本から再生成可能 |
| サムネイル | 派生ファイル | 変換ワーカー | 一覧、検索結果、詳細画面表示 | 原本またはWebPから再生成可能 |

PostgreSQLの`book_file`には、ファイル種別、保存キー、元ファイル名、Content-Type、ファイルサイズ、チェックサム、作成日時、削除日時を記録する。

## 保存領域

書籍ファイル保存領域のルートは環境ごとに設定する。アプリケーションコードにはホスト固有の絶対パスをハードコードしない。

初期の設定項目候補は次のとおり。

| 設定項目 | 用途 |
| --- | --- |
| `storage.books.root` | 書籍ファイル保存領域のルート |
| `storage.books.originals-dir` | 原本ファイル保存ディレクトリ名 |
| `storage.books.pages-dir` | 変換済みWebP保存ディレクトリ名 |
| `storage.books.thumbnails-dir` | サムネイル保存ディレクトリ名 |
| `storage.books.trash-dir` | 削除待ちファイルの退避ディレクトリ名 |

## ディレクトリ命名規則

保存領域は、書籍IDを基準に分離する。

```text
{storage_root}/
  books/
    {book_id}/
      originals/
        {original_file_id}.{extension}
      pages/
        {page_number}.webp
      thumbnails/
        cover.webp
        list.webp
  trash/
    books/
      {book_id}/
```

命名規則は次のとおり。

| 要素 | 方針 |
| --- | --- |
| `book_id` | PostgreSQL上の書籍IDを使う。実装時にUUIDまたは推測困難な公開IDを採用する場合は、その内部IDを保存キーに使う。 |
| `originals` | アップロードされた原本アーカイブを保存する。 |
| `pages` | 閲覧用の変換済みWebPを保存する。 |
| `thumbnails` | 表紙、一覧、検索結果で使うサムネイルを保存する。 |
| `trash` | 書籍削除時に物理削除前の退避先として使う。 |

書籍ごとのディレクトリ分離により、生成物の混在、削除時の対象誤り、再変換時の上書き範囲の拡大を防ぐ。

## ファイル命名規則

外部から受け取ったファイル名やアーカイブ内エントリ名は、そのまま物理ファイル名に使わない。

| 種別 | ファイル名 | 備考 |
| --- | --- | --- |
| 原本ファイル | `{original_file_id}.{extension}` | 拡張子は検証済みのアップロード形式から決定する。元ファイル名はPostgreSQLの`original_file_name`に保持する。 |
| 変換済みWebP | `{page_number}.webp` | `page_number`は1始まりのページ順。ゼロ埋め桁数は実装時に最大ページ数を踏まえて決める。 |
| 表紙サムネイル | `cover.webp` | 一覧や詳細で代表画像として使う。 |
| 一覧サムネイル | `list.webp` | 画面上の用途に応じて必要になった場合に生成する。初期実装では`cover.webp`と統合してもよい。 |

`storage_key`は、物理ルートからの相対キーとして扱う。APIは`storage_key`をそのままクライアントへ返さず、画像取得APIやサムネイル取得APIのリソースIDを返す。

## アップロード時の保存処理

1. バックエンドAPIは、管理ユーザの認証、書籍アップロード権限、入力値、アップロードファイルを検証する。
2. ファイル名、Content-Type、拡張子、サイズ方針、チェックサムを確認する。
3. 原本ファイルを一時保存し、検証に通った場合だけ書籍ファイル保存領域へ移動する。
4. `book`、`book_file`、`conversion_job`をPostgreSQLへ登録する。
5. 変換ジョブを専用キューへ投入する。

PostgreSQL更新に失敗した場合、保存済み原本ファイルは未参照ファイルとして検出できるようにする。実装時は、DB更新前の一時保存領域と確定保存領域を分ける。

## 変換時の保存処理

1. 変換ワーカーはPostgreSQLからジョブ、書籍、原本ファイル管理情報を取得する。
2. 原本ファイルを読み取り、ジョブごとの専用作業ディレクトリへ展開する。
3. 展開結果と画像ファイルを検証し、ページ順を決定する。
4. 変換済みWebPを`pages/`へ保存する。
5. サムネイルを`thumbnails/`へ保存する。
6. `book_page`と生成物の`book_file`をPostgreSQLへ登録する。

生成物保存後にPostgreSQL更新が失敗した場合は、PostgreSQLを正として再変換または生成物クリーンアップを行う。

## 削除時の扱い

書籍削除は管理ユーザのみが実行できる。

初版では、書籍削除時にPostgreSQL上の`book.deleted_at`を設定し、通常の一覧、検索、閲覧対象から除外する。物理ファイルの削除は、論理削除と同一トランザクションでは行わず、削除ジョブまたは管理コマンドで実行する。

| 対象 | 削除方針 |
| --- | --- |
| 原本ファイル | 通常運用では保存し続ける。書籍削除時は削除ジョブの対象にできるが、即時削除しない。 |
| 変換済みWebP | 書籍削除後は閲覧対象外にし、削除ジョブで物理削除する。 |
| サムネイル | 書籍削除後は表示対象外にし、削除ジョブで物理削除する。 |
| Elasticsearch | PostgreSQLを正として該当ドキュメントを削除し、必要に応じて再インデックスで回復する。 |

物理削除ジョブは、削除対象を一度`trash/`へ退避し、退避成功後に削除済みとして記録する方式を初期候補とする。退避や削除に失敗した場合は、管理者が確認できるように失敗理由を記録する。

## 再変換時の扱い

再変換では原本ファイルを再利用する。

既存の`pages/`と`thumbnails/`は、再変換用の一時出力先へ新しい生成物を作成し、PostgreSQL更新が成功してから差し替える。途中失敗時に閲覧可能な既存生成物を壊さないことを優先する。

## 整合性回復

ファイル保存領域とPostgreSQLの不整合は、PostgreSQLを基準に確認する。

| 不整合 | 回復方針 |
| --- | --- |
| PostgreSQLに原本管理情報があるが物理ファイルがない | 変換不可としてジョブを失敗扱いにし、運用者が原本消失を確認する。 |
| 原本ファイルはあるがPostgreSQLに管理情報がない | 未参照ファイルとして検出し、削除候補にする。 |
| 変換済みWebPがない | 原本ファイルがあれば再変換する。 |
| サムネイルがない | 原本または変換済みWebPから再生成する。 |
| PostgreSQLのページ情報と生成物が一致しない | PostgreSQLを正として再変換、再登録、または生成物クリーンアップを行う。 |

## セキュリティ

- アップロードファイル名、アーカイブ内パス、展開先パス、生成ファイル名は信頼しない。
- `../`、絶対パス、ドライブレター、シンボリックリンクによる保存領域外への脱出を拒否する。
- 物理パス、コンテナパス、`storage_key`をAPIレスポンスへ直接露出しない。
- 原本ファイル、変換済みWebP、サムネイルは認可済みAPI経由で参照する。
- パス、元ファイル名、ジョブ失敗情報をログに出す場合は、秘密情報や過度な個人情報を含めない。

## バックアップなし運用

バックアップは行わない。

そのため、原本ファイルを失うとWebPとサムネイルは再生成できなくなる。変換済みWebPとサムネイルは派生ファイルとして扱うが、原本ファイルは再生成できないため、通常運用では保存し続ける。

バックアップなし運用の詳細なリスクと許容範囲は、運用ドキュメントのバックアップなし方針で整理する。

## 後続設計で詳細化する事項

- 物理削除ジョブまたは管理コマンドのAPI、権限、実行タイミング。
- 再変換時の一時出力先、差し替え、ロールバック方式。
- サムネイルのサイズ、用途別生成数、命名。
- 画像変換設計におけるページ順序、対応画像形式、失敗時の部分生成物の扱い。
- Runbookにおける未参照ファイル検出、削除失敗時の確認手順。

## 更新方針

保存先、命名規則、削除タイミング、バックアップなし運用、API経由の配信方式が変わった場合は、このドキュメントを更新する。

実装で物理ディレクトリ構成や`storage_key`の形式を具体化した場合は、設計との差分を確認し、必要に応じてこの初版を改訂する。

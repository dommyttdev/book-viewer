# API設計方針初版

## 目的

このドキュメントは、Spring BootバックエンドAPIの契約を作成するための共通方針を定義する。

API契約は、Next.jsフロントエンド、Spring BootバックエンドAPI、変換ワーカー、PostgreSQL、Elasticsearch、ファイル保存領域の責務境界を明確にするために利用する。実装時のDTO名、Controller名、細かなバリデーション制約は、各API契約をもとに具体化する。

## 前提

- PostgreSQLを正本データストアとする。
- Elasticsearchは検索用の派生データとし、APIレスポンスの最終的な表示可否は必要に応じてPostgreSQLの正本で確認する。
- 原本ファイル、変換済みWebP、サムネイルの物理パスや`storage_key`はAPIレスポンスへ直接露出しない。
- APIモデル、永続化エンティティ、ドメインモデルは概念的に分離する。
- 業務ロジックはControllerへ置かず、アプリケーションサービスまたはドメインサービスへ寄せる。
- 認証、認可、入力検証はサーバ側で必ず実施する。

## URL命名

URLはリソース中心に命名し、バージョンを含める。

| 項目 | 方針 |
| --- | --- |
| ベースパス | `/api/v1` |
| リソース名 | 複数形の名詞を使う。例: `/books`, `/conversion-jobs` |
| 管理系API | `/api/v1/admin/...` 配下に置く。 |
| 認証系API | `/api/v1/auth/...` 配下に置く。 |
| ファイル配信 | 物理パスではなく、書籍IDやページIDを使う。 |
| アクション | リソース状態遷移は必要最小限の動詞サブリソースで表す。例: `/conversion-jobs/{id}/retry` |

IDは実装時に、内部DB IDをそのまま公開するか、UUIDなどの公開識別子を使うかを決める。初版の契約では`{bookId}`などの公開リソースIDとして扱う。

## HTTPメソッド

| メソッド | 用途 |
| --- | --- |
| `GET` | 取得。副作用を持たせない。 |
| `POST` | 作成、アップロード、ログイン、明示的な状態遷移。 |
| `PATCH` | 部分更新。 |
| `PUT` | 全体置換が自然な場合のみ使用する。初版では原則`PATCH`を優先する。 |
| `DELETE` | 削除または解除。業務上は論理削除になる場合がある。 |

`GET`で状態変更を行わない。アップロードや再実行など副作用のある操作は`POST`を使う。

## 認証と認可

初版では、HTTP onlyかつSecure属性付きCookieによるサーバ側セッションを第一候補とする。JWTは、フロントエンドとAPIの分離要件が強くなった場合に再検討する。

| 利用者 | 主なAPI |
| --- | --- |
| 未ログイン利用者 | 会員登録、メール確認、ログイン、パスワードリセット要求 |
| 一般ユーザ | 本一覧、検索、閲覧、お気に入り、閲覧履歴、会員情報 |
| 管理ユーザ | 書籍管理、変換ジョブ、一般ユーザ管理、管理ユーザ管理、ロール管理 |

管理APIは、管理ユーザとして認証済みであり、必要な権限を持つ場合だけ許可する。一般ユーザAPIは、自分の会員情報、お気に入り、閲覧履歴のみ操作できる。

## ページング

一覧APIと検索APIは必ずページングする。

| パラメータ | 方針 |
| --- | --- |
| `page` | 1始まりのページ番号。既定値は`1`。 |
| `size` | 1ページ件数。既定値は`20`、最大値は`100`を初期候補とする。 |
| `sort` | 許可済みのソートキーのみ受け付ける。 |

レスポンスには、現在ページ、件数、総件数、総ページ数を含める。総件数取得が高コストになる場合は、対象APIの契約で代替方針を定義する。

```json
{
  "items": [],
  "page": {
    "number": 1,
    "size": 20,
    "totalItems": 0,
    "totalPages": 0
  }
}
```

## エラーレスポンス

エラーは共通形式で返す。

```json
{
  "error": {
    "code": "validation_failed",
    "message": "入力値を確認してください。",
    "requestId": "req_...",
    "details": []
  }
}
```

| HTTPステータス | 用途 |
| --- | --- |
| `400 Bad Request` | リクエスト形式不正、型不一致、不正な条件 |
| `401 Unauthorized` | 未認証 |
| `403 Forbidden` | 認証済みだが権限不足 |
| `404 Not Found` | 対象なし、または存在を隠すべき対象 |
| `409 Conflict` | 重複、状態遷移不整合、楽観ロック競合 |
| `413 Payload Too Large` | 実装または環境上のアップロード制限に抵触した場合 |
| `415 Unsupported Media Type` | 非対応Content-Type |
| `422 Unprocessable Entity` | 業務ルール上処理できない入力 |
| `429 Too Many Requests` | ログイン失敗、メール再送などの制限 |
| `500 Internal Server Error` | 想定外のサーバエラー |
| `503 Service Unavailable` | 依存コンポーネント利用不可 |

認証失敗、権限不足、存在しないリソースの応答は、内部状態を推測されにくい形にする。

## バリデーションエラー形式

入力項目ごとのエラーは`details`へ配列で返す。

```json
{
  "error": {
    "code": "validation_failed",
    "message": "入力値を確認してください。",
    "requestId": "req_...",
    "details": [
      {
        "field": "title",
        "code": "required",
        "message": "タイトルは必須です。"
      }
    ]
  }
}
```

`field`にはリクエスト上の項目名を使い、永続化エンティティや内部カラム名を不用意に露出しない。

## 日時と状態

- 日時はISO 8601形式で返す。
- タイムゾーンの扱いは実装時に一貫させる。
- 状態値は文字列の列挙値として返す。
- クライアントは未知の状態値を壊れず扱えるようにする。

## セキュリティ

- クライアントからElasticsearch Query DSLを直接受け取らない。
- アップロードファイル名、アーカイブ内パス、画像取得対象IDを信頼しない。
- パスワード、トークン、セッションID、内部物理パス、不要な個人情報をログやレスポンスに含めない。
- CSRF対策はCookieベース認証の採用時に必須とする。
- 画像配信APIも認可済みAPIとして扱い、内部ファイルを直接公開しない。

## 更新方針

認証方式、エラー形式、ページング形式、URL命名、公開ID方針が変わった場合は、このドキュメントと各API契約を更新する。

# 権限設計初版

## 目的

このドキュメントは、自炊本閲覧Webアプリケーションにおける認証、利用者種別、管理ロール、ロール別操作可否、退会時のデータ扱いを整理する。

権限確認はフロントエンド表示制御だけに依存せず、Spring BootバックエンドAPIのサーバ側で必ず実施する。PostgreSQLを利用者、ロール、権限の正本とし、Elasticsearchやファイル保存領域に対する参照もAPIで権限確認した結果に従う。

## 前提

- 一般ユーザは書籍を保持しない。
- 一般ユーザは書籍アップロード、書籍メタ情報管理、書籍削除、変換ジョブ操作を行わない。
- 書籍アップロードは管理ユーザのみが実行できる。
- 管理ユーザは管理ロールにより操作可能範囲を制御する。
- 一般ユーザ同士で書籍を共有する概念は持たない。
- 管理者による一般ユーザ所有書籍の閲覧や削除は、一般ユーザが書籍を保持しないため対象外とする。
- 管理ユーザの書籍閲覧権限は、管理対象書籍の内容確認・品質確認のために使う。詳細な権限コードの意味は[権限マトリクス](02_permission_matrix.md)を正本とする。
- APIモデル、永続化エンティティ、ドメインモデルは概念的に分離する。

## 利用者種別

| 種別 | 概要 | 主な操作 |
| --- | --- | --- |
| 未ログイン利用者 | 認証前の利用者。 | 会員登録、ログイン、パスワードリセット要求 |
| 一般ユーザ | 登録済みの本を読む利用者。 | 本一覧表示、検索、閲覧、お気に入り管理、閲覧履歴利用、会員情報編集、退会 |
| 管理ユーザ | 書籍、利用者、ロール、変換ジョブを管理する利用者。 | 書籍アップロード、メタ情報管理、変換ジョブ確認、再実行、ユーザ管理、ロール管理 |
| 運用者 | システム運用を行う担当者。 | Docker Compose運用、ログ確認、再インデックス、障害対応 |

運用者は初版ではアプリケーション内ロールではなく、ホストや運用手順にアクセスできる担当者として扱う。アプリケーション上で運用操作を提供する場合は、管理ユーザの`super_admin`または専用権限として追加する。

## 認証方式

初版では、一般ユーザと管理ユーザのどちらもメールアドレスとパスワードを使った認証を前提とする。

| 項目 | 方針 |
| --- | --- |
| 会員登録時のメール認証 | 行う。メール確認が完了するまで通常利用を許可しない。 |
| ログイン時の2段階認証 | メールを活用する。初版ではワンタイムコードまたは確認リンクを候補とし、有効期限、使い捨て、再送制限を必須設計項目とする。 |
| パスワード保存 | 平文保存しない。安全なパスワードハッシュを保存する。 |
| パスワードリセット | 提供する。リセットトークンは有効期限、使い捨て、漏えい時の影響範囲を考慮する。 |
| セッション管理 | 初版ではHTTP onlyかつSecure属性付きCookieによるサーバ側セッションを第一候補とする。 |
| JWT | フロントエンドとAPIの分離要件が強くなった場合に検討する。初版では必須にしない。 |
| ログイン失敗制限 | メールアドレス、IPアドレス、アカウント単位のレート制限または一時ロックを実装候補とする。 |
| 退会済みユーザ | ログインを許可しない。既存セッションは退会時に無効化する。 |

認証失敗と権限不足の応答は、存在しないリソースや内部状態を推測されにくい形にする。詳細なエラーレスポンス形式はAPI契約で定義する。

### 認証トークンとセッションの扱い

| 対象 | 方針 |
| --- | --- |
| メール確認トークン | 有効期限付き、使い捨てとし、確認完了後は再利用できない。 |
| ログイン2段階認証チャレンジ | パスワード認証成功後に発行し、短い有効期限、試行回数制限、再送制限を持たせる。 |
| パスワードリセットトークン | 有効期限付き、使い捨てとし、要求時の応答はメールアドレス登録有無を推測されにくくする。 |
| セッション | サーバ側で失効可能にし、ログアウト、退会、重要な認証状態変更時に無効化できるようにする。 |
| ログ出力 | パスワード、トークン、セッションID、ワンタイムコードを記録しない。 |

認証トークンとセッションの正本は[データモデル初版](../04_data_model.md)の`email_verification_token`、`login_challenge`、`password_reset_token`、`session`に置く。APIで扱う`token`、`challengeId`、Cookie内のセッションIDは、DBへ平文保存せず、用途別のハッシュ値、期限、使用済み日時、失効日時、試行回数、再送回数と紐づけて検証する。

| 失効契機 | 対象 |
| --- | --- |
| メール確認完了 | 使用した`email_verification_token`を使用済みにする。同一目的の未使用トークンは失効する。 |
| メール確認の試行上限超過 | 対象の`email_verification_token`を失効または拒否状態として扱う。 |
| ログイン2段階認証成功 | 使用した`login_challenge`を使用済みにし、`session`を作成する。 |
| ログイン2段階認証の試行上限超過 | 対象の`login_challenge`を失効または拒否状態として扱う。 |
| パスワード変更 / リセット完了 | 対象主体の未使用`password_reset_token`と既存`session`を失効する。 |
| パスワードリセットの試行上限超過 | 対象の`password_reset_token`を失効または拒否状態として扱う。 |
| メールアドレス変更 | 対象主体の未使用`email_verification_token`と既存`session`を失効する。 |
| ログアウト | 現在の`session`を失効する。 |
| 一般ユーザ退会 / 停止 | 対象一般ユーザの`session`、未使用トークン、未使用チャレンジを失効する。 |
| 管理ユーザ停止 / 削除 / 重要なロール変更 | 対象管理ユーザの管理`session`、未使用チャレンジを失効する。 |

サーバ側セッションを第一候補にする理由は、単一Linuxホスト上のDocker Compose構成ではサーバ側失効を扱いやすく、退会、ログアウト、アカウント停止時に即時無効化しやすいためである。JWTを採用する場合は、短い有効期限、リフレッシュトークン、失効リスト、Cookie保存時のCSRF対策を追加で検討する。

## 一般ユーザの権限

一般ユーザは閲覧を中心とする権限のみを持つ。

| 操作 | 可否 | 備考 |
| --- | --- | --- |
| 本一覧表示 | 可 | 表示可能な有効書籍のみ対象とする。 |
| 本検索 | 可 | Elasticsearch検索結果もAPI側で表示可能状態を確認する。 |
| 本閲覧 | 可 | 変換済みで公開可能なページのみ閲覧できる。 |
| お気に入り登録 / 解除 | 可 | 自分の一般ユーザアカウントに紐づくお気に入りのみ操作できる。 |
| 閲覧履歴保存 | 可 | 自分の閲覧履歴のみ保存、更新できる。 |
| 会員情報編集 | 可 | 自分の一般ユーザ情報のみ操作できる。 |
| 退会 | 可 | 自分の一般ユーザアカウントのみ退会できる。 |
| 書籍アップロード | 不可 | 書籍アップロードは管理ユーザのみ。 |
| 書籍メタ情報管理 | 不可 | 管理ユーザのみ。 |
| 書籍削除 | 不可 | 管理ユーザのみ。 |
| 変換ジョブ操作 | 不可 | 管理ユーザのみ。 |
| ユーザ管理 / ロール管理 | 不可 | 管理ユーザのみ。 |

一般ユーザは書籍を保持しないため、「自分の書籍のみ操作できるか」という所有者ベースの書籍操作権限は設けない。

## 管理ユーザと管理ロール

管理ユーザには1つ以上の管理ロールを割り当てる。複数ロールを持つ場合は、許可の和集合をその管理ユーザの操作可能範囲とする。

| ロール | 目的 | 主な権限 |
| --- | --- | --- |
| `super_admin` | システム全体の管理。 | 管理ユーザ管理、ロール設定、一般ユーザ管理、書籍管理、変換ジョブ操作、再インデックス操作 |
| `admin` | 日常的な管理業務。 | 一般ユーザ管理、書籍アップロード、書籍メタ情報管理、書籍削除、変換ジョブ確認、再実行 |
| `operator` | 取込、変換、運用寄りの作業。 | 書籍アップロード、変換ジョブ確認、再実行、再インデックス操作 |
| `viewer` | 管理画面の参照。 | 書籍、一般ユーザ、変換ジョブ状態の参照 |

`super_admin`は初期管理ユーザの作成、ロール変更、管理ユーザ停止など影響の大きい操作を担当する。`viewer`は参照専用とし、更新操作を許可しない。

管理ユーザの書籍閲覧は、登録済み書籍の変換結果、ページ順、表示品質、メタ情報整合性を確認するための管理操作として扱う。一般ユーザが保持する書籍を管理者が閲覧する権限ではなく、初期設計では一般ユーザは書籍を保持しない。詳細は[権限マトリクス](02_permission_matrix.md)の `book.view` 定義に従う。

| 書籍一覧 / 詳細参照 | 可 | 可 | 可 | 可 | 可 |
| 書籍アップロード | 可 | 可 | 可 | 不可 | 不可 |
| 書籍メタ情報編集 | 可 | 可 | 不可 | 不可 | 不可 |
| 書籍削除 | 可 | 可 | 不可 | 不可 | 不可 |
| 変換ジョブ状態参照 | 可 | 可 | 可 | 可 | 不可 |
| 変換ジョブ再実行 | 可 | 可 | 可 | 不可 | 不可 |
| Elasticsearch再インデックス | 可 | 不可 | 可 | 不可 | 不可 |

この表は初版の権限設計であり、実装時は`permission`のコードとして操作単位を定義し、ロールとの関連をPostgreSQLで管理する。詳細な権限コード、初期ロール権限、API権限確認単位は[doc/04_design/08_authorization_design/02_permission_matrix.md](02_permission_matrix.md)で管理する。

## サーバ側権限確認

バックエンドAPIは、次の単位で権限確認を行う。

| 対象 | 確認内容 |
| --- | --- |
| 管理API | 管理ユーザとして認証済みであり、必要な管理権限を持つこと。 |
| 一般ユーザAPI | 一般ユーザとして認証済みであり、自分の会員情報、閲覧履歴、お気に入りだけを操作すること。 |
| 書籍アップロード | 管理ユーザが書籍アップロード権限を持つこと。 |
| 書籍閲覧 | 一般ユーザは対象書籍が有効かつ閲覧可能状態であること。管理ユーザは管理対象書籍の内容確認・品質確認として許可されること。詳細は[権限マトリクス](02_permission_matrix.md)に従う。 |
| 画像 / サムネイル配信 | 内部パスを直接受け取らず、書籍IDやページIDから権限確認後に配信すること。 |
| 検索 | 検索条件を検証し、結果は有効かつ表示可能な書籍に限定すること。 |
| 変換ジョブ操作 | 管理ユーザが変換ジョブ参照または再実行権限を持つこと。 |

フロントエンドのメニュー表示やボタン非表示は利便性のために行うが、セキュリティ境界とはみなさない。

## 退会時のデータ扱い

一般ユーザの退会時は、ユーザ情報を論理削除または`withdrawn`状態へ遷移させる。

| 対象 | 方針 |
| --- | --- |
| 一般ユーザ情報 | `withdrawn`状態として保持する。復元、監査、重複登録制御の詳細は実装時に定義する。 |
| セッション | 退会完了時に既存セッションを無効化する。 |
| お気に入り | 退会時に無効化する。物理削除または論理削除の最終判断はデータモデル実装時に決める。 |
| 閲覧履歴 | 退会時に無効化する。保持期間が必要な場合は運用方針で定義する。 |
| アップロード済み書籍 | 一般ユーザは書籍を保持しないため削除対象外。 |
| 書籍メタ情報 / 原本 / WebP / サムネイル | 一般ユーザ退会では影響を受けない。 |
| 監査ログ | 重要な管理操作やセキュリティイベントは、必要な範囲で保持する。 |

退会済みユーザはログインできない。退会後に同じメールアドレスで再登録を許可するかは、メール一意制約、監査要件、個人情報削除方針を踏まえて実装時に決定する。

## 後続設計で詳細化する事項

- メール認証、ログイン時2段階認証、パスワードリセット、セッションの有効期限、試行回数、再送制限の具体値。
- ログイン失敗制限、一時ロック、ロック解除条件の具体値。
- 管理ユーザ、ロール、権限の具体的なマイグレーションと初期データ。
- 管理操作の監査ログ保持範囲。
- 退会済みユーザのメールアドレス再利用可否。

## 更新方針

認証方式、ロール、権限、退会時のデータ扱い、一般ユーザと管理ユーザの責務境界が変わった場合は、このドキュメントを更新する。

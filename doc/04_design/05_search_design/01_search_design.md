# 検索設計初版

## 目的

このドキュメントは、自炊本閲覧Webアプリケーションにおける検索機能の初期設計を整理する。

対象は、検索対象項目、検索条件、ソート、ページング、日本語検索、Elasticsearchインデックス更新タイミング、更新失敗時の再試行である。

詳細なElasticsearchのindex name、mapping、analyzer定義、boost値、再インデックスコマンド仕様は、[doc/04_design/05_search_design/02_search_index_design.md](02_search_index_design.md) で扱う。

## 前提

- PostgreSQLを正本データストアとする。
- Elasticsearchは検索用の派生インデックスであり、PostgreSQLから再構築可能とする。
- 検索対象の正本は、`book`, `author`, `series`, `book_type`, `tag` とそれらの関連である。
- 一般ユーザは書籍を保持せず、公開または閲覧可能な書籍を検索、閲覧する。
- 書籍アップロードとメタ情報管理は管理ユーザのみが行う。
- 論理削除済みの書籍、著者、シリーズ、種別、タグは通常の検索対象から除外する。
- 変換未完了または非表示の書籍を検索結果に出すかどうかは、書籍の表示状態と検索用途で制御する。

## 検索の責務分担

| コンポーネント | 責務 |
| --- | --- |
| Next.jsフロントエンド | 検索条件入力、検索結果表示、ページング操作、絞り込み条件のUI状態管理を行う。 |
| Spring BootバックエンドAPI | 認証、入力検証、検索条件の正規化、Elasticsearch検索、必要に応じたPostgreSQL正本確認、APIレスポンス生成を行う。 |
| PostgreSQL | 書籍メタ情報、表示状態、論理削除状態、関連、検索インデックス更新状態の正本を保持する。 |
| Elasticsearch | 検索用ドキュメントを保持し、日本語検索、補完、部分一致、ソート、ページングを担う。 |
| 専用キュー / 再試行キュー | Elasticsearch更新失敗時の再試行要求を配送する。 |

Elasticsearchの検索結果だけを業務上の正本として扱わない。検索結果に表示する内容や表示可否が重要な場合は、PostgreSQLの正本データで確認する。

## 検索対象項目

初期検索対象は次のとおりとする。

| 項目 | 正本 | 検索用途 | 初期方針 |
| --- | --- | --- | --- |
| タイトル | `book.title` | キーワード検索、補完、部分一致 | 主検索対象とし、kuromojiベースのフィールドと補完用フィールドを用意する。 |
| 著者名 | `author.name`, `author.name_kana`, `book_author` | 著者検索、複合検索 | 書籍に紐づく著者を検索ドキュメントへ非正規化する。 |
| タグ | `tag.name`, `book_tag` | タグ検索、絞り込み、複合検索 | タグ名で検索、絞り込みできるようにする。 |
| シリーズ名 | `series.name`, `series.name_kana` | シリーズ検索、複合検索、ソート補助 | 書籍に紐づくシリーズを検索ドキュメントへ含める。 |
| 種別 | `book_type.code`, `book_type.name` | 絞り込み、検索結果表示 | 主に完全一致の絞り込み対象とする。 |
| シリーズ概要 | `series.description` | キーワード検索の補助 | 初期では検索対象に含めるが、タイトルや著者より低い優先度にする。 |
| 書籍説明 | `book.description` | キーワード検索の補助 | 初期検索対象に含め、低い優先度にする。 |
| 表示状態 | `book.visibility_status`, `book.deleted_at` | 表示可否制御 | 検索ヒット条件またはPostgreSQL確認で除外する。 |
| サムネイル参照 | `book_file`, `book_page` | 検索結果表示 | 検索結果に必要な識別子のみを持たせ、物理パスは露出しない。 |

シリーズ概要はissue #7の完了条件に含まれるため、初版では検索対象に含める。ただし、検索結果の順位付けではタイトル、著者、タグ、シリーズ名を優先する。

## 検索条件

### キーワード検索

キーワード検索は、ユーザが入力した文字列をタイトル、著者名、タグ、シリーズ名、シリーズ概要、書籍説明に対して検索する。

初期の優先度は次の順とする。

1. タイトル
2. 著者名
3. タグ
4. シリーズ名
5. 種別名
6. シリーズ概要、書籍説明

実装時のboost値は、検索インデックス設計で具体化する。キーワードが空の場合は、通常の本一覧APIまたは絞り込み検索として扱い、全件を無条件に高コスト検索しない。

### タイトル検索

タイトル検索は、書籍タイトルを主対象とする。

- 日本語の形態素解析にはanalysis-kuromojiを使用する。
- 全角 / 半角、大小文字、記号の差を吸収するため、ICU normalizerの利用を前提に検討する。
- 補完や前方一致が必要な場合は、タイトルのedge n-gram系フィールドを使用する。
- 完全一致またはそれに近い一致は、通常のキーワード一致より高く評価する。

### 著者検索

著者検索は、書籍に紐づく著者名と読みを対象とする。

- `author.name` と `author.name_kana` を検索ドキュメントへ含める。
- 著者、作画者、編集者などの役割は `book_author.author_role` として保持し、初期検索では同じ著者候補として扱う。
- 管理画面で役割別の絞り込みが必要になった場合は、後続設計で条件を追加する。

### タグ検索

タグ検索は、書籍に紐づくタグ名を対象とする。

- タグ名のキーワード検索と、タグIDまたはタグコードによる完全一致絞り込みを分けて扱う。
- ユーザがタグを選択するUIでは、完全一致絞り込みを優先する。
- タグ名の表記揺れ統合は、正本データ上のタグ管理と検索アナライザの両方で扱う。

### シリーズ検索

シリーズ検索は、シリーズ名、シリーズ読み、シリーズ概要を対象とする。

- シリーズ名とシリーズ読みは、タイトルに次ぐ主要検索対象として扱う。
- シリーズ概要は補助的な検索対象とし、説明文だけの一致でタイトル一致を上回らないようにする。
- シリーズ内の並び順は `book.series_order` を利用し、検索結果のソート条件として使えるようにする。

### 種別検索

種別は、主に絞り込み条件として扱う。

- `book_type.code` はAPI内部または画面選択値からの完全一致に使用する。
- `book_type.name` はキーワード検索の補助対象に含める。
- 種別の表示順は `book_type.display_order` を正本とし、種別一覧や絞り込み候補の表示に使う。

### 複合検索

複合検索では、キーワード、タグ、著者、シリーズ、種別、表示状態、ページング、ソートを組み合わせる。

初期で扱う条件は次のとおり。

| 条件 | 方針 |
| --- | --- |
| キーワード | タイトル、著者、タグ、シリーズ、説明系項目を横断検索する。 |
| タグ | 複数指定時はAND条件を初期方針とし、OR条件が必要になった場合はAPI契約で指定方法を追加する。 |
| 著者 | 著者ID指定は完全一致、文字列指定は著者名検索とする。 |
| シリーズ | シリーズID指定は完全一致、文字列指定はシリーズ検索とする。 |
| 種別 | `book_type.code` またはIDで絞り込む。 |
| 表示状態 | 一般ユーザ向け検索では閲覧可能な書籍を対象にする。管理画面では状態別検索を許可する。 |

## 日本語検索と表記揺れ

日本語検索にはanalysis-kuromojiを使用する。

初期方針は次のとおり。

| 項目 | 方針 |
| --- | --- |
| 形態素解析 | タイトル、著者名、タグ、シリーズ名、説明系項目にkuromojiベースのカスタムアナライザを使用する。 |
| 正規化 | 全角 / 半角、大小文字、互換文字などの表記揺れ対策としてICU normalizerの利用を検討する。 |
| 補完 | タイトル、著者名、タグ、シリーズ名にはedge n-gram系フィールドを追加できるようにする。 |
| 部分一致 | 補完用途と通常検索を分け、必要な項目だけedge n-gram系フィールドを使う。 |
| ひらがな / カタカナ | `name_kana` を持つ著者、シリーズは読み検索に利用する。追加のかな正規化は検索インデックス設計で検討する。 |
| typo許容 | 初期対象外とし、検索ログや利用状況を見てfuzzinessの導入を検討する。 |

edge n-gram系フィールドはインデックスサイズと更新コストを増やすため、全項目へ無条件に追加しない。初期ではタイトル、著者名、タグ、シリーズ名を候補とする。

## ソート条件

初期のソート条件は次のとおりとする。

| ソート | 用途 | 方針 |
| --- | --- | --- |
| 関連度順 | キーワード検索の既定 | Elasticsearchのscoreを利用する。 |
| 更新日時降順 | 一覧、管理画面 | `book.updated_at` を利用する。 |
| 登録日時降順 | 新着順 | `book.created_at` を利用する。 |
| タイトル昇順 | 一覧整理 | 正規化またはkeyword系フィールドを利用する。 |
| シリーズ順 | シリーズ詳細や絞り込み時 | `series.name` と `book.series_order` を利用する。 |

一般ユーザ向け検索の既定は、キーワードありの場合は関連度順、キーワードなしの場合は更新日時降順とする。

大量データで深いページングが必要になった場合は、`from` / `size` だけでなく `search_after` の利用を検討する。

## ページング

検索結果は必ずページングする。

初期方針は次のとおり。

| 項目 | 方針 |
| --- | --- |
| 既定件数 | 20件を候補とし、API契約で確定する。 |
| 最大件数 | 100件を候補とし、過大な要求はサーバ側で拒否または丸める。 |
| ページ指定 | 初期はページ番号方式を候補とする。 |
| 深いページング | 件数が増えた段階で `search_after` を検討する。 |
| 総件数 | UIに必要な範囲で返す。高コストになる場合は概算または上限付き表示を検討する。 |

ページングパラメータはバックエンドAPIで検証する。負数、過大なサイズ、未定義ソート項目は受け付けない。

## 検索結果表示

検索結果に含める初期項目は次のとおりとする。

| 項目 | 方針 |
| --- | --- |
| 書籍ID | APIの公開識別子を返す。内部DB IDを直接露出するかはAPI設計で判断する。 |
| タイトル | 検索結果の主表示項目とする。 |
| 著者 | 表示順に従って主要著者を表示する。 |
| タグ | 件数が多い場合は表示数を制限する。 |
| シリーズ | シリーズ名とシリーズ順を表示できるようにする。 |
| 種別 | 必要に応じてバッジや絞り込み表示に使う。 |
| 表紙サムネイル | サムネイル参照用のAPI識別子を返し、物理パスは返さない。 |
| 変換 / 表示状態 | 管理画面では表示し、一般ユーザ向けでは閲覧不可の書籍を原則出さない。 |

ハイライト表示は初期必須ではない。導入する場合は、Elasticsearchのhighlight結果をそのまま信頼せず、XSS対策を行う。

## インデックス更新タイミング

Elasticsearchの検索ドキュメントは、PostgreSQLの正本更新を起点に更新する。

| 更新契機 | 更新内容 |
| --- | --- |
| 書籍作成 | 書籍メタ情報、関連、表示状態を検索ドキュメントへ登録する。 |
| 書籍メタ情報更新 | タイトル、説明、種別、シリーズ、著者、タグなどを更新する。 |
| 著者更新 | 関連する書籍の著者情報を再インデックスする。 |
| タグ更新 | 関連する書籍のタグ情報を再インデックスする。 |
| シリーズ更新 | 関連する書籍のシリーズ名、読み、概要を再インデックスする。 |
| 種別更新 | 関連する書籍の種別名や表示情報を再インデックスする。 |
| 変換完了 | サムネイル参照、閲覧可能状態、ページ数など検索結果表示に必要な情報を更新する。 |
| 書籍非表示 / 論理削除 | 一般検索から除外する。必要に応じてElasticsearch上のドキュメントを削除または非表示状態へ更新する。 |
| 書籍削除 | PostgreSQLを正として、Elasticsearchから削除または再構築で除外できるようにする。 |

PostgreSQL更新とElasticsearch更新は、PostgreSQLを先に確定させる。Elasticsearch更新に失敗しても、正本更新を失わない設計にする。

## 更新失敗時の再試行

Elasticsearch更新に失敗した場合は、再試行キューに積む。

初期方針は次のとおり。

1. PostgreSQLの正本更新を完了する。
2. Elasticsearch更新を試行する。
3. 更新に失敗した場合、検索インデックス更新状態をPostgreSQLへ記録する。
4. 書籍ID、更新種別、対象エンティティ、失敗理由の要約、次回再試行時刻を再試行キューまたは正本側の管理情報へ渡す。
5. 再試行ワーカーまたは管理コマンドがPostgreSQLを読み直し、最新の正本からElasticsearchドキュメントを作り直す。
6. 再試行でも失敗する場合は、運用確認対象としてログと状態を残す。

再試行時は、失敗時点の古い差分ではなく、PostgreSQLの最新状態からドキュメントを再生成する。これにより、再試行待ちの間に複数回メタ情報が更新されても、最終的な検索ドキュメントを正本に寄せられる。

失敗理由には、シークレット、内部物理パス、不要な個人情報を記録しない。

## 再インデックス

Elasticsearchは再構築可能な派生データであるため、次の再インデックス手段を用意する。

| 種別 | 用途 |
| --- | --- |
| 書籍単位再インデックス | 特定書籍の検索結果が疑わしい場合、または関連メタ情報更新失敗を回復する場合に使用する。 |
| 関連単位再インデックス | 著者、タグ、シリーズ、種別の更新により関連書籍をまとめて更新する場合に使用する。 |
| 全件再インデックス | インデックス定義変更、Elasticsearch障害、整合性不明時にPostgreSQLから全検索ドキュメントを再構築する。 |
| インデックス破棄後再構築 | Elasticsearchの内容が信用できない場合に、PostgreSQLを正として作り直す。 |

具体的なコマンド、ロック、実行中の検索への影響、インデックスエイリアス切り替えは、検索インデックス設計またはRunbookで定義する。

## セキュリティと入力検証

検索APIでは次の検証を行う。

- 未ログインまたは権限のない利用者に、管理用の検索条件や非表示書籍を露出しない。
- 検索文字列、タグ、著者、シリーズ、種別、ソート、ページングをサーバ側で検証する。
- 過大なページサイズ、深すぎるページング、長すぎるキーワードを制限する。
- ElasticsearchのクエリDSLをクライアントから直接受け取らない。
- ハイライトや検索語を画面表示する場合はXSS対策を行う。
- 内部の物理ファイルパスやストレージキーを不用意にAPIレスポンスへ出さない。

## Elasticsearchインデックス設計との関係

Elasticsearchの具体的な設計は、[doc/04_design/05_search_design/02_search_index_design.md](02_search_index_design.md) に記録する。

同ドキュメントでは次を定義する。

- index name、read alias、write alias、physical indexの命名。
- kuromojiベースのカスタムアナライザ、ICU normalizer、edge n-gram系フィールド。
- 書籍単位のmapping、nested field、searchable fields、sortable fields。
- タイトル、著者、タグ、シリーズ、種別、シリーズ概要、書籍説明のboost設定。
- PostgreSQL更新を起点としたインデックス更新タイミング。
- 書籍単位再インデックスと全件再インデックスの手順方針。

検索API契約、リクエスト、レスポンス、エラー形式は [doc/04_design/03_api_contracts/04_search_api.md](../03_api_contracts/04_search_api.md) で扱う。

再試行キューの製品、メッセージ形式、リトライ回数、デッドレター扱いは、実装方式を決める段階で関連設計またはRunbookへ追記する。

## 更新方針

検索対象、検索条件、表示可否、インデックス更新タイミング、再試行、再インデックス方針が変わった場合は、このドキュメントを更新する。

Elasticsearchの具体的なmappingやanalyzerを決めた場合は、検索インデックス設計へ反映し、このドキュメントとの整合性を確認する。

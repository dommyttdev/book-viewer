# テスト戦略

## 目的

このドキュメントは、自炊本閲覧WebアプリケーションのMVPからv1.0までに必要なテスト方針を定義する。

テストは実装後の確認作業だけではなく、仕様を具体化し、設計の責務境界を保ち、回帰を防ぐための開発活動として扱う。特にビジネスルール、入力検証、権限確認、変換ジョブ状態遷移、検索インデックス更新、ファイル処理の安全性は、実装前または実装と同じ作業単位でテスト観点を明確にする。

## 前提

- 開発手法はTDDを基本とする。
- 新しい振る舞いを実装する前に、期待する振る舞いをテストまたは受入条件として表現する。
- 小さな単位では、失敗するテストを書く、最小実装で通す、必要に応じてリファクタリングする、という流れを優先する。
- UIや外部システム連携のように自動テストだけで表現しにくいものは、受入テスト観点と手動確認手順を先に整理する。
- PostgreSQLを正本、Elasticsearchを再構築可能な派生データとして扱う前提をテストでも守る。
- 7-Zip for Linux コンソール版、ファイルシステム、Elasticsearch、専用キューは、テスト種別に応じてモック、テストコンテナ、ローカル実行、手動確認を使い分ける。

## テスト範囲

対象範囲は次のとおり。

- Next.jsフロントエンドの画面、入力、API連携、権限別表示
- Spring BootバックエンドAPIのユースケース、入力検証、認証、認可、エラー応答
- Spring Boot変換ワーカーのアーカイブ展開、画像変換、サムネイル生成、ジョブ状態遷移
- PostgreSQL永続化、トランザクション、制約
- Elasticsearch検索、インデックス更新、再インデックス
- 書籍ファイル保存領域、作業ディレクトリ、パストラバーサル対策
- Docker Compose上の主要な結合動作
- MVP主要フローの受入確認と回帰確認

MVP初期では、すべてをE2Eで覆うのではなく、重要な業務ルールと失敗しやすい境界を下位テストで厚めに確認し、主要な利用者フローを少数のE2Eで確認する。

## TDDの進め方

実装タスクは、原則として次の順に進める。

1. 該当するユーザーストーリー、受入条件、設計ドキュメントを確認する。
2. 期待する振る舞い、境界値、異常系、権限条件をテスト名または受入テスト観点として書き出す。
3. 可能な範囲で、失敗する単体テストまたは結合テストを先に追加する。
4. テストが通る最小実装を行う。
5. 責務境界、命名、重複、例外処理、ログを見直してリファクタリングする。
6. 必要に応じてE2E、手動確認、ドキュメント更新を行う。

TDDの単位は小さく保つ。DB、外部プロセス、ブラウザ、Elasticsearchをすべて含む大きなテストから始めず、ドメインやユースケースの振る舞いを先に固定する。

## 単体テスト方針

単体テストは、フレームワークや外部依存から切り離した小さな振る舞いを確認する。

優先対象:

- ドメインモデルの不変条件
- ユースケースの分岐、状態遷移、権限判断
- 入力値の正規化とバリデーション
- 変換ジョブ状態の遷移可否
- ファイル名、アーカイブ内パス、展開後パスの安全性判定
- WebP品質値、同時実行数、タイムアウトなど設定値の扱い
- Elasticsearchへ渡す検索条件やインデックス更新指示の組み立て

外部依存はポートやインターフェースで置き換え、テスト対象の責務だけを検証する。モックは外部依存境界に限定し、業務ルールそのものをモックで隠さない。

## 結合テスト方針

結合テストは、複数レイヤーまたは実際の外部依存との接続を確認する。

優先対象:

- APIリクエストからユースケース、永続化までの流れ
- PostgreSQL制約、トランザクション、楽観ロックなどのDB依存動作
- 認証、認可、バリデーションエラー、HTTPステータス、エラーレスポンス形式
- Elasticsearchへのインデックス登録、検索、再インデックス
- 専用キューを介した変換ジョブ投入と取得
- ファイル保存領域への保存、削除、パス検証

PostgreSQLとElasticsearchは、実装が具体化した段階でテストコンテナまたはローカルDocker Composeを使った確認を優先する。外部依存を使う結合テストは、実行時間と安定性を見てCIの常時実行対象と手動実行対象を分ける。

## E2Eテスト方針

E2Eテストは、利用者視点の主要フローが壊れていないことを確認する。

MVPで優先するフロー:

- ログインして本一覧を表示する。
- 管理ユーザが本をアップロードし、変換ジョブが作成される。
- 変換済みの本を検索して詳細へ移動する。
- ビューアでページを表示し、ページ送りを行う。
- 権限不足のユーザが管理操作やアップロードを実行できない。

E2Eは少数の代表シナリオに絞り、細かな業務分岐は単体テストや結合テストで確認する。ブラウザ依存の表示崩れ、レスポンシブ表示、操作感は必要に応じて手動確認を併用する。

## パフォーマンステスト方針

パフォーマンステストは、MVPの利用想定と品質特性に対して大きな劣化がないことを確認する。

確認対象:

- 本一覧、検索結果、詳細取得のレスポンス時間
- ページ画像取得の応答とキャッシュ方針
- アップロード受付時のAPI応答
- Elasticsearch検索の応答時間
- PostgreSQLとElasticsearchの再インデックス処理時間
- 変換ジョブの処理時間と失敗率

初期段階では、厳密なSLAよりも基準値を記録し、変更時に劣化を検知できる状態を優先する。性能目標は `doc/03_architecture/07_quality_attributes/01_quality_attributes.md` と整合させる。

## 画像変換ワーカーの負荷テスト方針

画像変換ワーカーは、同時実行数10、1ジョブ30分タイムアウト、7-Zip外部プロセス利用を前提に確認する。

確認対象:

- 同時実行数10でジョブが過剰に起動しない。
- zip、rar、7zipの展開処理がジョブごとの専用作業ディレクトリで行われる。
- 破損アーカイブ、暗号化アーカイブ、非対応画像が失敗状態として追跡できる。
- パストラバーサルを含むアーカイブが作業ディレクトリ外へ書き込めない。
- タイムアウト時に外部プロセスとジョブ状態が適切に扱われる。
- WebP品質値80とサムネイル生成が設定値に従う。
- 失敗ジョブが他ジョブの処理を妨げない。

大量の実画像や大容量アーカイブはリポジトリへ含めず、テスト用の最小サンプル、生成データ、ローカル専用データを使う。

## 手動確認方針

手動確認は、自動テストでは判断しにくい見た目、操作感、変換画像の可読性、運用時のログ追跡を補う目的で行う。

確認対象:

- スマートフォン表示で本一覧、検索、ビューアが利用できる。
- 変換後WebPの本文やセリフが読める。
- アップロード、変換失敗、検索結果なし、権限不足の表示が理解しやすい。
- 管理ユーザと一般ユーザで表示される操作が適切に分かれる。
- 変換ジョブのログと状態から失敗原因を追跡できる。

手動確認を行った場合は、確認した環境、対象ブラウザ、確認結果、既知の制約を作業記録またはプルリクエストに残す。

## 回帰テスト方針

回帰テストは、MVP主要機能の既存動作が壊れていないことを確認する。

優先対象:

- ログイン、ログアウト、メール認証、パスワードリセット
- 本アップロード、メタ情報更新、削除
- 変換ジョブ作成、状態更新、失敗、再実行
- 検索、表記揺れ、日本語検索、再インデックス
- 本一覧、詳細、ビューア、閲覧位置保存
- お気に入り
- 管理ユーザ管理、ロール、権限確認

不具合修正時は、同じ不具合が再発したときに失敗するテストを先に追加する。回帰テスト一覧は `doc/06_testing/03_regression_tests/01_regression_tests.md` で管理する。

## CIと実行区分

CIが整備されたら、テストを次の実行区分に分ける。

| 区分 | 対象 | 実行タイミング |
| --- | --- | --- |
| 常時実行 | 単体テスト、軽量な結合テスト、静的解析 | push、pull request |
| PR前確認 | 変更範囲に関わる結合テスト、主要E2E | pull request作成前 |
| 手動実行 | 重いE2E、性能、ワーカー負荷、実画像確認 | リリース前、影響範囲が大きい変更時 |
| 障害対応後 | 再発防止テスト、関連回帰テスト | 修正時 |

CI未整備の間も、実行したコマンド、手動確認、未実行理由を作業記録に残す。

## テストデータ方針

テストデータは、最小で再現性のあるものを優先する。

- 秘密情報、実利用者情報、個人的な原本ファイルを使わない。
- 大容量の原本ファイルや変換済み画像をGitへ含めない。
- アーカイブテストは、正常、破損、パストラバーサル、非対応形式など目的別に小さく分ける。
- PostgreSQLのテストデータは、テストごとに独立して作成し、順序依存を避ける。
- ElasticsearchはPostgreSQLから再構築可能であることを前提に、インデックス内容を直接正本として扱わない。

詳細なテストデータ一覧は、受入テストと回帰テストが増えた段階で `doc/06_testing/05_test_data/` に整理する。

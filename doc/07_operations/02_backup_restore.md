# バックアップなし方針初版

## 目的

このドキュメントは、自炊本閲覧Webアプリケーションの初期運用において、バックアップを行わない方針、対象、リスク、許容範囲を明確にする。

## 関連文書

バックアップなし運用のRPO / RTO、復旧不能リスク、禁止操作、削除系操作の安全柵、見直し条件は、このドキュメントを正本とする。

- 運用手順は [Runbook](01_runbook.md) に記載する。
- 障害時の記録項目は [障害ログテンプレート](05_incident_log_template.md) に記載する。
- 品質特性上の位置づけは [品質特性](../03_architecture/07_quality_attributes.md) に記載する。

## 方針

初期運用ではバックアップを行わない。

PostgreSQL、Elasticsearch、原本ファイル、変換済みWebP、サムネイル、RabbitMQの永続データは、いずれもバックアップ対象にしない。

この方針は、個人または小規模の初期運用で構成を単純に保つための判断である。データ保全を保証する運用ではなく、利用者と運用者が復旧不能リスクを受け入れることを前提にする。

## RPO / RTO

バックアップなし運用では、バックアップ復元を前提にした復旧時点目標と復旧時間目標を定義しない。

| 項目 | 初期方針 |
| --- | --- |
| RPO | バックアップがないため、PostgreSQLまたは原本ファイルが失われた場合の復旧可能時点は存在しない。消失した正本データは復旧不能とする。 |
| RTO | 正本喪失時はバックアップ復元による復旧目標時間を定義しない。残存する正本から再構築できる範囲のみ、個別の運用判断で回復する。 |

Elasticsearch、変換済みWebP、サムネイル、RabbitMQの処理中データは、PostgreSQLまたは原本ファイルが残っている場合に限って再構築または整理を行う。PostgreSQLまたは原本ファイルが失われた場合は、派生データが残っていても完全な復旧根拠として扱わない。

## 想定する利用形態

この方針を適用する利用形態は、初期の個人利用または小規模チーム利用に限定する。

- 利用者が、書籍メタ情報、アカウント、権限、閲覧状態、原本ファイルを失う可能性を受け入れている。
- 有償提供、長期保存保証、業務継続保証、第三者データの保全責任を前提にしない。
- 運用者が少人数で、破壊的な操作を通常運用から外せる。
- 障害時は、復旧不能な範囲を明確に説明し、再登録または再アップロードで対応できる範囲だけを扱う。

## 対象データ

| 対象 | 位置づけ | バックアップ | 消失時の扱い |
| --- | --- | --- | --- |
| PostgreSQL | 正本 | 行わない | 書籍メタ情報、ユーザ、権限、ジョブ状態を復元できない |
| Elasticsearch | 派生データ | 行わない | PostgreSQLが残っていれば再インデックスで再構築する |
| 原本ファイル | 永続ファイル | 行わない | 再生成できない |
| 変換済みWebP | 派生ファイル | 行わない | 原本ファイルが残っていれば再変換で再生成する |
| サムネイル | 派生ファイル | 行わない | 原本ファイルまたはWebPが残っていれば再生成する |
| RabbitMQ | 処理中データ | 行わない | PostgreSQLのジョブ状態を正として再投入または失敗扱いにする |
| ログ | 診断情報 | 行わない | 過去調査情報を失う |

## 再構築可能なもの

ElasticsearchはPostgreSQLから再構築可能な派生データとして扱う。

変換済みWebPとサムネイルは、原本ファイルと変換条件から再生成可能な派生ファイルとして扱う。

ただし、PostgreSQLまたは原本ファイルが失われた場合、Elasticsearch、WebP、サムネイルの再構築に必要な正本が失われる。

## 再構築できないもの

次のデータは、バックアップなし運用では消失すると復元できない。

- PostgreSQL上のユーザ、権限、書籍メタ情報、ページ情報、ジョブ状態、閲覧履歴、お気に入り。
- 原本ファイル。
- PostgreSQLにしか記録していない検索インデックス更新状態。
- 障害調査に必要だった過去ログ。

原本ファイルは通常運用では保存し続けるが、バックアップはしないため、ストレージ障害、誤削除、ホスト障害で失われる可能性がある。

## 許容するリスク

初期運用では次のリスクを許容する。

| リスク | 影響 | 許容理由 |
| --- | --- | --- |
| ホスト障害による全データ消失 | アプリケーション状態と書籍ファイルを復元できない | 初期運用の単純さを優先する |
| 誤操作によるDBまたはファイル削除 | 対象データを復元できない | 少人数運用を前提に操作を限定する |
| Elasticsearch障害 | 検索結果が欠落または古くなる | PostgreSQLから再インデックスできる |
| WebPまたはサムネイル消失 | 閲覧または一覧表示ができなくなる | 原本が残っていれば再変換できる |
| ログ消失 | 過去障害の原因分析が難しくなる | 重要な障害は障害ログに要約を残す |

## 許容しない扱い

バックアップなし方針であっても、次は行わない。

- PostgreSQL以外を正本として扱う。
- Elasticsearchだけに業務上重要な情報を保持する。
- 原本ファイルの消失をWebPから完全に復元できるものとして扱う。
- バックアップなしであることを隠して、データ保全があるように説明する。
- 秘密情報をログや障害ログに残す。

## 運用上の注意

- `docker compose down -v`、Docker volume削除、書籍ファイル保存領域削除、PostgreSQLデータ領域削除は通常運用で禁止する。
- Docker Composeのデータボリュームを削除する操作は、バックアップなし運用では復旧不能なデータ消失に直結するため、障害対応や容量確保の手段として使用しない。
- ストレージ容量確保のための削除は、PostgreSQLを正として対象を確認してから行う。
- Elasticsearchの再構築は、PostgreSQLが正しい状態であることを確認してから実行する。
- 書籍削除時は、論理削除、Elasticsearch除外、物理ファイル削除の順序を区別する。
- 原本ファイルがない書籍は、再変換できないものとして扱う。

## 削除系操作の安全柵

削除系の管理コマンドや運用スクリプトを実装する場合は、次を必須とする。

- 既定はdry-runとし、実削除は明示オプションでのみ実行する。
- 削除対象の件数、書籍ID、ジョブID、対象種別、対象パスの要約を実行前に表示する。
- 実削除前に、対象件数と確認語句を使った二段階確認を行う。
- PostgreSQL上の正本状態と物理ファイルの対応を確認し、正本に存在する原本ファイルを未参照ファイルとして削除しない。
- Elasticsearchなどの派生データ削除と、PostgreSQLや原本ファイルの正本削除を同じ確認なしに混在させない。
- 削除結果、実行者、実行日時、対象件数をログまたは障害ログへ記録する。

## 方針を見直す条件

次のいずれかに該当した場合は、バックアップ方針を見直す。

- 実利用者が増え、データ消失の影響が大きくなった。
- 原本ファイルやメタ情報の再登録コストが許容できなくなった。
- 有償提供や継続利用を前提にする。
- 運用者が複数になり、誤操作リスクが増えた。
- 障害復旧時間や復旧時点の目標を定義する必要が出た。
- RPOまたはRTOを利用者へ約束する必要が出た。
- 誤操作対策だけでは許容できない破壊的操作リスクが残る。

見直し時は、PostgreSQL、原本ファイル、変換済みWebP、サムネイル、Elasticsearch、ログのうち、どれをバックアップ対象にするかを個別に決める。

## 復元手順について

バックアップを行わないため、バックアップからの復元手順は提供しない。

障害時は、残っている正本データを基準に次の回復を行う。

- PostgreSQLが残っている場合は、Elasticsearchを再インデックスする。
- 原本ファイルが残っている場合は、WebPとサムネイルを再変換する。
- RabbitMQの状態が失われた場合は、PostgreSQLのジョブ状態を確認して再投入または失敗扱いにする。
- PostgreSQLまたは原本ファイルが失われた場合は、復元不可として障害ログに記録する。

## 更新方針

バックアップ対象、復元目標、ストレージ構成、データ保全の要求が変わった場合は、このドキュメントを更新する。

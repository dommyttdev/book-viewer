# 監視方針初版

## 目的

このドキュメントは、自炊本閲覧Webアプリケーションの初期運用で確認する監視候補、異常の見方、初期対応を整理する。

## 前提

- 単一Linuxホスト上のDocker Compose構成で運用する。
- PostgreSQLを正本とする。
- Elasticsearchは再構築可能な派生データとする。
- 画像変換はSpring Boot変換ワーカーが非同期ジョブとして実行する。
- バックアップは行わないため、ストレージ障害や誤削除は復元できない可能性がある。

## 監視の基本方針

初期運用では、まず障害検知と原因切り分けに必要な最小限の観点を整理する。

本格的な監視基盤、アラート通知、ダッシュボード、SLOは後続で具体化する。実装時はSpring Boot Actuator、Dockerログ、PostgreSQL、Elasticsearch、ホストメトリクスを利用する方針を候補とする。

## 監視候補

| 領域 | 監視項目 | 見る理由 | 初期対応 |
| --- | --- | --- | --- |
| API | Spring BootバックエンドAPI死活 | 画面とAPI操作が利用できるか確認する | コンテナ状態、APIログ、DB接続を確認する |
| Worker | Spring Boot変換ワーカー死活 | 変換ジョブが処理されるか確認する | コンテナ状態、RabbitMQ接続、ワーカーログを確認する |
| JVM | JVMメモリ、GC、スレッド数 | メモリ不足や処理詰まりを検知する | コンテナ制限、ヒープ設定、負荷を確認する |
| DB | PostgreSQL接続 | 正本データへアクセスできるか確認する | DBコンテナ、接続情報、接続数を確認する |
| Search | Elasticsearch接続 | 検索と再インデックスが利用できるか確認する | ESコンテナ、クラスタ状態、APIログを確認する |
| Storage | ストレージ容量 | 原本、WebP、サムネイル、DBが保存できるか確認する | 容量増加箇所、残存作業ディレクトリを確認する |
| RabbitMQ | 変換ジョブ滞留 | 変換が遅延していないか確認する | ワーカー死活、同時実行数、失敗数を確認する |
| RabbitMQ | 未ackメッセージ数 | Workerが取得したまま完了していないジョブを確認する | Workerログ、長時間ジョブ、停止中コンテナを確認する |
| RabbitMQ | dead letter queue件数 | 再試行上限を超えたジョブを検知する | PostgreSQLのジョブ状態と失敗理由を確認する |
| Conversion | 変換失敗数 | 原本形式、7-Zip、画像変換、保存失敗を検知する | 失敗コードとワーカーログを確認する |
| Conversion | 画像変換処理時間 | ホスト負荷や巨大ファイルの影響を検知する | ページ数、解像度、CPU、I/Oを確認する |
| 7-Zip | 7-Zip外部プロセス失敗数 | アーカイブ展開の問題を検知する | 終了コード、タイムアウト、暗号化、破損を確認する |
| Worker | 変換ワーカー同時実行数 | 画像変換設計で定義する設定値の妥当性を確認する | RabbitMQキュー滞留とホスト負荷から調整を検討する |
| Timeout | 1ジョブ30分タイムアウト発生数 | 重すぎる原本や処理詰まりを検知する | ジョブ内容、リソース、設定値を確認する |

## ヘルスチェック

初期実装では次のヘルスチェックを候補とする。

| 対象 | チェック内容 |
| --- | --- |
| フロントエンド | HTTP応答が返ること |
| API | HTTPヘルスチェックが成功すること |
| APIからPostgreSQL | DB接続確認が成功すること |
| APIからElasticsearch | ES接続確認が成功すること |
| APIからRabbitMQ | RabbitMQ接続確認が成功すること |
| Worker | プロセスが起動し、RabbitMQからジョブを取得できること |
| Workerから7-Zip | 7-Zip実行ファイルを呼び出せること |

ヘルスチェックURLや具体的なActuator endpointは実装時に確定する。

## ログ監視

ログでは次を確認する。

- APIの認証失敗、認可失敗、入力検証失敗。
- PostgreSQL接続失敗。
- Elasticsearch接続失敗、更新失敗、再試行。
- RabbitMQ接続失敗、ack失敗、再投入、dead letter。
- 変換ジョブの失敗工程と失敗コード。
- 7-Zip終了コード、タイムアウト、標準エラー要約。
- ストレージ書き込み失敗。

ログに秘密情報、トークン、パスワード、セッションID、内部物理パス、不要な個人情報を出さない。

## 変換ジョブ監視

変換ジョブはPostgreSQLの状態を正として確認する。

| 状態 | 監視観点 |
| --- | --- |
| `queued` | 件数、滞留時間、古いジョブの有無 |
| `extracting` | 長時間継続、7-Zipタイムアウト |
| `converting` | 長時間継続、ページ数、CPU / I/O負荷 |
| `completed` | 処理時間、生成ページ数、サムネイル有無 |
| `failed` | 失敗数、失敗工程、失敗コード、対象形式 |
| `canceled` | 取消理由、再実行要否 |

初期アラート候補は次のとおり。

- `queued`の滞留が一定時間を超える。
- `failed`が短時間に増加する。
- `extracting`または`converting`が30分を超える。
- 7-Zip失敗が増加する。
- 変換ワーカー同時実行数を引き上げた状態でCPU、メモリ、I/O、一時ディスクが継続的に逼迫する。

## RabbitMQ監視

RabbitMQは変換ジョブ配送に必要なため、次を確認する。

- 接続可否。
- `book.conversion.queue`の滞留数。
- 未ackメッセージ数。
- 再配送数。
- dead letter queue件数。
- APIからのpublish失敗。
- Workerのconsumer接続状態。

RabbitMQ上のメッセージは業務状態の正本ではない。障害時はPostgreSQLの`conversion_job`を正として再投入、再実行、失敗整理を行う。

## Elasticsearch監視

Elasticsearchは派生データだが、検索機能には必要である。

監視候補は次のとおり。

- 接続可否。
- インデックス存在確認。
- 検索応答時間。
- インデックス更新失敗数。
- 再試行キュー滞留。
- 全件再インデックスの成功件数、失敗件数。
- PostgreSQL上の検索更新Outboxとの不整合。

Elasticsearch障害時は、PostgreSQLが残っていれば再インデックスで回復できる。ただし、障害中は検索結果が利用できない、古い、または欠落する可能性がある。

## ストレージ監視

バックアップなし運用では、ストレージ障害や誤削除が重大な影響を持つ。

監視候補は次のとおり。

- ホスト全体の空き容量。
- PostgreSQLデータ領域の容量。
- Elasticsearchデータ領域の容量。
- 書籍ファイル保存領域の容量。
- 変換ワーカー作業領域の容量。
- `trash/`や残存作業ディレクトリの増加。

容量逼迫時は、PostgreSQLを正として削除対象を確認する。未参照ファイルや残存作業ディレクトリを削除する場合も、対象が保存領域外へ広がっていないことを確認する。

## 初期アラート候補

| 重要度 | 条件 | 対応 |
| --- | --- | --- |
| 高 | APIが応答しない | コンテナ、APIログ、DB接続を確認する |
| 高 | PostgreSQLに接続できない | DBコンテナ、容量、認証情報、ネットワークを確認する |
| 高 | ストレージ空き容量が危険域 | 書き込み停止リスクを確認し、容量確保を検討する |
| 中 | Workerが停止している | 変換滞留とワーカーログを確認する |
| 中 | Elasticsearchに接続できない | 検索影響を確認し、再起動または再インデックスを検討する |
| 中 | 変換失敗が急増 | 失敗コード、原本形式、7-Zip、リソースを確認する |
| 中 | RabbitMQキュー滞留が増加 | Worker死活、同時実行数、ホスト負荷を確認する |
| 低 | 再インデックス失敗 | 対象書籍と正本データを確認する |

## 更新方針

実装でヘルスチェックURL、メトリクス名、ログ形式、管理コマンド、アラートしきい値が確定した場合は、このドキュメントを更新する。

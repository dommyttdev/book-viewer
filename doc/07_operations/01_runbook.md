# Runbook初版

## 目的

このRunbookは、自炊本閲覧Webアプリケーションを単一Linuxホスト上のDocker Compose構成で運用するための初期手順を整理する。

対象は、起動、停止、再起動、ログ確認、変換ジョブ失敗時の確認、Elasticsearch再インデックスである。

## 前提

- 本番運用環境は単一Linuxホスト上のDocker Compose構成とする。
- Next.jsフロントエンド、Spring BootバックエンドAPI、Spring Boot変換ワーカー、PostgreSQL、Elasticsearch、専用キューを同一ホストに配置する。
- PostgreSQLをメタ情報、ユーザ、権限、ジョブ状態、検索インデックス更新状態の正本とする。
- ElasticsearchはPostgreSQLから再構築可能な派生データとして扱う。
- 原本ファイルは通常運用では保存し続ける。
- 変換済みWebPとサムネイルは、原本ファイルと変換条件から再生成可能な派生ファイルとして扱う。
- バックアップは行わない。

## 運用対象コンポーネント

| コンポーネント | 役割 | 主な確認観点 |
| --- | --- | --- |
| Next.jsフロントエンド | 一般ユーザ画面、管理ユーザ画面を提供する | HTTP応答、API接続 |
| Spring BootバックエンドAPI | 認証、書籍管理、検索、閲覧、管理操作を提供する | HTTP応答、DB接続、Elasticsearch接続、キュー接続 |
| Spring Boot変換ワーカー | アーカイブ展開、WebP変換、サムネイル生成を実行する | キュー取得、ジョブ状態、7-Zip実行、保存領域アクセス |
| PostgreSQL | 正本データを保持する | 接続、容量、ジョブ状態、検索更新状態 |
| Elasticsearch | 検索用派生インデックスを保持する | 接続、インデックス状態、検索応答 |
| 専用キュー | APIと変換ワーカーを非同期に接続する | 滞留、失敗、再試行 |
| 書籍ファイル保存領域 | 原本、WebP、サムネイルを保存する | 容量、権限、欠落、未参照ファイル |

## 事前確認

本番ホストで作業する前に次を確認する。

- 操作対象ホストが本番ホストであること。
- 作業者がDocker Compose、アプリケーション設定、ログ、保存領域を確認できる権限を持つこと。
- `.env`などの環境別設定が配置済みで、秘密情報がGit管理されていないこと。
- 書籍ファイル保存領域、変換ワーカー作業領域、PostgreSQL、Elasticsearchのデータ領域がDocker Composeから参照できること。
- 7-Zip for Linuxコンソール版が変換ワーカーコンテナ内から実行できること。

## 起動手順

1. 本番ホストへログインする。
2. デプロイ対象ディレクトリへ移動する。
3. 環境変数とDocker Compose設定を確認する。
4. ミドルウェアとアプリケーションを起動する。

```bash
docker compose up -d
```

5. コンテナ状態を確認する。

```bash
docker compose ps
```

6. バックエンドAPIのヘルスチェックを確認する。
7. 変換ワーカーが専用キューへ接続できていることをログで確認する。
8. PostgreSQL、Elasticsearch、専用キューへの接続エラーが出ていないことを確認する。
9. 管理画面にログインできることを確認する。

ヘルスチェックURL、管理コマンド、サービス名は実装時のDocker Compose定義に合わせてこのRunbookへ追記する。

## 停止手順

1. 実行中の変換ジョブ数とキュー滞留を確認する。
2. 急ぎでない停止の場合は、新規アップロードや変換ジョブ投入を止め、実行中ジョブの完了を待つ。
3. コンテナを停止する。

```bash
docker compose down
```

4. 停止後、コンテナ状態を確認する。

```bash
docker compose ps
```

変換中に停止した場合、ジョブ状態が実行中のまま残る可能性がある。再起動後にジョブ状態、キュー状態、作業ディレクトリ残存を確認し、必要に応じて再実行または失敗扱いへ整理する。

## 再起動手順

通常の再起動は、停止手順と起動手順を順に実行する。

```bash
docker compose down
docker compose up -d
```

設定変更を反映する場合は、対象サービスのイメージ再作成やコンテナ再作成が必要かを確認する。PostgreSQLとElasticsearchのデータボリュームは削除しない。

## ログ確認手順

全体ログを確認する。

```bash
docker compose logs --tail=200
```

特定サービスのログを確認する。

```bash
docker compose logs --tail=200 api
docker compose logs --tail=200 worker
docker compose logs --tail=200 frontend
docker compose logs --tail=200 postgres
docker compose logs --tail=200 elasticsearch
```

継続監視する場合は次を使用する。

```bash
docker compose logs -f worker
```

ログには、パスワード、トークン、セッションID、秘密鍵、不要な個人情報、内部物理パスを出さない。障害調査で必要な場合も、共有前に秘密情報が含まれていないことを確認する。

## 変換ジョブ失敗時の確認手順

変換ジョブが失敗した場合は、PostgreSQLのジョブ状態を正として確認する。

1. 管理画面または管理コマンドで対象ジョブを特定する。
2. `conversion_job.status`、`failure_phase`、`failure_code`、`failure_message`、`external_exit_code`、`timed_out`を確認する。
3. ワーカーログで同じジョブIDの処理を確認する。
4. 原本ファイル管理情報と物理ファイルの存在を確認する。
5. ジョブ専用作業ディレクトリが残っている場合は、保存領域外のファイルを参照していないことを確認する。
6. 失敗原因を分類する。

| 失敗分類 | 確認観点 | 初期対応 |
| --- | --- | --- |
| 原本ファイル欠落 | PostgreSQLに管理情報があり、物理ファイルがない | 原本消失として障害ログへ記録する |
| アーカイブ展開失敗 | 7-Zip終了コード、暗号化、破損、非対応形式 | ユーザに安全なエラーとして扱い、必要なら再アップロードを案内する |
| パス検証失敗 | 絶対パス、`..`、シンボリックリンク | 不正または危険なアーカイブとして失敗扱いにする |
| 画像判定失敗 | 対応画像なし、破損画像、非対応形式 | 原本内容を確認し、再アップロードまたは非対応として扱う |
| WebP変換失敗 | 対象ページ、画像形式、リソース不足 | 設定とリソースを確認し、必要なら再変換する |
| タイムアウト | 1ジョブ30分超過 | ページ数、解像度、ホスト負荷を確認する |
| PostgreSQL更新失敗 | DB接続、制約違反、トランザクション失敗 | PostgreSQLを正として生成物の残存を確認する |

再変換は既存ジョブを直接戻さず、新しい変換ジョブを作成する方式を原則とする。途中生成物が残っている場合は、PostgreSQLへ登録されている生成物だけを閲覧対象とし、未登録生成物はクリーンアップ候補にする。

## Elasticsearch再インデックス手順

Elasticsearchは派生データであり、PostgreSQLから再構築できるものとして扱う。

再インデックス前に次を確認する。

- PostgreSQLが利用可能であること。
- 対象書籍、著者、タグ、シリーズ、種別の正本データがPostgreSQL上で期待どおりであること。
- Elasticsearchへの接続が可能であること。
- 実行中に検索結果が一時的に古くなる、または揺れる可能性があることを関係者へ共有していること。

### 書籍単位再インデックス

特定書籍の検索結果、サムネイル参照、表示状態が疑わしい場合に実行する。

1. 対象書籍IDを確認する。
2. PostgreSQL上の書籍、著者、タグ、シリーズ、種別、表示状態、論理削除状態を確認する。
3. 書籍単位の再インデックス管理コマンドを実行する。
4. Elasticsearch上の対象ドキュメントが更新されたことを確認する。
5. 検索APIまたは画面で検索結果を確認する。

実装時のコマンド例は次の形式を候補とする。

```bash
docker compose exec api ./app-admin search reindex-book --book-id <book-id>
```

### 全件再インデックス

インデックス定義変更、Elasticsearch障害、整合性不明時に実行する。

1. PostgreSQLが正しい状態であることを確認する。
2. 必要に応じてメンテナンス時間を確保する。
3. 全件再インデックス管理コマンドを実行する。
4. 処理件数、失敗件数、スキップ件数を確認する。
5. 主要な検索条件で検索結果を確認する。
6. 失敗した書籍がある場合は、書籍単位再インデックスまたは原因調査を行う。

実装時のコマンド例は次の形式を候補とする。

```bash
docker compose exec api ./app-admin search reindex-all
```

Elasticsearchの内容が信用できない場合は、インデックスを破棄してPostgreSQLから再構築する。ただし、破棄手順、alias切り替え、検索停止時間は実装時のインデックス設計で確定してからこのRunbookへ追記する。

## ストレージ確認手順

書籍ファイル保存領域と変換ワーカー作業領域の容量を確認する。

```bash
df -h
```

容量逼迫時は次を確認する。

- 原本ファイル、変換済みWebP、サムネイルの増加量。
- 変換ワーカーのジョブ専用作業ディレクトリの残存。
- 削除済み書籍の退避領域。
- Elasticsearch、PostgreSQLのデータ領域。

バックアップは行わないため、容量確保のためにデータを削除する場合は、対象がPostgreSQL上の正本と一致していることを確認してから行う。

## 障害時の記録

障害や運用上の重要な判断は、障害ログテンプレートへ記録する。

記録対象の例は次のとおり。

- 本番環境の停止または再起動を伴う障害。
- 原本ファイル消失。
- PostgreSQLの整合性問題。
- Elasticsearch全件再インデックスを必要とする不整合。
- 変換ジョブ失敗の大量発生。
- 7-Zip外部プロセスの失敗増加。
- ストレージ容量逼迫。

## 更新方針

Docker Composeサービス名、ヘルスチェックURL、管理コマンド、再インデックス方式、ジョブ状態、ログ出力先が実装で確定または変更された場合は、このRunbookを更新する。

# Runbook初版

## 目的

このRunbookは、自炊本閲覧Webアプリケーションを単一Linuxホスト上のDocker Compose構成で運用するための初期手順を整理する。

対象は、起動、停止、再起動、ログ確認、変換ジョブ失敗時の確認、検索更新Outbox確認、Elasticsearch再インデックスである。

## 前提

- 本番運用環境は単一Linuxホスト上のDocker Compose構成とする。
- Next.jsフロントエンド、Spring BootバックエンドAPI、Spring Boot変換ワーカー、PostgreSQL、Elasticsearch、RabbitMQを同一ホストに配置する。
- PostgreSQLをメタ情報、ユーザ、権限、ジョブ状態、検索更新Outboxの正本とする。
- ElasticsearchはPostgreSQLから再構築可能な派生データとして扱う。
- 検索更新状態はPostgreSQLの`search_index_outbox`を正本とし、RabbitMQは再試行通知、配送手段として扱う。
- Elasticsearch必須プラグインは [技術スタック](../03_architecture/02_technology_stack.md#elasticsearch必須プラグイン) を正本とする。
- 原本ファイルは通常運用では保存し続ける。
- 変換済みWebPとサムネイルは、原本ファイルと変換条件から再生成可能な派生ファイルとして扱う。
- バックアップは行わない。
- バックアップなし運用のRPO / RTO、復旧不能リスク、見直し条件は [バックアップなし方針](02_backup_restore.md) を正本とする。

## 運用対象コンポーネント

| コンポーネント | 役割 | 主な確認観点 |
| --- | --- | --- |
| Next.jsフロントエンド | 一般ユーザ画面、管理ユーザ画面を提供する | HTTP応答、API接続 |
| Spring BootバックエンドAPI | 認証、書籍管理、検索、閲覧、管理操作を提供する | HTTP応答、DB接続、Elasticsearch接続、RabbitMQ接続 |
| Spring Boot変換ワーカー | アーカイブ展開、WebP変換、サムネイル生成を実行する | RabbitMQ取得、ジョブ状態、7-Zip実行、保存領域アクセス |
| PostgreSQL | 正本データを保持する | 接続、容量、ジョブ状態、検索更新状態 |
| Elasticsearch | 検索用派生インデックスを保持する | 接続、インデックス状態、検索応答 |
| RabbitMQ | APIと変換ワーカーを非同期に接続する | キュー滞留、未ack、dead letter、再配送 |
| 書籍ファイル保存領域 | 原本、WebP、サムネイルを保存する | 容量、権限、欠落、未参照ファイル |

## 事前確認

本番ホストで作業する前に次を確認する。

- 操作対象ホストが本番ホストであること。
- 作業者がDocker Compose、アプリケーション設定、ログ、保存領域を確認できる権限を持つこと。
- `.env`などの環境別設定が配置済みで、秘密情報がGit管理されていないこと。
- 書籍ファイル保存領域、変換ワーカー作業領域、PostgreSQL、Elasticsearchのデータ領域がDocker Composeから参照できること。
- Elasticsearchイメージまたはコンテナに技術スタックで定義された必須プラグインが導入されていること。
- 7-Zip for Linuxコンソール版が変換ワーカーコンテナ内から実行できること。

## 禁止操作

バックアップなし運用では、次の操作を通常運用、容量確保、障害一次対応の手段として実行しない。

- `docker compose down -v`
- `docker volume rm`、`docker volume prune`などのDocker volume削除
- PostgreSQLデータ領域の削除
- 書籍ファイル保存領域の削除
- 原本ファイルを含む保存領域への一括削除
- 正本確認なしの未参照ファイル一括削除

これらの操作は、PostgreSQL、認証、権限、ジョブ状態、原本ファイルの復旧不能な消失につながる。実施が避けられない場合は、作業目的、対象、影響範囲、復旧不能になるデータ、承認者、実施時刻を障害ログに記録し、利用者へ事前または事後に説明する。

## 起動手順

1. 本番ホストへログインする。
2. デプロイ対象ディレクトリへ移動する。
3. 環境変数とDocker Compose設定を確認する。
4. ミドルウェアとアプリケーションを起動する。

```bash
docker compose up -d
```

5. コンテナ状態を確認する。

```bash
docker compose ps
```

6. バックエンドAPIのヘルスチェックを確認する。
7. 変換ワーカーがRabbitMQへ接続できていることをログで確認する。
8. PostgreSQL、Elasticsearch、RabbitMQへの接続エラーが出ていないことを確認する。
9. API起動時またはインデックス作成前のElasticsearch必須プラグイン確認が成功していることを確認する。
10. 管理画面にログインできることを確認する。

ヘルスチェックURL、管理コマンド、サービス名は実装時のDocker Compose定義に合わせてこのRunbookへ追記する。

## 停止手順

1. 実行中の変換ジョブ数とキュー滞留を確認する。
2. 急ぎでない停止の場合は、新規アップロードや変換ジョブ投入を止め、実行中ジョブの完了を待つ。
3. コンテナを停止する。

```bash
docker compose down
```

4. 停止後、コンテナ状態を確認する。

```bash
docker compose ps
```

変換中に停止した場合、ジョブ状態が実行中のまま残る可能性がある。再起動後にジョブ状態、キュー状態、作業ディレクトリ残存を確認し、必要に応じて再実行または失敗扱いへ整理する。

## 再起動手順

通常の再起動は、停止手順と起動手順を順に実行する。

```bash
docker compose down
docker compose up -d
```

設定変更を反映する場合は、対象サービスのイメージ再作成やコンテナ再作成が必要かを確認する。PostgreSQLとElasticsearchのデータボリュームは削除しない。

再起動時も [禁止操作](#禁止操作) に該当するvolume削除は行わない。

## ログ確認手順

全体ログを確認する。

```bash
docker compose logs --tail=200
```

特定サービスのログを確認する。

```bash
docker compose logs --tail=200 api
docker compose logs --tail=200 worker
docker compose logs --tail=200 frontend
docker compose logs --tail=200 postgres
docker compose logs --tail=200 elasticsearch
docker compose logs --tail=200 rabbitmq
```

継続監視する場合は次を使用する。

```bash
docker compose logs -f worker
```

ログには、パスワード、トークン、セッションID、秘密鍵、不要な個人情報、内部物理パスを出さない。障害調査で必要な場合も、共有前に秘密情報が含まれていないことを確認する。

## 変換ジョブ失敗時の確認手順

変換ジョブが失敗した場合は、PostgreSQLのジョブ状態を正として確認する。

1. 管理画面または管理コマンドで対象ジョブを特定する。
2. `conversion_job.status`、`failure_phase`、`failure_code`、`failure_message`、`external_exit_code`、`timed_out`を確認する。
3. ワーカーログで同じジョブIDの処理を確認する。
4. RabbitMQのキュー滞留、未ack、dead letter queueを確認する。
5. 原本ファイル管理情報と物理ファイルの存在を確認する。
6. ジョブ専用作業ディレクトリが残っている場合は、保存領域外のファイルを参照していないことを確認する。
7. 失敗原因を分類する。

| 失敗分類 | 確認観点 | 初期対応 |
| --- | --- | --- |
| 原本ファイル欠落 | PostgreSQLに管理情報があり、物理ファイルがない | 原本消失として障害ログへ記録する |
| アーカイブ展開失敗 | 7-Zip終了コード、暗号化、破損、非対応形式 | ユーザに安全なエラーとして扱い、必要なら再アップロードを案内する |
| パス検証失敗 | 絶対パス、`..`、シンボリックリンク | 不正または危険なアーカイブとして失敗扱いにする |
| 画像判定失敗 | 対応画像なし、破損画像、非対応形式 | 原本内容を確認し、再アップロードまたは非対応として扱う |
| WebP変換失敗 | 対象ページ、画像形式、リソース不足 | 設定とリソースを確認し、必要なら再変換する |
| タイムアウト | 1ジョブ30分超過 | ページ数、解像度、ホスト負荷を確認する |
| 安全上限超過 | アップロード1ファイル、展開後総量、エントリ数、画像ピクセル数、1ジョブ一時領域、同時アップロード数 | 設定値と原本内容を確認し、必要なら安全上限または運用手順を見直す |
| PostgreSQL更新失敗 | DB接続、制約違反、トランザクション失敗 | PostgreSQLを正として生成物の残存を確認する |

再変換は既存ジョブを直接戻さず、新しい変換ジョブを作成する方式を原則とする。途中生成物が残っている場合は、PostgreSQLへ登録されている生成物だけを閲覧対象とし、未登録生成物はクリーンアップ候補にする。

RabbitMQのメッセージがdead letter queueへ送られている場合は、PostgreSQL上の`conversion_job`を確認し、必要に応じて管理操作から新しい再実行ジョブを作成する。dead letter queueのメッセージを直接再投入する場合も、PostgreSQLの状態と対象ジョブIDを確認してから行う。

## Elasticsearch再インデックス手順

Elasticsearchは派生データであり、PostgreSQLから再構築できるものとして扱う。

再インデックス前に次を確認する。

- PostgreSQLが利用可能であること。
- 対象書籍、著者、タグ、シリーズ、種別の正本データがPostgreSQL上で期待どおりであること。
- Elasticsearchへの接続が可能であること。
- 技術スタックで定義されたElasticsearch必須プラグインが導入済みであること。
- 実行中に検索結果が一時的に古くなる、または揺れる可能性があることを関係者へ共有していること。

### 検索更新Outbox確認

検索対象のメタ情報更新や変換完了後に検索結果が反映されない場合は、PostgreSQLの`search_index_outbox`を正として確認する。

1. 管理画面または管理コマンドで対象書籍IDを特定する。
2. `search_index_outbox.status`、`target_type`、`target_id`、`book_id`、`operation`、`attempt_count`、`next_attempt_at`、`last_error_code`、`last_error_message`を確認する。
3. API、変換ワーカー、再試行ワーカーのログで同じOutbox IDまたは書籍IDの処理を確認する。
4. RabbitMQの検索更新再試行通知キューの滞留、未ack、dead letterを確認する。
5. Elasticsearch接続、必須プラグイン、対象インデックスの状態を確認する。
6. PostgreSQLの最新正本から書籍単位再インデックスできるか判断する。

| Outbox状態 | 初期対応 |
| --- | --- |
| `pending` | 即時更新が未完了の可能性がある。APIまたはWorkerログを確認し、必要に応じて再試行通知を送る。 |
| `processing` | 処理中または中断の可能性がある。更新時刻とワーカー死活を確認する。長時間残る場合は再試行待ちへ整理する。 |
| `retry_waiting` | `next_attempt_at`とRabbitMQ通知を確認する。通知が失われている場合はOutbox IDを再通知する。 |
| `dead_letter` | 自動再試行上限に達している。失敗理由を確認し、原因解消後に書籍単位再インデックスまたは手動再試行を行う。 |
| `completed` | 検索結果が古い場合は、別の未完了Outbox、alias、インデックス定義、検索条件を確認する。 |

RabbitMQのメッセージだけを根拠に検索更新を完了扱いにしない。再試行ワーカーはOutbox IDからPostgreSQLを読み直し、最新正本からElasticsearchドキュメントを再生成する。

### 必須プラグイン確認と復旧

Elasticsearchインデックス作成前、API起動時、または再インデックス前に、[技術スタックで定義された必須プラグイン](../03_architecture/02_technology_stack.md#elasticsearch必須プラグイン) を確認する。

```bash
docker compose exec elasticsearch bin/elasticsearch-plugin list
docker compose exec elasticsearch curl -s http://localhost:9200/_cat/plugins
```

必須プラグインに不足がある場合、インデックス作成は失敗させる。復旧は次の方針で行う。

1. Docker Compose定義またはElasticsearch用Dockerfileで、技術スタックで定義された必須プラグインを導入する。
2. Elasticsearchコンテナを再作成する。既存インデックス定義とプラグイン状態が不一致の場合は、PostgreSQLを正として再インデックスする。
3. `_cat/plugins`で必須プラグインが表示されることを確認する。
4. API起動時またはインデックス作成前の必須プラグイン確認を再実行する。
5. 必要に応じて書籍単位または全件再インデックスを実行する。

### 書籍単位再インデックス

特定書籍の検索結果、サムネイル参照、表示状態が疑わしい場合に実行する。

1. 対象書籍IDを確認する。
2. PostgreSQL上の書籍、著者、タグ、シリーズ、種別、表示状態、論理削除状態を確認する。
3. 書籍単位の再インデックス管理コマンドを実行する。
4. Elasticsearch上の対象ドキュメントが更新されたことを確認する。
5. 検索APIまたは画面で検索結果を確認する。

実装時のコマンド例は次の形式を候補とする。

```bash
docker compose exec api ./app-admin search reindex-book --book-id <book-id>
```

### 全件再インデックス

インデックス定義変更、Elasticsearch障害、整合性不明時に実行する。

1. PostgreSQLが正しい状態であることを確認する。
2. 必要に応じてメンテナンス時間を確保する。
3. 全件再インデックス管理コマンドを実行する。
4. 処理件数、失敗件数、スキップ件数を確認する。
5. 主要な検索条件で検索結果を確認する。
6. 失敗した書籍がある場合は、書籍単位再インデックスまたは原因調査を行う。

実装時のコマンド例は次の形式を候補とする。

```bash
docker compose exec api ./app-admin search reindex-all
```

Elasticsearchの内容が信用できない場合は、インデックスを破棄してPostgreSQLから再構築する。ただし、破棄手順、alias切り替え、検索停止時間は実装時のインデックス設計で確定してからこのRunbookへ追記する。

## ストレージ確認手順

書籍ファイル保存領域と変換ワーカー作業領域の容量を確認する。

```bash
df -h
```

容量逼迫時は次を確認する。

- 原本ファイル、変換済みWebP、サムネイルの増加量。
- 変換ワーカーのジョブ専用作業ディレクトリの残存。
- 安全上限を超える変換ジョブや同時実行数引き上げが行われていないか。
- 削除済み書籍の退避領域。
- Elasticsearch、PostgreSQLのデータ領域。

バックアップは行わないため、容量確保のためにデータを削除する場合は、対象がPostgreSQL上の正本と一致していることを確認してから行う。

安全上限の具体値は [画像変換設計のリソース制限と設定](../04_design/07_image_conversion_design.md#リソース制限と設定) を正本とする。変換ワーカー同時実行数を引き上げる場合は、単一ホストのCPU、メモリ、一時ディスク、I/Oを確認してから変更する。

## 削除系管理コマンドの実行手順

書籍削除、未参照ファイル削除、失敗ジョブ一時ファイル削除、Elasticsearchインデックス削除など、データを削除する管理コマンドは次の手順で実行する。

1. 対象操作がPostgreSQLの正本削除か、派生データの削除かを区別する。
2. dry-runで対象件数、対象ID、対象種別、対象パスの要約を確認する。
3. PostgreSQL上の正本状態と物理ファイルの対応を確認する。
4. 対象件数、対象範囲、復旧不能になるデータを作業記録に残す。
5. 実削除オプションを指定する。
6. 対象件数または確認語句による二段階確認を行う。
7. 実行結果、削除件数、失敗件数、実行者、実行日時を記録する。
8. 必要に応じてElasticsearch再インデックス、閲覧確認、検索確認を行う。

実装時の削除系管理コマンドは、既定でdry-runとし、実削除は明示オプションと二段階確認がそろった場合のみ実行できるようにする。確認表示では秘密情報や内部物理パスの詳細を不要に出さず、運用判断に必要な範囲へ要約する。

## 障害時の記録

障害や運用上の重要な判断は、障害ログテンプレートへ記録する。

記録対象の例は次のとおり。

- 本番環境の停止または再起動を伴う障害。
- 原本ファイル消失。
- PostgreSQLの整合性問題。
- Elasticsearch全件再インデックスを必要とする不整合。
- 変換ジョブ失敗の大量発生。
- 7-Zip外部プロセスの失敗増加。
- ストレージ容量逼迫。

## 障害時の利用者向け説明方針

障害時に利用者へ説明する場合は、原因が未確定の情報と確定情報を分け、復旧できる範囲と復旧できない範囲を明確に伝える。バックアップなし運用により復旧不能なデータがある場合は、バックアップ復元を予定しているように説明しない。

説明に含める項目は次のとおり。

- 発生日時または検知日時。
- 影響を受けた機能。
- 影響を受けた可能性があるデータ。
- 復旧済み、対応中、復旧不能の区分。
- 利用者に依頼する操作。例: 再アップロード、再ログイン、対象書籍の確認。
- 今後の再発防止または方針見直しの予定。

説明テンプレートは次の形式を候補とする。

```markdown
YYYY-MM-DD HH:mm 頃から、<影響機能>に影響する障害を確認しました。

現在の確認状況:
- 影響範囲: <影響範囲>
- 復旧できた内容: <復旧済み内容>
- 復旧できない可能性がある内容: <復旧不能または確認中の内容>

この環境は初期運用方針としてバックアップを取得していないため、PostgreSQLまたは原本ファイルが失われた場合、バックアップからの復元はできません。必要に応じて、対象書籍の再アップロードまたは設定の再登録をお願いします。

追加の確認結果と再発防止策は、確認でき次第共有します。
```

## 更新方針

Docker Composeサービス名、ヘルスチェックURL、管理コマンド、再インデックス方式、削除系管理コマンド、ジョブ状態、ログ出力先が実装で確定または変更された場合は、このRunbookを更新する。

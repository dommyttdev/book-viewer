# ユーザーストーリー: ロール設定

## 目的

管理ユーザのロールと権限を整理し、`super_admin`, `admin`, `operator`, `viewer` の役割に応じて管理操作の可否を制御できるようにする。

## 背景

管理機能では、管理ユーザ管理、一般ユーザ管理、書籍管理、変換ジョブ操作、参照のみの操作を分ける必要がある。初版では固定ロールを基本とし、ロールと権限の対応は PostgreSQL の `role`, `permission`, `admin_user_role`, `role_permission` を正本として扱う。

ロール設定の変更はシステム全体の権限境界に影響するため、`super_admin` のみが実行できる。MVPでは固定ロールの確認を優先し、ロール設定変更APIは Beta / v1.0 候補として扱う。

## 利用者

- 主利用者: `super_admin`
- 関係者: 管理ユーザ、運用者

## ストーリー

### AD-US-011: super_admin としてロール一覧を確認したい

`super_admin` として、`super_admin`, `admin`, `operator`, `viewer` のロールと付与権限を確認したい。なぜなら、管理ユーザにどの権限を割り当てるべきか判断したいから。

### AD-US-012: super_admin としてロールに権限を設定したい

`super_admin` として、ロールごとの権限を設定したい。なぜなら、運用フェーズに応じて管理操作の範囲を調整したいから。

### AD-US-013: 管理ユーザとして自分の権限範囲を把握したい

管理ユーザとして、自分に許可された管理画面と操作だけを利用したい。なぜなら、権限外の操作を誤って実行せず、自分の担当作業に集中したいから。

### AD-US-014: viewer として管理情報を参照したい

`viewer` として、書籍、一般ユーザ、変換ジョブ状態を参照したい。なぜなら、更新権限を持たずに状況確認や問い合わせ対応を行いたいから。

### AD-US-015: operator として取込と変換運用を行いたい

`operator` として、書籍アップロード、変換ジョブ確認、再実行、再インデックスを行いたい。なぜなら、日常的な取込と変換失敗の復旧を担当したいから。

## 対象範囲

- 管理ロール一覧
- ロールと権限の対応確認
- ロール設定変更候補
- 管理ユーザへのロール割り当て
- 複数ロール付与時の権限和集合
- 管理APIの権限コード設計
- フロントエンド表示制御とサーバ側権限確認の同期

## 対象外

- 一般ユーザ向け権限の詳細なロール分割
- 組織、グループ、チーム単位の権限管理
- カスタムロール作成UI
- 属性ベースアクセス制御
- 外部IDプロバイダ側グループとの同期

## 受入条件

- `super_admin`, `admin`, `operator`, `viewer` のロール候補が定義されている。
- ロール別の操作可否は権限マトリクスで確認できる。
- `super_admin` だけがロール設定を変更できる。
- 複数ロールを持つ管理ユーザは、付与ロールの許可の和集合で操作可否を判断できる。
- `viewer` は参照専用で、更新系操作を実行できない。
- `operator` は書籍アップロード、変換ジョブ確認、再実行、再インデックスを実行できる。
- `admin` は一般ユーザ管理と日常的な書籍管理を実行できるが、管理ユーザ管理とロール設定は実行できない。
- ロール設定や権限不足はバックエンドAPI側で判定できる。
- 権限コードは `role_permission` で管理できる粒度になっている。

# ユーザーストーリー: ログイン・ログアウト

## 目的

一般ユーザがメールアドレス、パスワード、メールによる2段階認証で安全にログインし、利用終了時にログアウトできるようにする。

## 背景

ログインは、会員登録済みかつメール確認済みの一般ユーザに限定する。初版ではHTTP onlyかつSecure属性付きCookieによるサーバ側セッションを第一候補とし、JWTはフロントエンドとAPIの分離要件が強くなった場合の検討対象とする。

ログイン時はパスワード認証だけで完了させず、メールを利用した2段階認証を行う。認証失敗、2段階認証失敗、過剰な再試行にはレート制限または一時ロックを適用し、アカウント存在や内部状態を推測されにくい応答にする。

## 利用者

- 主利用者: 一般ユーザ
- 関係者: 未ログイン利用者

## ストーリー

### AM-US-005: 一般ユーザとしてログインしたい

一般ユーザとして、メールアドレスとパスワードを入力してログインを開始したい。なぜなら、自分の閲覧履歴、お気に入り、会員情報を利用したいから。

### AM-US-006: 一般ユーザとしてメール2段階認証を完了したい

一般ユーザとして、メールで届くワンタイムコードまたは確認リンクで2段階認証を完了したい。なぜなら、パスワードだけに依存せず安全にログインしたいから。

### AM-US-007: 一般ユーザとしてログアウトしたい

一般ユーザとして、利用終了時に現在のセッションを無効化したい。なぜなら、共有端末や離席時に自分のアカウントを守りたいから。

### AM-US-008: 一般ユーザとしてパスワードをリセットしたい

一般ユーザとして、パスワードを忘れた場合にメールを使ってリセットしたい。なぜなら、本人確認を行ったうえでアカウント利用を再開したいから。

### AM-US-009: 一般ユーザとしてログイン失敗時に保護されたい

一般ユーザとして、ログイン失敗が繰り返された場合に一時的な制限をかけてほしい。なぜなら、総当たり攻撃や認証情報の悪用からアカウントを守りたいから。

## 対象範囲

- メールアドレスとパスワードによるログイン開始
- メールを利用したログイン時2段階認証
- サーバ側セッションの発行と無効化
- ログアウト
- パスワードリセット要求と確認
- ログイン失敗制限、再送制限、一時ロック
- メール未確認、停止、退会済みユーザのログイン制御

## 対象外

- 外部IDプロバイダ連携
- 端末ごとのセッション一覧管理
- 常時ログイン端末の個別失効
- 管理ユーザログイン

## 受入条件

- メール確認済みで有効な一般ユーザだけがログインを完了できる。
- パスワード認証成功後、メールによる2段階認証を要求できる。
- 2段階認証のコードまたは確認リンクは有効期限と使い捨て性を持つ。
- ログイン成功時はHTTP onlyかつSecure属性付きCookieを第一候補としてセッションを発行できる。
- ログアウト時は現在のセッションを無効化できる。
- パスワードリセットは登録有無を推測されにくい応答で要求できる。
- ログイン失敗、2段階認証失敗、再送要求にはレート制限または一時ロックを適用できる。
- 認証失敗理由は、アカウント存在や内部状態を過度に漏らさない。

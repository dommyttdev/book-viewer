# 受入条件: 本削除

## 対象ストーリー

- `BD-US-001`: 管理ユーザとして本を削除する
- `BD-US-002`: 一般ユーザとして削除済みの本が表示されない
- `BD-US-003`: 運用者として派生ファイルを後から整理する

## 受入条件

| ID | 条件 | 確認観点 |
| --- | --- | --- |
| BD-AC-001 | `book.delete`権限を持つ管理ユーザだけが本削除APIを実行できる。 | 管理ユーザ、権限不足の管理ユーザ、一般ユーザ、未ログインで結果が分かれること。 |
| BD-AC-002 | 一般ユーザは本削除APIを実行できない。 | 403相当の認可エラーとなり、対象本の状態を変更しない。 |
| BD-AC-003 | 本削除は初版ではPostgreSQL上の論理削除として扱う。 | `book.deleted_at`または同等の状態を設定し、通常の一覧、検索、閲覧対象から除外する。 |
| BD-AC-004 | 削除済みの本は一般向け本一覧に表示されない。 | `GET /api/v1/books`で返らない。 |
| BD-AC-005 | 削除済みの本は一般向け検索結果に表示されない。 | Elasticsearchが未反映でもAPI側でPostgreSQLを正として表示対象外にできる。 |
| BD-AC-006 | 削除済みの本は一般向け閲覧APIで読めない。 | 存在秘匿または閲覧不可として安全に扱う。 |
| BD-AC-007 | 削除時に原本ファイルを同一トランザクションで物理削除しない。 | 通常運用では原本を保存し続け、削除ジョブまたは管理コマンドの対象にできる。 |
| BD-AC-008 | 変換済みWebPとサムネイルは削除後に閲覧、表示対象外にする。 | 物理削除は削除ジョブまたは管理コマンドで実行する。 |
| BD-AC-009 | Elasticsearchの該当ドキュメントは削除または再インデックスによりPostgreSQLと整合させる。 | Elasticsearchは派生データとして再構築可能にする。 |
| BD-AC-010 | 物理削除処理を実装する場合、削除対象を誤らないよう書籍ID単位で分離された保存キーを使う。 | `trash/`退避などの安全策を検討する。 |
| BD-AC-011 | 削除操作の失敗時、論理削除状態と派生データの整合性を確認できる。 | 失敗理由は安全な範囲で記録し、内部パスや秘密情報を露出しない。 |
| BD-AC-012 | 管理向け一覧や詳細では、権限に応じて削除済み状態を確認できる。 | 運用確認のため、通常利用者向け表示とは分ける。 |

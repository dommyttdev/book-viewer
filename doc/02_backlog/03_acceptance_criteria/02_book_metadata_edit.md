# 受入条件: 本メタ情報編集

## 対象ストーリー

- `BM-US-001`: 管理ユーザとして基本メタ情報を編集する
- `BM-US-002`: 管理ユーザとして公開状態を制御する
- `BM-US-003`: 管理ユーザとして検索更新の再試行余地を保つ

## 受入条件

| ID | 条件 | 確認観点 |
| --- | --- | --- |
| BM-AC-001 | `book.update`権限を持つ管理ユーザだけが本メタ情報を更新できる。 | 管理ユーザ、権限不足の管理ユーザ、一般ユーザ、未ログインで結果が分かれること。 |
| BM-AC-002 | 一般ユーザは本メタ情報更新APIを実行できない。 | 403相当の認可エラーとなり、PostgreSQLを更新しない。 |
| BM-AC-003 | タイトルは必須とし、空文字または空白のみを拒否する。 | バリデーションエラーとなり、既存メタ情報を維持する。 |
| BM-AC-004 | 説明、著者、タグ、シリーズ、シリーズ内順序、種別を更新できる。 | APIモデルからドメイン更新へ明示的に変換し、永続化モデルを直接公開しない。 |
| BM-AC-005 | 存在しない著者ID、タグID、シリーズID、種別コードを指定した場合は拒否する。 | 参照整合性を保ち、部分更新で不整合を残さない。 |
| BM-AC-006 | シリーズ未設定の場合、シリーズ内順序だけを有効な表示条件として扱わない。 | 実装時はバリデーションエラーまたは無視ではなく、仕様に沿って明示的に扱う。 |
| BM-AC-007 | 公開状態を`draft`, `available`, `hidden`などの許可された状態へ更新できる。 | 一般向け一覧、検索、閲覧対象は公開可能な状態に限定する。 |
| BM-AC-008 | 変換未完了または変換失敗状態の本は、メタ情報が更新されても一般閲覧対象にならない。 | 公開状態だけでなく変換状態も閲覧可否に反映する。 |
| BM-AC-009 | 検索対象項目が更新された場合、Elasticsearch更新を行うか、再試行可能なインデックス更新状態を残す。 | PostgreSQLを正本とし、Elasticsearch更新失敗でメタ情報更新全体を失わない。 |
| BM-AC-010 | 更新成功時、管理本詳細で更新後の値を確認できる。 | APIレスポンスまたは再取得結果がPostgreSQLの正本と一致する。 |
| BM-AC-011 | 更新失敗時、ユーザに安全なエラーを返し、秘密情報、内部パス、スタックトレースを露出しない。 | ログにも不要な個人情報やシークレットを出さない。 |
| BM-AC-012 | 同時更新が想定される場合、実装時に楽観ロックまたは更新競合の扱いを明確にする。 | 上書き事故を避ける設計を検討する。 |

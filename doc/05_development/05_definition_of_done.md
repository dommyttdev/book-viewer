# Definition of Done

## 目的

このドキュメントは、自炊本閲覧Webアプリケーションにおいて、ユーザーストーリー、タスク、バグ修正、ドキュメント更新を「完了」と判断するための共通基準を定義する。

Definition of Doneは、作業者個人の確認メモではなく、レビュー、テスト、リリース判断で共通して使う品質基準として扱う。機能の性質や変更範囲に応じて必要な確認は増減するが、このドキュメントに記載した観点を確認せずに完了扱いにしない。

## 前提

- 詳細な実装ルールは [rules/CODING_STANDARDS.md](../../rules/CODING_STANDARDS.md) に従う。
- コミット、ブランチ、プルリクエスト、検証結果の扱いは [rules/CONTRIBUTING.md](../../rules/CONTRIBUTING.md) に従う。
- 関連する計画や未完了作業は [doc/TODO.md](../TODO.md) を確認する。
- 受入条件が存在する作業では、該当する [doc/02_backlog/03_acceptance_criteria/](../02_backlog/03_acceptance_criteria/) 配下の文書を基準にする。
- 重要な仕様判断やアーキテクチャ判断は、TODOやコードコメントだけでなくADRまたは設計ドキュメントに記録する。

## 共通の完了条件

すべての作業は、次の条件を満たしたときに完了とみなす。

- 作業の目的が明確で、依頼内容または該当TODOの範囲を満たしている。
- 関連する受入条件がある場合、その条件を満たしている。
- 変更範囲が必要最小限で、無関係な変更を含んでいない。
- 命名、責務分離、エラー処理、ログ、権限確認、テストの観点が確認されている。
- 関連ドキュメントが更新されている。
- [doc/TODO.md](../TODO.md) の該当ステータスが実態に合わせて更新されている。
- レビュー担当者が変更意図、影響範囲、検証結果を追える状態になっている。

## 機能実装の完了条件

機能追加や仕様変更を完了とするには、共通条件に加えて次を満たす。

### 受入条件

- ユーザーストーリーまたはタスクに対応する受入条件を満たしている。
- 受入条件が不足している場合は、必要に応じて受入条件文書またはTODOを更新している。
- MVP、Beta、v1.0、将来対応の範囲を越える仕様を勝手に実装していない。

### 設計と責務分離

- ビジネスロジックがコントローラやインフラ層に漏れていない。
- APIモデル、永続化エンティティ、ドメインモデルが概念的に分離されている。
- API、変換ワーカー、ミドルウェアを将来分離できるよう、境界を不必要に結合していない。
- PostgreSQLを正本として扱い、Elasticsearchを再構築可能な派生データとして扱っている。
- 設定値、パス、外部コマンド、品質値、同時実行数などを環境固有の値としてハードコードしていない。

### セキュリティと権限

- 外部入力、アップロードファイル、アーカイブ内容、ファイルパス、ジョブパラメータを検証している。
- 管理ユーザのみが実行できる操作に権限確認が実装されている。
- 一般ユーザが管理操作や書籍アップロードを実行できないことを確認している。
- パストラバーサル、想定外のファイル形式、破損ファイル、過大な入力への扱いを確認している。
- パスワード、トークン、シークレット、セッション識別子、不要な個人情報をログやレスポンスへ出力していない。

### ログとエラー処理

- 正常系の主要なライフサイクルイベント、長時間処理、ジョブ状態変化を追跡できる。
- 失敗時に調査に必要な安全な情報がログまたは状態として残る。
- バリデーションエラー、認証エラー、認可エラー、外部依存の失敗、予期しないシステムエラーを区別して扱っている。
- 変換、アーカイブ展開、検索インデックス更新の失敗は、必要に応じて再試行または失敗状態として扱える。
- 同じ例外を複数レイヤーで重複ログ出力していない。

## テストの完了条件

コード変更を含む作業は、変更範囲とリスクに応じて次を確認する。

- 新しい振る舞いを実装する前に、期待する振る舞いをテストまたは受入条件として明確にしている。
- 自動化しやすい振る舞いでは、TDDの流れに沿って失敗するテストを先に追加している。
- 単体テストが通っている。
- 変更したビジネスルール、権限確認、状態遷移、バリデーションに対するテストを追加または更新している。
- 主要な正常系を確認している。
- 主要な異常系を確認している。
- 境界値、未認証、権限不足、存在しないリソース、重複、競合、外部依存失敗など、該当するリスクを確認している。
- PostgreSQL、Elasticsearch、ファイルシステム、7-Zip外部プロセス、RabbitMQなどの外部依存は、必要に応じてモック、統合テスト、手動確認のいずれかで検証している。
- フロントエンド変更では、主要画面、入力エラー、レスポンシブ表示、権限別表示を確認している。
- 実行したテストや手動確認の結果を、プルリクエストまたは作業記録に残している。

## ドキュメントの完了条件

仕様、設計、運用、テスト、権限、API、データ構造に影響する変更では、関連ドキュメントを同じ作業で更新する。

更新対象の例は次のとおり。

- プロダクト範囲や優先度を変える場合: [doc/01_product/](../01_product/)
- エピック、ユーザーストーリー、受入条件を変える場合: [doc/02_backlog/](../02_backlog/)
- 技術選定やアーキテクチャ判断を変える場合: [doc/03_architecture/](../03_architecture/) またはADR
- UI、API、データモデル、検索、ファイル保存、画像変換、権限を変える場合: [doc/04_design/](../04_design/)
- 開発手順や完了基準を変える場合: [doc/05_development/](../05_development/)
- テスト方針、受入テスト、回帰テストを変える場合: [doc/06_testing/](../06_testing/)
- 運用手順、監視、障害対応、リリースノートを変える場合: [doc/07_operations/](../07_operations/)

ドキュメントのみの変更では、Markdownの見出し構造、リンク、パス、TODOステータスが正しいことを確認する。

## レビュー前チェックリスト

レビューに出す前に、作業者は次を確認する。

- 依頼内容またはTODOに対する完了範囲が説明できる。
- 受入条件を満たしている。
- 単体テストまたは必要な確認を実施した。
- 主要な異常系を確認した。
- ログ、エラー処理、権限確認を確認した。
- セキュリティ上の入力検証と秘密情報の扱いを確認した。
- 関連ドキュメントを更新した。
- [doc/TODO.md](../TODO.md) のステータスを更新した。
- 既知の未対応事項、制約、リスクがあればTODO、設計文書、またはプルリクエストに明記した。

## 完了扱いにしない例

次の状態では完了扱いにしない。

- 正常系だけ動作し、主要な異常系が未確認である。
- 単体テストが失敗している、または必要なテストを実行していない理由が記録されていない。
- 権限確認が画面表示だけに依存し、バックエンドAPI側で確認されていない。
- エラーが握りつぶされ、利用者や運用者が失敗状態を追跡できない。
- PostgreSQLとElasticsearchの責務が曖昧になり、Elasticsearchを正本のように扱っている。
- 仕様変更がコードだけに反映され、設計ドキュメント、受入条件、TODOが更新されていない。
- 重要な判断がコメントや会話だけに残り、ADRまたは設計ドキュメントに記録されていない。
- 無関係なリファクタリングやフォーマット変更が混ざり、レビュー範囲が不明確になっている。

## 更新方針

- Definition of Doneは、開発の進行に合わせて継続的に見直す。
- テスト戦略、ブランチ戦略、ローカル開発手順、Runbookが整備されたら、必要な参照やチェック項目を追加する。
- 完了条件が重くなりすぎて実態に合わない場合は、品質を落とすのではなく、作業単位を小さくする方向で調整する。

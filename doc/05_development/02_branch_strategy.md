# ブランチ戦略

## 目的

このドキュメントは、自炊本閲覧Webアプリケーションにおけるブランチ運用、作業単位、コミット粒度、プルリクエスト運用の基本方針を定義する。

本プロジェクトでは、安定したリリース履歴を残しながら小さく継続的に開発するため、基本原則としてGit-flowを採用する。ただし、開発規模が小さい段階では運用を重くしすぎず、ブランチの役割とレビュー可能な変更単位を守ることを優先する。

## 前提

- コミット、ブランチ、プルリクエスト、検証結果の共通ルールは [rules/CONTRIBUTING.md](../../rules/CONTRIBUTING.md) に従う。
- 完了条件は [doc/05_development/05_definition_of_done.md](05_definition_of_done.md) に従う。
- 実装ルールは [rules/CODING_STANDARDS.md](../../rules/CODING_STANDARDS.md) と [doc/05_development/01_coding_rules.md](01_coding_rules.md) に従う。
- 作業開始前に [doc/TODO.md](../TODO.md) を確認し、関連するTODO、設計文書、受入条件を把握する。
- 仕様やアーキテクチャ上の重要な判断は、TODOやプルリクエスト説明だけでなくADRまたは設計ドキュメントへ記録する。

## 基本方針

- `main` はリリース済み、またはいつでもリリース可能な安定状態として扱う。
- `develop` は次回リリースへ向けた統合ブランチとして扱う。
- 日常的な開発は、`develop` から作成した短命ブランチで行う。
- リリース準備は `release/*` ブランチで行い、リリース後に `main` と `develop` へ反映する。
- 本番障害や緊急修正は `hotfix/*` ブランチで行い、修正後に `main` と `develop` へ反映する。
- PostgreSQL、Elasticsearch、ファイル保存、変換ワーカー、権限、セキュリティに影響する変更は、影響範囲と検証結果をプルリクエストで明確にする。
- 無関係な変更を同じブランチやプルリクエストに混ぜない。

## ブランチ種別

### main

`main` は安定版を表すブランチである。

- 直接コミットしない。
- マージ元は原則として `release/*` または `hotfix/*` とする。
- `main` へ反映したリリースにはタグを付与する。
- `main` 上の内容は、リリースノートや運用手順と矛盾しない状態にする。

### develop

`develop` は次回リリース候補を統合するブランチである。

- 直接コミットしない。
- 通常の機能追加、修正、ドキュメント更新は、短命ブランチからプルリクエストで取り込む。
- `develop` へ取り込む前に、変更範囲に応じたテストまたは確認を行う。
- `release/*` や `hotfix/*` で行った修正は、取りこぼしがないよう `develop` へ反映する。

### feature

機能追加やユーザーストーリー対応は `feature/*` ブランチで行う。

作成元:

```text
develop
```

マージ先:

```text
develop
```

例:

```text
feature/book-upload
feature/book-search
feature/conversion-job-status
```

### fix

通常の不具合修正は `fix/*` ブランチで行う。リリース済みバージョンへの緊急修正ではない場合は、`hotfix/*` ではなく `fix/*` を使う。

作成元:

```text
develop
```

マージ先:

```text
develop
```

例:

```text
fix/job-status-update
fix/search-pagination
```

### docs

ドキュメントのみの変更は `docs/*` ブランチで行う。

作成元:

```text
develop
```

マージ先:

```text
develop
```

例:

```text
docs/branch-strategy
docs/api-design-policy
```

### release

リリース準備は `release/*` ブランチで行う。

作成元:

```text
develop
```

マージ先:

```text
main
develop
```

例:

```text
release/0.1.0
release/1.0.0
```

`release/*` では、原則として次の作業に限定する。

- バージョン番号の更新
- リリースノートの作成または更新
- 軽微な不具合修正
- ドキュメントの整合性修正
- リリース前検証で見つかった設定や手順の修正

大きな仕様変更、新機能追加、設計変更は `release/*` に入れず、次の開発サイクルの `feature/*` として扱う。

### hotfix

リリース済みの `main` に対する緊急修正は `hotfix/*` ブランチで行う。

作成元:

```text
main
```

マージ先:

```text
main
develop
```

例:

```text
hotfix/archive-path-validation
hotfix/login-failure
```

`hotfix/*` は、利用者影響、データ破損、セキュリティ、運用停止など、次回通常リリースを待てない問題に限定する。修正後は `main` に反映してタグを付与し、同じ内容を `develop` へも反映する。

## ブランチ命名規則

ブランチ名は、[rules/CONTRIBUTING.md](../../rules/CONTRIBUTING.md) の形式に従う。

```text
<type>/<short-description>
```

使用する主なtypeは次のとおり。

| type | 用途 |
|---|---|
| `feature` | Git-flow上の機能追加ブランチ |
| `fix` | 通常の不具合修正 |
| `docs` | ドキュメントのみの変更 |
| `refactor` | 振る舞いを変えない整理 |
| `test` | テスト追加または更新 |
| `chore` | 開発環境、設定、保守作業 |
| `style` | フォーマットやスタイルのみの変更 |
| `perf` | パフォーマンス改善 |
| `security` | セキュリティ修正または強化 |
| `build` | ビルド設定や依存関係 |
| `ci` | CI/CD設定 |
| `release` | リリース準備 |
| `hotfix` | リリース済みバージョンへの緊急修正 |
| `revert` | 変更の取り消し |

`feature/*` はGit-flowの名称として使用し、コミット種別やプルリクエストの性質は `feat` として扱ってよい。

命名時は次を守る。

- 英小文字、数字、ハイフンを基本にする。
- 短く、変更内容が分かる名前にする。
- チケット番号やTODO番号がある場合は、必要に応じて末尾または先頭へ付ける。
- `work`、`temp`、`misc`、`update` のように内容が分からない名前を避ける。
- 個人名やローカル環境名を含めない。

## 作業単位

1つのブランチは、1つの目的に対応させる。

適切な作業単位の例:

- 1つのユーザーストーリーを実装する。
- 1つの不具合を修正する。
- 1つの設計文書を作成または更新する。
- 1つのリファクタリング目的に絞って変更する。
- 1つのテスト観点を追加する。

分割を検討する例:

- フロントエンド、バックエンド、ワーカー、DB、検索インデックスを同時に大きく変更する。
- 機能追加と無関係なリファクタリングが混在している。
- ドキュメント整備と実装変更の関係が薄い。
- レビューで変更意図を追うのが難しい。

仕様、設計、運用手順に影響する変更では、関連ドキュメントと [doc/TODO.md](../TODO.md) の更新を同じ作業単位に含める。

## コミット粒度

コミットは小さく、論理的にまとまった単位にする。形式は [rules/CONTRIBUTING.md](../../rules/CONTRIBUTING.md) に従い、日本語で次の形式を使う。

```text
<type>: <Japanese summary>
```

例:

```text
docs: ブランチ戦略を追加
feat: 書籍アップロードAPIを追加
fix: 変換ジョブ失敗時の状態更新を修正
security: アーカイブ展開時のパス検証を強化
```

コミット時は次を守る。

- 無関係な変更を同じコミットに含めない。
- 生成された大容量ファイル、原本ファイル、変換済み画像、秘密情報、ローカル環境ファイルを含めない。
- 振る舞いの変更とテスト追加は、レビューしやすい範囲で同じコミットまたは隣接するコミットにまとめる。
- ドキュメントやTODOの更新がDefinition of Doneに関わる場合は、実装変更と同じブランチ内に含める。
- 互換性を壊す変更は、コミット本文またはプルリクエスト説明に `BREAKING CHANGE` として明記する。

## プルリクエスト運用

プルリクエストは、レビュー担当者が目的、影響範囲、検証結果を追える単位で作成する。

プルリクエストには次を記載する。

- 変更の目的
- 主な変更点
- 影響範囲
- 実行したテストまたは確認
- 更新したドキュメント
- 関連するTODO、ユーザーストーリー、受入条件、ADR
- UI変更がある場合のスクリーンショットまたは確認内容
- 移行作業、設定変更、運用上の注意点

レビュー前に次を確認する。

- 作業ブランチが最新の `develop` または適切な作成元に追従している。
- 変更範囲がプルリクエストの目的と一致している。
- Definition of Doneの該当項目を満たしている。
- 関連ドキュメントと [doc/TODO.md](../TODO.md) が更新されている。
- テストや手動確認の結果が説明できる。
- 秘密情報や不要な個人情報が含まれていない。

## マージ方針

- `feature/*`、`fix/*`、`docs/*` などの日常作業ブランチは、レビュー後に `develop` へマージする。
- `release/*` はリリース検証後に `main` へマージし、タグを付与する。その後、同じ内容を `develop` へ反映する。
- `hotfix/*` は修正検証後に `main` へマージし、タグを付与する。その後、同じ内容を `develop` へ反映する。
- マージ前に競合を解消し、競合解消で意図しない差分が混ざっていないことを確認する。
- マージ方式は、履歴の読みやすさとレビュー容易性を優先して選ぶ。プロジェクトでCIやリポジトリ設定が整備された時点で、squash merge、merge commit、rebase mergeの使い分けを具体化する。

## リリースとタグ

リリースタグは、原則としてセマンティックバージョニングに沿って付与する。

```text
v<major>.<minor>.<patch>
```

例:

```text
v0.1.0
v1.0.0
v1.0.1
```

リリース時は次を確認する。

- `release/*` または `hotfix/*` の変更内容が `main` に反映されている。
- リリースタグが `main` 上の対象コミットを指している。
- リリースノートが作成または更新されている。
- 運用手順、環境変数、DB変更、検索インデックス再構築など、必要な移行作業が文書化されている。
- `develop` への反映漏れがない。

## 緊急修正

緊急修正では、速度を優先しつつも、原因、影響範囲、検証結果を追跡できる状態を保つ。

緊急修正の流れ:

1. `main` から `hotfix/*` を作成する。
2. 修正と必要最小限のテストを行う。
3. 影響範囲、検証結果、残リスクをプルリクエストに記載する。
4. `main` へマージしてタグを付与する。
5. 同じ修正を `develop` へ反映する。
6. 必要に応じてRunbook、障害ログ、リリースノートを更新する。

セキュリティ修正では、詳細な攻撃手順、秘密情報、個人情報をプルリクエストやログに記載しない。

## ドキュメントのみの変更

ドキュメントのみの変更でも、次を確認する。

- Markdownの見出し構造が読みやすい。
- リンク先やファイルパスが実在する、または今後作成予定であることがTODOから分かる。
- [doc/TODO.md](../TODO.md) のステータスが実態と一致している。
- 仕様判断が文書内に閉じすぎていない。重要な判断はADRまたは設計文書に記録する。
- 周辺文書の参照と矛盾していない。

## 更新方針

このドキュメントは、CI/CD、リリース手順、バージョニング、レビュー体制が具体化した時点で更新する。運用中にブランチが長寿命化しすぎる、プルリクエストが大きくなりすぎる、リリース反映漏れが起きるなどの問題が見つかった場合も、この文書を見直す。

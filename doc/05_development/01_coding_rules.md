# コーディングルール

## 目的

このドキュメントは、自炊本閲覧Webアプリケーションにおける実装時の基本ルールを定義する。

[rules/CODING_STANDARDS.md](../../rules/CODING_STANDARDS.md) はリポジトリ全体に適用する上位のコーディング標準であり、このドキュメントはその内容を本プロジェクトの構成に合わせて具体化する。判断に迷う場合は、可読性、正しさ、保守性、セキュリティを優先し、重要な設計判断はADRまたは設計ドキュメントへ記録する。

## 適用範囲

このルールは、次のすべてに適用する。

- Next.jsフロントエンド
- Spring BootバックエンドAPI
- Spring Boot変換ワーカー
- テストコード
- 開発、運用、変換処理を支援するスクリプト

言語やフレームワーク固有の詳細ルールは、今後必要に応じてこの文書または個別文書へ追加する。

## 基本方針

- 読みやすく、意図が追いやすいコードを優先する。
- クリーンコードを意識し、名前、関数、クラス、モジュールがそれぞれ一つの明確な意図を持つようにする。
- 仕様にない拡張点や将来機能を先回りして作らない。
- 小さく凝集した変更を心がける。
- 暗黙の副作用を避け、状態変更、I/O、外部プロセス実行、DB更新がある処理は呼び出し側から分かる名前と配置にする。
- ビジネスルールをコントローラ、画面コンポーネント、インフラ実装へ埋め込まない。
- クリーンアーキテクチャを意識し、ドメインやユースケースがフレームワーク、DB、外部サービス、UIの詳細に依存しない構成を優先する。
- PostgreSQLを正本、Elasticsearchを再構築可能な派生データとして扱う。
- API、変換ワーカー、ミドルウェアを将来分離できるよう、責務境界を不必要に結合しない。

## 命名

- ドメイン上の意味が分かる名前を使う。
- 書籍、原本ファイル、変換済み画像、サムネイル、著者、シリーズ、タグ、変換ジョブ、閲覧履歴などの用語は、用語集と設計ドキュメントに合わせる。
- `data`、`info`、`manager`、`processor`、`handler` のような曖昧な名前は、役割が本当に汎用的な場合だけ使う。
- 真偽値は `is...`、`has...`、`can...`、`should...` など、条件として読める名前にする。
- メソッド名は、取得、作成、更新、削除、検証、変換、ジョブ投入などの動作が分かる名前にする。
- 副作用がないように見える名前のメソッドで、DB更新、ファイル操作、外部コマンド実行、ジョブ投入を行わない。
- 不要な省略語を避け、フロントエンド、バックエンド、DB、API、ドキュメントで同じ概念に同じ名前を使う。

## 責務分離

本プロジェクトでは、クリーンアーキテクチャの考え方を取り入れ、依存関係が外側の詳細から内側の方針へ向かうように設計する。

- ドメイン層は、フレームワーク、HTTP、DB、Elasticsearch、ファイルシステム、外部コマンドに依存しない。
- アプリケーション層は、ユースケースの流れを表現し、具体的な永続化や外部連携はポートまたはインターフェース越しに扱う。
- インフラ層は、DB、検索、ファイル、7-Zip、専用キューなどの詳細を実装する。
- UI、HTTP、DB、検索インデックスなどの外部形式を、ドメインモデルへ直接持ち込まない。
- 依存方向を保つための変換処理は、境界付近に閉じ込める。

### バックエンドAPI

- コントローラはHTTPリクエスト、レスポンス、ステータスコード、入力形式の扱いに集中する。
- アプリケーションサービスは、ユースケースの調整、認可確認、トランザクション境界、ドメイン処理の呼び出しを担う。
- ドメインモデルとドメインサービスは、業務ルール、状態遷移、不変条件を表現する。
- リポジトリは永続化の詳細を隠蔽する。
- PostgreSQL、Elasticsearch、ファイルシステム、外部プロセス、専用キューとの接続はインフラ層へ閉じ込める。
- バックエンドAPIは、フロントエンドで入力チェック済みの値であっても信用せず、必ずサーバ側でバリデーションする。
- バックエンドAPIは、画面上で選択肢や入力範囲を制限している項目であっても、改ざんされたリクエストが送られる前提で検証する。

### 変換ワーカー

- 変換ワーカーは、アーカイブ展開、画像変換、サムネイル生成、変換ジョブ状態更新を担う。
- HTTPリクエストのライフサイクルに依存した処理を書かない。
- 7-Zip for Linux コンソール版の呼び出しは、明示的なタイムアウト、終了コード確認、標準出力と標準エラーの扱いを定義して実装する。
- ジョブごとの専用作業ディレクトリを前提にし、作業ディレクトリ外へ書き込ませない。
- 失敗は握りつぶさず、再試行可能性と失敗状態を追跡できる形で扱う。

### フロントエンド

- ページ、画面状態、再利用可能なUIコンポーネント、API呼び出し処理を分ける。
- 画面上の権限制御は利用者体験のための補助とし、セキュリティ判断はバックエンドAPIで必ず行う。
- APIレスポンス、読み込み中、空状態、入力エラー、権限不足、通信失敗を明示的に扱う。
- 書籍一覧、検索結果、ページ一覧のような大きくなり得るデータはページングまたは段階的読み込みを前提にする。

## モデル分離

APIモデル、永続化エンティティ、ドメインモデルは概念的に分離する。

- APIリクエストモデルは、外部入力の形とバリデーションを表す。
- APIレスポンスモデルは、公開してよい情報だけを表す。
- 永続化エンティティは、DBテーブルや永続化の都合を表す。
- ドメインモデルは、業務上の状態、不変条件、振る舞いを表す。
- Elasticsearchの検索ドキュメントは、検索最適化用の派生表現として扱う。

永続化エンティティをそのままAPIへ返さない。内部ID、ファイルパス、変換ワーカー内部の作業パス、検索インデックスの内部構造などは、API契約で明示しない限り公開しない。

固定的な状態は任意文字列ではなく、enum、値オブジェクト、定数などで表現する。対象例は、変換ジョブ状態、処理フェーズ、ロール、権限、公開範囲、画像変換設定である。

モデル変換は、層の境界で明示的に行う。APIリクエストをそのままドメインモデルとして扱わず、バリデーションと変換を経てユースケースへ渡す。永続化エンティティをドメインの代用品として広く使い回さない。

## クリーンコード

- 関数やメソッドは一つの目的に絞り、複数の抽象度の処理を混在させない。
- 条件分岐が複雑になる場合は、仕様上の状態やルールを名前付きのメソッド、値オブジェクト、ドメインサービスへ切り出す。
- コメントで補わなければ理解できない名前や構造は、まず名前と構造を見直す。
- コメントは「なぜそうしているか」を説明するために使い、「何をしているか」の説明に頼らない。
- マジックナンバーやマジックストリングを避け、設定値、定数、enum、値オブジェクトで意味を表す。
- null、空文字、空配列、未設定値の扱いを曖昧にしない。
- 共通化は、同じ知識や同じ業務ルールが重複したときに行う。形が似ているだけのコードを無理に抽象化しない。
- テストしにくいコードは責務や依存方向が崩れている可能性があるため、設計を見直す。

## バリデーション

バックエンドは、フロントエンド、ブラウザ、APIクライアント、管理画面、バッチ、外部ツールから送られる入力値を信用しない。フロントエンドの入力制限は利用者体験のための補助であり、正しさと安全性はバックエンドで保証する。

- リクエストボディ、クエリパラメータ、パスパラメータ、ヘッダー、Cookie、アップロードファイル、ジョブパラメータをサーバ側で検証する。
- 必須、文字数、数値範囲、形式、列挙値、日付、ページング、ソート条件を検証する。
- IDやロールなど、画面上で変更できない想定の値も改ざんされる前提で扱う。
- 書籍アップロード、管理操作、変換ジョブ再実行などは、入力検証に加えて認可確認を必ず行う。
- ファイル名、アーカイブ内パス、展開後パスは正規化し、許可された作業ディレクトリ配下に収まることを確認する。
- バリデーションエラーは、一貫したエラーレスポンスで返す。
- ドメイン不変条件は、API入力チェックだけでなく、ドメイン層またはユースケース層でも破れないようにする。

## 例外処理

- 例外を空のcatchで握りつぶさない。
- バリデーションエラー、業務エラー、認証エラー、認可エラー、外部依存の失敗、予期しないシステムエラーを区別する。
- ユーザ向けメッセージは安全で理解できる内容にする。
- 内部の詳細、スタックトレース、ホスト上のパス、コンテナ内パス、外部コマンドの生ログをレスポンスへ出さない。
- 広すぎる例外捕捉は、コントローラ境界、ジョブ実行境界、外部依存境界など、失敗を変換する責務がある場所に限定する。
- リトライは安全に再実行できる処理だけに限定し、回数、間隔、失敗時の状態を明示する。
- アーカイブ展開、画像変換、Elasticsearch更新の失敗は、必要に応じて変換ジョブまたはインデックス更新ジョブの失敗状態として表現する。

## ログ

- ログは、障害調査、変換ジョブ追跡、検索インデックス更新、運用確認に役立つ内容にする。
- パスワード、トークン、セッション識別子、秘密鍵、不要な個人情報、ファイル内容を出力しない。
- 同じ例外を複数レイヤーで重複してログ出力しない。
- `DEBUG` は詳細調査、`INFO` は主要なライフサイクルやジョブ進行、`WARN` は復旧可能だが注意が必要な問題、`ERROR` は調査が必要な失敗に使う。
- 変換ジョブ、アーカイブ展開、画像変換、サムネイル生成、検索インデックス更新は、ジョブIDや書籍IDなど安全な識別子で追跡できるようにする。
- 利用者へ見せるエラーと運用者向けログを混同しない。

## テスト

- 振る舞いを変える実装では、TDDを基本とする。
- 新しい業務ルール、状態遷移、入力検証、権限確認を実装する前に、期待する振る舞いを失敗する自動テストとして表現する。
- 小さな単位で、失敗するテストを書く、最小実装で通す、構造を整える、という流れを繰り返す。
- UIや外部依存が強く、最初から自動化しにくい振る舞いは、受入テスト観点または手動確認項目を先に明確にし、設計が固まった段階で自動テストを追加する。
- 変更範囲とリスクに応じてテストを追加または更新する。
- 命名は、何を期待しているテストか分かる説明的な名前にする。
- 正常系だけでなく、境界値、異常系、権限不足、未認証、存在しないリソース、重複、競合、外部依存失敗を必要に応じて確認する。
- ビジネスルール、権限確認、変換ジョブ状態遷移、入力バリデーション、検索インデックス更新は優先してテスト対象にする。
- ファイルシステム、7-Zip外部プロセス、Elasticsearch、専用キュー、長時間の画像変換は、必要に応じてモック、統合テスト、手動確認を使い分ける。
- 不具合修正では、再発を防ぐテストを追加する。
- カバレッジ数値だけを目的にせず、重要な振る舞いを意味のあるアサーションで確認する。

## セキュリティ

詳細なセキュリティ方針は [rules/SECURITY.md](../../rules/SECURITY.md) に従う。実装時は少なくとも次を確認する。

- HTTPリクエスト、クエリパラメータ、パスパラメータ、アップロードファイル名、アーカイブ内エントリ、メタデータ、ジョブパラメータを検証する。
- フロントエンドで検証済み、非表示、読み取り専用、選択式として扱っている値でも、バックエンドで再検証する。
- 管理ユーザのみ可能な操作はバックエンド側で認可確認する。
- 一般ユーザが書籍アップロード、書籍削除、管理機能、変換ジョブ再実行などの管理操作を実行できないようにする。
- フロントエンドの表示制御だけを権限制御として扱わない。
- パストラバーサル、絶対パス、ドライブレター、シンボリックリンクによる作業ディレクトリ外への脱出を防ぐ。
- アーカイブ展開は管理された作業ディレクトリ内で行い、任意ファイルの上書きを許可しない。
- ファイル拡張子だけを信用せず、必要に応じてMIME typeや実体の検証を行う。
- WebP品質値、ワーカー同時実行数、外部プロセスタイムアウト、保存パス、Elasticsearchインデックス名、DB接続情報などをハードコードしない。
- 秘密情報、ローカル環境ファイル、生成された大容量ファイル、原本ファイル、変換済み画像をGitへ含めない。
- Elasticsearchには検索に必要な派生情報だけを保持し、秘密情報や不要な個人情報を入れない。

## 依存関係

- 新しいライブラリは、必要性、保守状況、セキュリティ、既存機能との重複を確認してから追加する。
- アーカイブ展開、画像処理、認証、認可、ファイルアクセス、検索に関わる依存関係は特に慎重に扱う。
- rar / 7zipの展開は、変換ワーカーから7-Zip for Linux コンソール版を外部プロセスとして呼び出す方針に従う。
- 置き換えの可能性がある依存は、呼び出し箇所をインフラ層やアダプタへ閉じ込める。

## レビュー時の確認観点

レビューでは、次を確認する。

- 名前が目的とドメインの意味を表している。
- クリーンコードの観点で、関数、クラス、モジュールの責務が小さく明確である。
- クリーンアーキテクチャの観点で、ドメインやユースケースが外側の技術詳細へ依存していない。
- 責務境界が守られている。
- ビジネスルールが適切な層に置かれている。
- APIモデル、永続化エンティティ、ドメインモデルが混ざっていない。
- フロントエンドの入力チェックに依存せず、バックエンドで入力値を検証している。
- 入力検証、認可、エラー処理、ログが安全に実装されている。
- PostgreSQLを正本、Elasticsearchを派生データとして扱っている。
- 変換ワーカーの外部プロセス実行、タイムアウト、失敗状態が追跡できる。
- テストまたは手動確認が変更範囲に対して十分である。
- 関連ドキュメントと [doc/TODO.md](../TODO.md) が更新されている。

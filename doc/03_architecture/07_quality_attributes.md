# 品質特性初版

## 目的

このドキュメントは、自炊本閲覧Webアプリケーションの初期品質特性を整理する。

対象は、性能、容量、スケーラビリティ、信頼性、整合性、セキュリティ、運用性、保守性である。ここに記載する数値は、実装前の初期目標または初期想定であり、実測値、利用状況、運用上の制約が分かった段階で見直す。

## 前提

- 初期運用は単一Linuxホスト上のDocker Compose構成とする。
- アプリケーション規模は小から中規模を想定する。
- 一般ユーザは書籍を検索、閲覧、お気に入り管理する。
- 管理ユーザは書籍アップロード、メタ情報管理、変換ジョブ確認、ユーザ管理、ロール管理を行う。
- 書籍アップロードは管理ユーザのみが実行できる。
- PostgreSQLを正本、Elasticsearchを再構築可能な派生データとして扱う。
- 原本ファイル、変換済みWebP、サムネイルは書籍ファイル保存領域に保存する。
- アーカイブ展開、WebP変換、サムネイル生成は非同期変換ジョブとして処理する。
- 変換ワーカーの同時実行数、安全上限、1ジョブタイムアウトは [画像変換設計のリソース制限と設定](../04_design/07_image_conversion_design.md#リソース制限と設定) を正本とする。
- 業務上の保存容量課金や1冊あたりの登録容量には初期上限を設けない。
- アップロード、展開、画像デコード、一時ディスク使用には技術的な安全上限を設ける。
- 1ユーザあたりの保存容量上限は設けない。
- WebP品質値の既定値は80とする。
- 初期運用ではバックアップを行わない。RPO / RTO、復旧不能リスク、誤操作対策は [バックアップなし方針](../07_operations/02_backup_restore.md) を正本とし、具体的な運用手順は [Runbook](../07_operations/01_runbook.md) に従う。

## 品質特性サマリ

| 品質特性 | 初期目標 / 方針 |
| --- | --- |
| 応答性能 | 閲覧、検索、管理操作の通常APIは、重い変換処理を同期実行しない。 |
| 変換性能 | 変換ワーカー同時実行数とタイムアウトは画像変換設計の安全上限を正本とし、実測により調整する。 |
| 容量拡張性 | 業務上の保存容量上限は置かないが、アップロード、展開、一時領域には安全上限を置き、ストレージ監視と運用判断で扱う。 |
| 検索性能 | Elasticsearchで日本語検索と表記揺れ正規化を行う。必須プラグインは技術スタックを正本とする。 |
| 可用性 | 単一ホスト構成のため高可用構成は初期対象外とし、障害時は復旧手順を重視する。 |
| 整合性 | PostgreSQLを正本とし、Elasticsearchと生成ファイルは再構築、再生成可能にする。 |
| セキュリティ | 認証、認可、入力検証、パストラバーサル対策、シークレット非露出を必須とする。 |
| 運用性 | ジョブ状態、失敗理由、再実行、再インデックス、ログ確認を運用できる形にする。 |
| 保守性 | モジュラーモノリス方針に従い、API、Worker、ミドルウェアの将来分離余地を保つ。 |

## レスポンス時間の目標

レスポンス時間の初期目標は、サーバ側処理時間を中心に定義する。ネットワーク回線、ブラウザ描画、巨大ファイルのアップロード転送時間は環境差が大きいため、別途計測対象として扱う。

| 操作 | 初期目標 | 備考 |
| --- | --- | --- |
| ログイン、ログアウト、会員情報取得 | p95で500ms以内 | メール送信や2段階認証の外部待ち時間は別計測にする。 |
| 本一覧、書籍詳細、ページ一覧 | p95で700ms以内 | ページングまたは分割取得を前提とする。 |
| 検索API | p95で1秒以内 | Elasticsearchの検索時間とPostgreSQL確認時間を含める。 |
| お気に入り登録、解除、閲覧履歴保存 | p95で500ms以内 | PostgreSQL更新を主処理とする。 |
| 管理ユーザのメタ情報登録、更新 | p95で700ms以内 | Elasticsearch更新は失敗時に再試行できるようにする。 |
| アップロード受付 | ファイル転送完了後、p95で3秒以内に受付結果を返す | 原本保存、正本登録、ジョブ投入までを対象とし、変換完了は待たない。 |
| ページ画像配信 | p95で1秒以内に配信開始 | 変換済みWebPを対象とし、原本ファイル配信は行わない。 |

これらの目標は初期値であり、実装後にAPM、ログ、手動計測、負荷試験の結果を見て調整する。

## アップロード後の変換待ち時間

アップロード要求では、原本ファイル保存、PostgreSQL登録、変換ジョブ投入までを行い、HTTPレスポンスでは変換完了を待たない。

変換待ち時間は、次の要素に依存する。

- 原本ファイルのサイズ
- アーカイブ形式
- ページ数
- 入力画像の解像度と形式
- ストレージI/O
- 変換ワーカーの同時実行数
- キュー滞留数
- 7-Zip外部プロセスの処理時間

初期方針は次のとおり。

| 項目 | 方針 |
| --- | --- |
| ユーザへの表示 | 変換待ち、変換中、完了、失敗を明示する。 |
| 変換完了待ち | アップロード画面では完了を待たず、ジョブ状態確認で追跡する。 |
| タイムアウト | 1ジョブ30分を既定値とする。 |
| 再実行 | 失敗ジョブは原因を確認し、管理操作または運用手順で再実行できるようにする。 |
| 優先度制御 | 初期対象外とし、キュー滞留が問題になった段階で検討する。 |

30分を超えるジョブが頻発する場合は、ページ数、解像度、ファイルサイズ、変換条件、ワーカー同時実行数、ホストリソースを見直す。

## 同時利用ユーザ数の想定

初期運用では、個人または小規模チーム利用を主な想定とする。

| 利用形態 | 初期想定 |
| --- | --- |
| 登録ユーザ数 | 数十人規模 |
| 同時に画面操作する一般ユーザ | 10人程度 |
| 同時にビューアでページ閲覧する一般ユーザ | 5人程度 |
| 同時に管理操作する管理ユーザ | 1から2人程度 |
| 同時アップロード | 画像変換設計の安全上限を正本とする |
| 変換ジョブ同時実行 | 画像変換設計の安全上限を正本とする |

同時利用が増えた場合は、次の順で確認する。

1. PostgreSQL接続数とクエリ時間
2. Elasticsearch検索時間
3. ページ画像配信のI/O
4. 変換ワーカーのCPU、メモリ、一時ディスク使用量
5. RabbitMQのキュー滞留数
6. Docker Compose単一ホストのCPU、メモリ、ストレージI/O

## 1冊あたりのページ数とファイルサイズ

1冊あたりの業務上の保存容量上限は設けない。ただし、アップロード受付、アーカイブ展開、画像デコード、ジョブ一時領域には技術的な安全上限を設ける。これは容量課金や保有方針の制限ではなく、7-Zip展開、巨大アーカイブ、zip bomb、画像デコード爆発、同時実行によるストレージ枯渇を防ぐための制限である。

初期想定は次のとおり。具体的な安全上限値は [画像変換設計のリソース制限と設定](../04_design/07_image_conversion_design.md#リソース制限と設定) を正本とする。

| 項目 | 初期想定 / 方針 |
| --- | --- |
| 1冊あたりのページ数 | 数十から数百ページを主対象とする。 |
| アップロード1ファイル | 技術的な安全上限を設け、業務上の保存容量上限とは分けて扱う。 |
| 展開後総量 | 技術的な安全上限を設ける。 |
| アーカイブエントリ数 | 技術的な安全上限を設ける。 |
| 1画像の最大ピクセル数 | 技術的な安全上限を設ける。 |
| 原本ファイル | 保存し続ける。 |
| 変換済みWebP | 閲覧用に保存する。 |
| サムネイル | 一覧、検索結果、詳細画面用に保存する。 |
| 1ジョブ一時ディスク使用量 | 技術的な安全上限を設ける。 |

これらの上限値はapplication.propertiesで変更可能にする。上限を超えた場合は、システム障害として扱わず、対象ファイルまたはジョブを安全な業務エラーとして拒否または`failed`へ遷移させる。原本保存前に検出できるアップロードサイズ超過はHTTPエラーで返し、展開後総量、エントリ数、画像ピクセル数、一時領域超過は変換ジョブの失敗理由として記録する。

極端に大きいファイルやページ数の多い本では、変換時間、ストレージ使用量、一時ディスク使用量、メモリ使用量が増える。運用上問題になる場合は、安全上限、ページ数上限、変換時の最大幅や最大高さ、ジョブ優先度、ワーカー分離を検討する。

## 変換ワーカーのスループット目標

変換ワーカーは、APIとは別プロセス、別コンテナとして扱い、CPU、メモリ、ディスクI/O、外部プロセス実行の負荷をHTTP処理から分離する。

| 項目 | 初期値 / 目標 |
| --- | --- |
| 同時実行数 | 画像変換設計の安全上限を正本とする |
| 1ジョブのタイムアウト | 画像変換設計の安全上限を正本とする |
| WebP品質値 | 80 |
| サムネイル生成 | 行う |
| 1ジョブの一時ディスク使用量 | 画像変換設計の安全上限を正本とする |
| メモリ / CPU制限 | OSまたはコンテナ側で制御する |
| 設定方法 | application.propertiesで変更可能にする |

スループットはページ数、画像解像度、アーカイブ形式、ホスト性能に強く依存するため、初期段階では安全上限の範囲内で安定性を優先して調整する。1時間あたりの冊数は実測後に定義する。

監視、負荷試験、運用で確認する主な指標は次のとおり。

- キュー滞留数
- ジョブ待ち時間
- ジョブ処理時間
- 30分タイムアウトの発生数
- 変換失敗数
- 7-Zip外部プロセス失敗数
- CPU使用率
- メモリ使用量
- 一時ディスク使用量
- 書籍ファイル保存領域のI/O

## ストレージ容量増加への対応方針

1ユーザあたりの保存容量上限は設けない。一般ユーザは書籍を保持せず、書籍アップロードは管理ユーザのみが実行する。

ストレージ容量は次の要素で増加する。

- 原本ファイル
- 展開後の一時ファイル
- 変換済みWebP
- サムネイル
- PostgreSQLデータ
- Elasticsearchインデックス
- ログ

初期方針は次のとおり。

| 項目 | 方針 |
| --- | --- |
| 原本ファイル | 保存し続ける。 |
| 変換済みWebP | 閲覧用に保存する。 |
| サムネイル | 表示性能向上のため保存する。 |
| 一時ファイル | ジョブ完了または失敗後にクリーンアップする。 |
| Elasticsearch | PostgreSQLから再構築可能な派生データとして扱う。 |
| バックアップ | 行わない。 |
| 監視 | ストレージ使用量、増加傾向、残容量を監視対象にする。 |

容量不足が近づいた場合は、書籍削除、未参照ファイル削除、失敗ジョブの一時ファイル削除、ストレージ拡張、変換条件の見直しを検討する。

## 検索レスポンスの目標

検索はElasticsearchを利用し、タイトル、著者、タグ、シリーズなどの日本語検索と表記揺れ正規化を行う。Elasticsearch必須プラグインは技術スタックを正本とする。

| 項目 | 初期目標 / 方針 |
| --- | --- |
| 検索API応答 | p95で1秒以内 |
| 検索結果ページング | 必須 |
| 検索対象 | タイトル、著者、タグ、シリーズを初期候補とする。 |
| 正本確認 | 権限や表示可否が重要な場合はPostgreSQLで確認する。 |
| インデックス更新 | PostgreSQL更新後にElasticsearchを更新し、失敗時は再試行する。 |
| 再構築 | 書籍単位と全件の再インデックスを可能にする。 |

検索レスポンスが悪化した場合は、クエリ、mapping、analyzer、edge n-gram系フィールド、ソート条件、ページング方式、PostgreSQLでの追加確認件数を見直す。

## 可用性と復旧

初期構成は単一Linuxホスト上のDocker Composeであり、冗長化された高可用構成は対象外とする。

初期運用ではバックアップを行わないため、バックアップ復元を前提にした復旧目標は持たない。RPO / RTOと復旧不能リスクの詳細は [バックアップなし方針](../07_operations/02_backup_restore.md) を正本とし、残存データから回復できる範囲のみRunbookに従って対応する。

| 事象 | 初期方針 |
| --- | --- |
| API停止 | Docker Composeで再起動し、ログを確認する。 |
| 変換ワーカー停止 | 未完了ジョブをPostgreSQLとキューから確認し、再開または再実行する。 |
| PostgreSQL障害 | 正本障害として扱う。バックアップなし方針のリスクを受け入れる。 |
| Elasticsearch障害 | 検索機能に影響するが、PostgreSQLから再構築可能な派生データとして扱う。 |
| 書籍ファイル保存領域障害 | 原本、WebP、サムネイルに影響する。バックアップなし方針のリスクを受け入れる。 |
| キュー障害 | ジョブ配送に影響する。PostgreSQL上のジョブ状態を基準に再投入を検討する。 |

破壊的操作と削除系管理コマンドの安全柵は、バックアップなし方針とRunbookを正本とする。誤操作による正本喪失は、初期運用における主要リスクとして扱う。

可用性を高める必要が出た場合は、API / Worker / PostgreSQL / Elasticsearch / キュー / ファイル保存領域の分離、バックアップ、監視、復旧手順を再検討する。

## 整合性と再構築性

整合性の基本方針は次のとおり。

- PostgreSQLを正本とする。
- Elasticsearchは派生データとして扱い、PostgreSQLから再インデックスできるようにする。
- 変換済みWebPとサムネイルは、原本ファイルと変換条件から再生成できるようにする。
- 変換ジョブ状態、失敗理由、検索インデックス更新状態をPostgreSQLへ記録する。
- キューの配送状態だけを業務上の正本にしない。

整合性回復の詳細な流れは [doc/03_architecture/06_data_flow.md](06_data_flow.md) で扱う。

## セキュリティ

セキュリティ上の品質目標は次のとおり。

- フロントエンドの検証結果を信頼せず、バックエンドAPIで認証、認可、入力検証を行う。
- 書籍アップロード、削除、変換ジョブ再実行、ユーザ管理、ロール管理はサーバ側で権限確認する。
- アップロードファイル、アーカイブ内パス、展開先パス、生成ファイル名を検証し、パストラバーサルを防ぐ。
- 7-Zip外部プロセスは、ジョブごとの作業ディレクトリ、明示的なタイムアウト、終了コード確認、エラー記録を行う。
- パスワード、トークン、シークレット、不要な個人情報をログやレスポンスへ出力しない。
- 内部の物理ファイルパスをAPIレスポンスへ不用意に露出しない。

詳細は [rules/SECURITY.md](../../rules/SECURITY.md) と、今後作成する権限設計、ファイル保存設計、画像変換設計で扱う。

## 運用性と可観測性

運用で確認できるようにする主な情報は次のとおり。

| 対象 | 観測したい情報 |
| --- | --- |
| API | エラー率、レスポンス時間、DB接続状態、Elasticsearch接続状態 |
| 変換ワーカー | 実行中ジョブ数、成功数、失敗数、処理時間、タイムアウト数 |
| RabbitMQ | キュー滞留数、未ack数、再配送数、dead letter queue件数 |
| PostgreSQL | 接続数、クエリ時間、データサイズ |
| Elasticsearch | 検索時間、インデックス更新失敗、インデックスサイズ |
| 書籍ファイル保存領域 | 使用量、増加傾向、残容量、一時ファイル残存 |
| 7-Zip外部プロセス | 終了コード、タイムアウト、標準エラーの要約 |

監視方針、障害対応、Runbookは後続の運用ドキュメントで詳細化する。

## 保守性と拡張性

保守性と拡張性の方針は次のとおり。

- モジュラーモノリス方針に従い、モジュール境界を明確にする。
- API、Worker、ミドルウェアを将来的に別アプリケーションまたは別ホストへ分離できるようにする。
- コントローラ、アプリケーションサービス、ドメイン、リポジトリ、インフラストラクチャの責務を分ける。
- APIモデル、永続化エンティティ、ドメインモデルを概念的に分離する。
- WebP品質値、ワーカー同時実行数、タイムアウト、保存パス、Elasticsearchインデックス名などを設定で変更可能にする。
- PostgreSQLを正本、Elasticsearchを派生データとする責務分離を保つ。

## 見直し条件

次の状況が発生した場合は、このドキュメントの目標値と方針を見直す。

- 検索APIまたは閲覧APIのp95応答時間が目標を継続的に超える。
- キュー滞留が継続し、アップロード後の変換待ち時間が利用上問題になる。
- 30分タイムアウトが頻発する。
- ストレージ使用量の増加が運用上問題になる。
- 同時利用ユーザ数が初期想定を超える。
- 単一ホスト構成でCPU、メモリ、ディスクI/Oが不足する。
- バックアップなし方針のリスク許容が変わる。
- API、Worker、PostgreSQL、Elasticsearch、RabbitMQを分離する必要が出る。

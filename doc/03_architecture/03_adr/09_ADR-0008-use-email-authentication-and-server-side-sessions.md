# ADR-0008: メール認証とサーバ側セッションを採用する

Superseded by: [ADR-0014: Passkey / WebAuthn認証とサーバ側セッションを採用する](15_ADR-0014-use-passkey-webauthn-and-server-side-sessions.md)

## Context

本システムでは、一般ユーザと管理ユーザの認証、会員登録、ログイン、ログアウト、パスワードリセット、退会、停止、ロール変更を扱う。

管理ユーザは書籍アップロード、書籍削除、変換ジョブ再実行、一般ユーザ停止、ロール管理など影響の大きい操作を行うため、パスワードだけに依存しない認証と、必要なタイミングで既存ログイン状態を失効できる仕組みが必要である。

初期運用は単一Linuxホスト上のDocker Compose構成であり、フロントエンドとバックエンドAPIは同一システムとして運用する。初期段階では外部IDプロバイダへの依存を増やさず、アカウント状態、認証チャレンジ、セッション失効をPostgreSQLの正本データと整合させて管理したい。

## Decision

一般ユーザと管理ユーザの初期認証方式は、メールアドレス、パスワード、メールを利用した2段階認証とする。

会員登録時にはメール認証を行い、メール確認が完了するまで通常利用を許可しない。ログイン時には、パスワード認証成功後にメール2段階認証を要求する。パスワードリセットもメールを利用して提供する。

ログイン状態は、HTTP onlyかつSecure属性付きCookieによるサーバ側セッションを第一候補とする。JWTは、フロントエンドとAPIの分離要件が強くなった場合に再検討する。

認証トークン、ログイン2段階認証チャレンジ、パスワードリセットトークン、セッションの正本はPostgreSQLに置く。平文トークン、ワンタイムコード、セッションIDは保存せず、用途別のハッシュ値、有効期限、使用済み日時、失効日時、試行回数、再送回数を保持する。

退会、停止、パスワード変更、メールアドレス変更、ログアウト、管理ユーザ削除、重要なロール変更時には、関連する未使用トークン、未使用チャレンジ、既存セッションを失効する。

## Consequences

- メール確認により、登録メールアドレスの到達性を確認してから通常利用を許可できる。
- ログイン時にメール2段階認証を要求することで、パスワード漏えい時の影響を抑えやすい。
- サーバ側セッションにより、ログアウト、退会、停止、ロール変更時に既存セッションを即時失効しやすい。
- 認証トークンとセッションの正本をPostgreSQLに置くため、監査、失効、再送制限、試行回数制限を一貫して扱いやすい。
- メール送信基盤が必要になり、メール遅延、送信失敗、再送制限、到達不能時のユーザ体験を設計する必要がある。
- Cookieを利用するため、SameSite属性、Secure属性、CSRF対策、CORS設定をAPI契約とセキュリティ設計で継続して確認する必要がある。
- セッションテーブルの肥大化を防ぐため、期限切れセッション、使用済みトークン、失効済みチャレンジのクリーンアップ方針が必要になる。

## Alternatives

- パスワードのみでログインする。
  - 実装は単純になるが、管理操作の影響が大きく、パスワード漏えい時の防御が弱いため採用しない。
- JWTをアクセストークンとして採用する。
  - API分離やステートレス性には利点があるが、退会、停止、ロール変更、ログアウト時の即時失効に追加設計が必要になる。初期の単一システム構成ではサーバ側セッションを優先する。
- 外部IDプロバイダを必須にする。
  - 多要素認証やアカウント管理を委譲できるが、初期運用の依存先と設定が増えるため採用しない。将来、利用者規模やセキュリティ要件が変わった場合に再検討する。
- メールではなく認証アプリによるTOTPを初期採用する。
  - セキュリティ面の利点はあるが、初期ユーザ体験とサポートの単純さを優先し、メール2段階認証を初期方針とする。

# ADR-0001: Spring BootをバックエンドAPIと変換ワーカーに採用する

## Status

Accepted

## Context

本システムは、認証、認可、入力検証、書籍管理、変換ジョブ管理、PostgreSQLアクセス、Elasticsearch連携、外部プロセス呼び出しを扱う。

バックエンドAPIと変換ワーカーは責務が異なるが、どちらも設定管理、ログ、監視、テスト、DBアクセス、ジョブ状態更新を一貫して扱う必要がある。

また、Java 25を実行基盤とし、小から中規模のWebアプリケーションとして開発速度と保守性を両立する必要がある。

## Decision

バックエンドAPIと変換ワーカーにはSpring Boot 4.0.6を採用する。

HTTP API、認証、バリデーション、トランザクション、DBアクセス、設定管理、監視、テスト支援はSpring Bootと周辺エコシステムを前提に設計する。

ビジネスロジックはコントローラやインフラストラクチャに置かず、アプリケーションサービスまたはドメイン層へ配置する。

## Consequences

- APIと変換ワーカーで技術基盤、設定、ログ、監視、テスト方針を揃えやすい。
- Spring Bootの標準機能を使い、認証、入力検証、DBアクセス、運用監視を統合しやすい。
- Java 25とSpring Bootの組み合わせにより、型安全性と長期保守性を確保しやすい。
- フレームワーク機能に寄せすぎるとドメイン層がSpringへ依存しやすくなるため、層の責務分離を継続して確認する必要がある。
- 起動時間やメモリ使用量は軽量フレームワークより大きくなる可能性があるため、必要に応じてJVM設定やアプリケーション構成を調整する。

## Alternatives

- Node.js / TypeScriptでバックエンドを実装する。
  - フロントエンドと技術を揃えられるが、トランザクション、バッチ、外部プロセス、長時間ジョブの実装基盤を別途慎重に整える必要がある。
- Goでバックエンドを実装する。
  - 軽量で性能面の利点があるが、チームの習熟性と業務アプリ向けの統合機能を優先し、今回は採用しない。
- APIとワーカーで異なる技術スタックを採用する。
  - 各処理に最適化しやすいが、初期段階では運用、テスト、保守の複雑さが増えるため採用しない。

# ADR-0007: RabbitMQと変換ワーカーで画像変換を非同期実行する

## Status

Accepted

## Context

本システムでは、管理ユーザがアップロードしたzip / rar / 7zip形式の自炊本アーカイブを展開し、ページ画像をWebPへ変換し、サムネイルを生成する。

アーカイブ展開、画像変換、サムネイル生成は、CPU、メモリ、ディスクI/O、一時ディスク使用量、7-Zip外部プロセス実行を伴う重い処理である。これをHTTPリクエスト内で同期実行すると、アップロード後の待ち時間が長くなり、APIの応答性、タイムアウト制御、障害分離が難しくなる。

一方で、PostgreSQLは書籍メタ情報、ページ情報、変換ジョブ状態の正本であり、ジョブ状態の一貫性を管理する必要がある。ただし、PostgreSQLをジョブ配送のためのキューとして強く使うと、ロック、ポーリング、再試行、滞留監視、将来のワーカー水平スケールがデータベース負荷に結び付きやすい。

初期運用は単一Linuxホスト上のDocker Compose構成とするが、将来的にバックエンドAPI、変換ワーカー、ミドルウェアを別プロセス、別コンテナ、別ホストへ分離できる余地を残す必要がある。

## Decision

画像変換は、RabbitMQとSpring Boot変換ワーカーによる非同期ジョブとして実行する。

バックエンドAPIは、アップロードされた原本ファイルを保存し、PostgreSQLへ書籍情報と変換ジョブ状態を記録し、RabbitMQへ変換ジョブメッセージを投入する。HTTPレスポンスでは変換完了を待たず、受付結果とジョブを追跡できる情報を返す。

変換ワーカーは、RabbitMQからジョブメッセージを取得し、ジョブごとの専用作業ディレクトリでアーカイブ展開、WebP変換、サムネイル生成を実行する。処理の開始、進行、成功、失敗、タイムアウトなどはPostgreSQLの変換ジョブ状態へ反映する。

PostgreSQLはジョブ状態と生成物管理情報の正本として使い、ジョブ配送そのものはRabbitMQに任せる。これにより、DB依存を排除する方針は「DBをキューとして使わない」という意味で適用し、正本データの保存責務はPostgreSQLに残す。

失敗したジョブは、失敗理由、失敗した工程、診断に必要な情報を変換ジョブ状態へ記録する。再実行は、管理操作または運用手順から新しいキューメッセージを投入し、PostgreSQL上の正本状態と照合して実行する。

RabbitMQとの連携にはSpring AMQPを使用する。変換ジョブメッセージには`conversionJobId`を含め、必要に応じて要求日時、相関ID、再実行元ジョブIDを含める。メッセージに書籍メタ情報、内部物理パス、秘密情報は含めない。

配送保証はat-least-onceを前提とする。変換ワーカーはメッセージを受け取った後、PostgreSQLの`conversion_job`を確認し、処理可能な状態である場合だけ処理する。重複配送、ワーカー再起動、ack前クラッシュが起きても、PostgreSQLの状態遷移と生成物差し替え方針で冪等に扱う。

変換ワーカーは、PostgreSQL更新と生成物保存が成功した後にメッセージをackする。処理失敗時は失敗理由をPostgreSQLへ記録し、RabbitMQの再配送またはdead letter queueで運用確認できる状態にする。再試行回数、dead letter exchange、キュー名、exchange名、routing keyは画像変換設計とRunbookで管理する。

変換ワーカーの同時実行数は既定で10とし、application.propertiesで変更可能にする。1ジョブのタイムアウトは30分を基本とする。

将来スケールが必要になった場合は、APIと変換ワーカーを別ホストへ分離し、変換ワーカーの数や同時実行数を増やす。RabbitMQのキュー滞留数、未ack数、再配送数、dead letter queue件数、変換処理時間を監視し、PostgreSQLとファイル保存領域の整合性を基準に復旧できる構成を保つ。

## Consequences

- アップロード要求は変換完了を待たないため、APIの応答性を保ちやすい。
- 重い変換処理を変換ワーカーへ隔離でき、API、閲覧、管理操作への影響を抑えやすい。
- RabbitMQにより、ジョブ滞留、再試行、デッドレター、ワーカー増減を扱いやすくなる。
- PostgreSQLをキューとして使わないため、正本データ管理とジョブ配送の責務を分離できる。
- PostgreSQLにはジョブ状態を記録するため、利用者や管理者は成功、失敗、処理中、再実行対象を追跡できる。
- キューとPostgreSQLの両方を扱うため、ジョブ投入失敗、重複配送、ワーカー停止、処理途中失敗などの整合性回復を設計する必要がある。
- 変換ワーカーの同時実行数10は初期値であり、ホストのCPU、メモリ、ディスクI/O、一時ディスク使用量に応じて調整が必要になる。
- RabbitMQのexchange、queue、routing key、再試行ポリシー、dead letter exchange、キャンセル、優先度、冪等性の詳細は後続設計で明確化する必要がある。

## Alternatives

- HTTPリクエスト内で同期変換する。
  - 実装は単純に見えるが、アップロード後の待ち時間が長くなり、APIタイムアウト、外部プロセス失敗、リソース競合をHTTP処理に持ち込むため採用しない。
- PostgreSQLをキューとして利用する。
  - ミドルウェアを減らせるが、ポーリング、ロック、滞留監視、再試行制御、ワーカー水平スケールがDB負荷と結び付きやすいため採用しない。
- Redis Streamsを利用する。
  - 軽量でconsumer groupとackを扱えるが、dead letter queue、遅延再試行、運用監視を自前で設計する範囲がRabbitMQより大きいため採用しない。
- Apache Kafkaを利用する。
  - 大量イベントやストリーム処理には強いが、長時間の画像変換ジョブ配送には構成と運用が重く、初期の単一ホスト構成には過剰なため採用しない。
- ActiveMQ Artemisを利用する。
  - JMS系として有力だが、Spring Bootで単純なワークキューを構成する初期方針では、RabbitMQとSpring AMQPの方が情報量と運用の単純さで優位と判断する。
- NATS JetStreamを利用する。
  - 軽量で永続化とackを扱えるが、Spring Bootでの標準的な採用例と運用知見はRabbitMQより少ないため初期採用は見送る。
- APIプロセス内のインメモリキューで処理する。
  - 初期実装は軽いが、プロセス再起動でジョブを失いやすく、APIと変換負荷を分離できず、将来スケールにも不向きなため採用しない。
- OS cronや定期バッチで未処理ジョブを処理する。
  - キュー製品は不要になるが、即時性、同時実行制御、失敗時の再実行、滞留監視が粗くなりやすいため採用しない。
- 外部SaaSのジョブ基盤を利用する。
  - 運用機能は豊富だが、初期方針は単一Linuxホスト上のDocker Compose構成であり、依存と運用コストが増えるため採用しない。

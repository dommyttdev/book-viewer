# ADR-0002: モジュラーモノリス構成を採用する

## Status

Accepted

## Context

本システムは小から中規模のWebアプリケーションとして開始する。

主な責務には、書籍カタログ、ファイル取込、アーカイブ展開、画像変換、変換ジョブ管理、検索インデックス、閲覧、お気に入り、ユーザ、認証、権限、管理機能がある。

初期運用は単一Linuxホスト上のDocker Compose構成とする。一方で、画像変換、検索、APIは負荷特性が異なるため、将来的にAPI、Worker、PostgreSQL、Elasticsearch、RabbitMQを分離できる余地を残す必要がある。

## Decision

初期アーキテクチャはモジュラーモノリス構成とする。

Spring Bootアプリケーション内では、ドメインやユースケースの責務に沿ってモジュール境界を明確にする。APIと変換ワーカーは初期から責務を分け、将来別アプリケーションまたは別ホストへ分離可能な構成を保つ。

モジュール間では、実装詳細や永続化エンティティを不用意に共有しない。ユースケース層、ドメイン層、リポジトリ境界、インフラストラクチャの責務分離を守る。

## Consequences

- 初期開発、ローカル実行、デプロイ、運用を単純に保てる。
- モジュール境界を明確にすれば、責務の混在を抑えながら開発できる。
- APIとWorkerの分離余地を残すことで、変換負荷が増えた場合にも段階的に構成を変えやすい。
- 厳密な境界を意識しないと、単一アプリ内で依存が広がりやすい。
- 将来サービス分割する場合は、トランザクション境界、イベント連携、データ所有権を改めて整理する必要がある。

## Alternatives

- 初期からマイクロサービス構成にする。
  - 責務分離は明確になるが、デプロイ、監視、通信、障害対応、データ整合性の複雑さが初期規模に対して大きい。
- レイヤのみで分ける単純なモノリスにする。
  - 実装は簡単だが、書籍、変換、検索、認証などのドメイン境界が曖昧になりやすく、将来分離が難しくなる。
- APIとWorkerを完全に別プロジェクトとして開始する。
  - 責務分離はしやすいが、初期の開発速度と運用の単純さを優先し、まずは分離可能な境界を保つ方針とする。

# ADR-0014: Passkey / WebAuthn認証とサーバ側セッションを採用する

## Status

Accepted

## Context

本システムでは、一般ユーザと管理ユーザの認証、会員登録、ログイン、ログアウト、アカウント復旧、退会、停止、ロール変更を扱う。

ADR-0008では、メールアドレス、パスワード、メール2段階認証、パスワードリセットを初期認証方式として採用していた。しかし、パスワード保存、パスワードリセット、ログイン時メール2段階認証を前提にすると、認証秘密情報、復旧フロー、メール到達性、ログ安全性、アカウント乗っ取り対策の設計範囲が広がる。

今後はパスワードを保持しない方式を前提にし、フィッシング耐性とユーザ体験を高めるため、一般ユーザと管理ユーザの認証方式をPasskey / WebAuthn方式へ統一する。

## Decision

一般ユーザと管理ユーザの認証方式は、Passkey / WebAuthn方式を採用する。

会員登録ではメールアドレスの到達性を確認したうえで、Passkey credentialを登録する。ログインでは、Passkey / WebAuthn認証に成功した主体へサーバ側セッションを発行する。

パスワードは保存しない。ログイン時メール2段階認証とパスワードリセットは採用しない。Passkeyを使えない場合は、パスワードリセットではなくアカウント復旧として別途設計する。

WebAuthnの登録 / 認証チャレンジ、credential ID、公開鍵、sign count、RP ID、origin、user verification、credential状態、セッション失効条件はPostgreSQLを正本として管理する。Elasticsearchには認証情報を持たせない。

ログイン状態は、HTTP onlyかつSecure属性付きCookieによるサーバ側セッションを採用する。JWTは、フロントエンドとAPIの分離要件が強くなった場合に再検討する。

初期MVPでは、メールアドレスなどの識別子入力後にPasskey認証へ進む識別子優先フローを第一候補とする。完全なユーザー名なしログインは、UXと実装複雑度を確認したうえで後続改善として扱う。

## Consequences

- パスワード保存とパスワードリセットを前提から外せる。
- Passkey / WebAuthnにより、パスワード漏えいやフィッシングのリスクを抑えやすい。
- サーバ側セッションにより、ログアウト、退会、停止、ロール変更、credential無効化、復旧完了時に既存セッションを失効しやすい。
- WebAuthn credential、登録チャレンジ、認証チャレンジ、復旧チャレンジ、セッションの状態管理が必要になる。
- Passkey紛失、端末移行、同期不可、credential削除に備えたアカウント復旧フローが必要になる。
- 初期super_adminのcredential登録手順と、管理ユーザのcredential喪失時の緊急復旧手順を別途決める必要がある。
- RP ID、origin、HTTPS / local開発時の設定、対応ブラウザ、attestation扱い、user verification要件を明示する必要がある。

## Alternatives

- ADR-0008のメールアドレス、パスワード、メール2段階認証、パスワードリセットを継続する。
  - 実装前提は既存ドキュメントと一致するが、パスワード保存と復旧フローのリスクが残るため採用しない。
- メールマジックリンク方式を採用する。
  - パスワードレスにはできるが、メールアカウントへの依存が強く、メール遅延や到達不能時のログイン体験が悪化しやすいため採用しない。
- メールワンタイムコード方式を採用する。
  - パスワードレスにはできるが、フィッシング耐性とユーザ体験の面でPasskey / WebAuthnを優先する。
- 外部IDプロバイダを必須にする。
  - 認証と多要素認証を委譲できるが、初期運用の依存先と設定が増えるため採用しない。将来、利用者規模やセキュリティ要件が変わった場合に再検討する。

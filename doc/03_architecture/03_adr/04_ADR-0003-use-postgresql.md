# ADR-0003: PostgreSQLを正本データストアとして採用する

## Status

Accepted

## Context

本システムでは、ユーザ、管理ユーザ、ロール、権限、書籍メタ情報、著者、タグ、シリーズ、原本ファイル管理情報、ページ情報、変換ジョブ状態、閲覧履歴、お気に入りを扱う。

これらのデータは関連が多く、整合性、トランザクション、制約、検索インデックス更新状態との対応を管理する必要がある。

Elasticsearchや変換済みWebP、サムネイルは再構築可能な派生データとして扱うため、業務上の正本を明確にする必要がある。

## Decision

PostgreSQLを正本データストアとして採用する。

メタ情報、ユーザ、権限、ジョブ状態、閲覧履歴、お気に入り、検索インデックス更新状態はPostgreSQLで管理する。

Elasticsearchは検索用の派生インデックスとし、PostgreSQLから再構築可能にする。ファイル保存領域との整合性確認もPostgreSQLを基準に行う。

## Consequences

- リレーショナルな関連と制約を使い、書籍、著者、タグ、シリーズ、ユーザ、ロール、ジョブ状態を一貫して管理しやすい。
- トランザクションにより、アップロード受付、メタ情報更新、ジョブ状態更新の整合性を守りやすい。
- PostgreSQLを正本とすることで、Elasticsearchや生成ファイルの不整合から回復する基準を持てる。
- 大量検索や日本語のあいまい検索はPostgreSQL単体では要件を満たしにくいため、Elasticsearchと組み合わせる。
- ファイル実体はPostgreSQLに保存せず、ファイル保存領域と管理情報の整合性を設計で扱う必要がある。

## Alternatives

- MySQLを採用する。
  - 一般的なRDBMSとして十分利用できるが、今回はPostgreSQLの制約表現、拡張性、運用経験を優先する。
- Elasticsearchを正本として扱う。
  - 検索は高速になるが、トランザクション、関連管理、権限、ジョブ状態の正確性を担保しにくいため採用しない。
- ファイルやJSON中心の保存方式にする。
  - 初期実装は単純に見えるが、権限、タグ、シリーズ、履歴、ジョブ状態の関連管理が複雑になるため採用しない。

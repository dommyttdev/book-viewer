# ADR-0012: 初期本番運用を単一Linuxホスト上のDocker Compose構成にする

## Status

Accepted

## Context

本システムは、小から中規模の自炊本閲覧Webアプリケーションとして開始する。初期構成では、Next.jsフロントエンド、Spring BootバックエンドAPI、Spring Boot変換ワーカー、PostgreSQL、Elasticsearch、RabbitMQ、書籍ファイル保存領域を組み合わせる。

PostgreSQLはメタ情報、ユーザ、権限、ジョブ状態、閲覧履歴などの正本を保持する。ElasticsearchはPostgreSQLから再構築可能な検索用派生インデックスである。RabbitMQは変換ジョブや検索更新再試行通知の配送を担当する。書籍ファイル保存領域には、原本ファイル、変換済みWebP、サムネイルを保存する。

初期段階では、開発と運用の単純さ、構成の見通し、ローカル開発環境との近さを優先する。一方で、画像変換はCPU、メモリ、ディスクI/O、外部プロセス実行の負荷が大きく、検索やミドルウェアも将来的に独立したリソースが必要になる可能性がある。

そのため、初期は単一ホストで始めつつ、API、Worker、PostgreSQL、Elasticsearch、RabbitMQ、書籍ファイル保存領域を将来的に別ホストまたは別運用基盤へ分離できる責務境界を保つ必要がある。

## Decision

初期本番運用環境は、単一Linuxホスト上のDocker Compose構成とする。

同一ホスト上に、Next.jsフロントエンドWebアプリケーション、Spring BootバックエンドAPI、Spring Boot変換ワーカー、PostgreSQL、Elasticsearch、RabbitMQ、書籍ファイル保存領域を配置する。7-Zip for Linux コンソール版は変換ワーカーコンテナ内で利用する。

バックエンドAPIと変換ワーカーは別プロセス、別コンテナとして扱う。APIはHTTPリクエスト、認証、認可、入力検証、書籍管理、検索、閲覧、管理操作、ジョブ投入を担当する。変換ワーカーはRabbitMQからジョブを取得し、アーカイブ展開、WebP変換、サムネイル生成、ジョブ状態更新を担当する。

PostgreSQL、Elasticsearch、RabbitMQ、書籍ファイル保存領域は、Docker Composeの永続ボリュームまたはホスト上の永続領域で保持する。ただし、PostgreSQLを正本、Elasticsearchを再構築可能な派生データ、RabbitMQを配送手段として扱う責務分離は維持する。

将来的に負荷、容量、可用性、運用要件が高まった場合は、API、Worker、PostgreSQL、Elasticsearch、RabbitMQ、書籍ファイル保存領域を個別に別ホストまたは別基盤へ移せるよう、接続情報、保存先、キュー名、インデックス名、外部プロセスパスを環境設定で差し替え可能にする。

## Consequences

- 初期本番運用とローカル開発環境の構成差を抑えられる。
- Docker Composeで起動、停止、ログ確認、ミドルウェア構成をまとめて扱いやすい。
- API、Worker、ミドルウェアの責務をコンテナ単位で明確にできる。
- 単一ホスト障害時はアプリケーション全体に影響する。高可用性や自動フェイルオーバーは初期方針に含めない。
- 同一ホスト上で画像変換、検索、DB、キューがリソースを共有するため、CPU、メモリ、ディスクI/O、ストレージ容量の監視と安全上限が必要になる。
- PostgreSQLと原本ファイルは再生成できない正本または重要な保存対象であり、バックアップなし運用のリスクを受け入れる必要がある。
- 将来的な分離を容易にするため、環境依存値やシークレットをハードコードせず、application.propertiesや環境変数で設定する必要がある。

## Alternatives

- Kubernetesを採用する。
  - スケーリング、ローリング更新、自己修復に強いが、初期の小から中規模運用では構成と運用が重くなる。初期段階ではDocker Composeを採用し、必要になった段階で再検討する。
- クラウドマネージドサービスを採用する。
  - PostgreSQL、Elasticsearch、キュー、オブジェクトストレージの運用負荷を下げられるが、外部依存、コスト、ネットワーク、権限設定が増える。初期方針では単一Linuxホストを採用する。
- 複数ホスト構成で開始する。
  - 負荷分離や障害分離には有効だが、初期運用ではネットワーク、デプロイ、監視、障害対応が複雑になる。単一ホストから始め、境界を保って将来分離する。
- Dockerを使わず手動プロセス運用にする。
  - 個別プロセスとして起動できるが、ミドルウェアのバージョン、環境差、起動順、ログ、永続化の管理が散らばるため採用しない。
- PaaSやサーバレス構成にする。
  - WebフロントエンドやAPIには適用余地があるが、変換ワーカーの外部プロセス実行、一時ディスク、書籍ファイル保存、Elasticsearchプラグイン前提と相性を確認する範囲が大きいため初期方針では採用しない。

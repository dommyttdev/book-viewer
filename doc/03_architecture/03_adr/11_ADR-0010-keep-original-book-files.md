# ADR-0010: 原本ファイルを通常運用で保存し続ける

## Status

Accepted

## Context

本システムでは、管理ユーザがアップロードしたzip / rar / 7zip形式の自炊本アーカイブを展開し、閲覧用のWebP画像とサムネイルを生成する。

変換済みWebPとサムネイルは閲覧性能と保存容量のバランスを取るための派生ファイルであり、変換条件、品質値、サムネイルサイズ、対応画像形式、ページ順序の扱いは将来変更される可能性がある。

原本ファイルを削除すると、変換条件の見直し、変換失敗時の再実行、WebPやサムネイルの消失時の再生成ができなくなる。バックアップなし運用では、原本ファイルを失った場合に復元する手段もない。

一方で、原本ファイルを保存し続けるとストレージ使用量は増える。業務上の保存容量上限は設けないが、巨大ファイル、zip bomb、画像デコード爆発、同時実行によるストレージ枯渇を避けるため、アップロード、展開、エントリ数、画像ピクセル数、ジョブ一時領域、同時アップロードには技術的な安全上限が必要である。

## Decision

アップロードされた原本ファイルは、通常運用では保存し続ける。

原本ファイルは書籍ファイル保存領域に保存し、PostgreSQLの`book_file`などのファイル管理情報を正本として扱う。APIレスポンスには内部物理パス、コンテナパス、ホストパス、`storage_key`を直接露出しない。

変換済みWebPとサムネイルは、原本ファイルと変換条件から再生成可能な派生ファイルとして扱う。再変換時は原本ファイルを再利用し、既存生成物を壊さないように一時出力先で生成してから差し替える。

書籍削除時は、PostgreSQL上の論理削除と一覧、検索、閲覧対象からの除外を優先する。物理ファイルの削除は同一トランザクションで即時実行せず、削除ジョブまたは管理コマンドで扱う。

業務上の保存容量上限は設けない。ただし、アップロード1ファイル、展開後総量、アーカイブエントリ数、画像ピクセル数、ジョブ一時領域、同時アップロード数には設定可能な技術的安全上限を設ける。具体値は画像変換設計のリソース制限と設定を正本とする。

## Consequences

- 変換条件や品質設定を見直した場合でも、原本ファイルからWebPとサムネイルを再生成できる。
- 変換済みWebP、サムネイル、ページ生成物に不整合や欠損が発生しても、原本ファイルが残っていれば回復できる。
- WebP品質値、サムネイルサイズ、ページ順序、対応画像形式の改善に対応しやすい。
- 原本ファイルは再生成できないため、バックアップなし運用では特に慎重に扱う必要がある。
- ストレージ使用量が増えるため、容量監視、技術的安全上限、削除ジョブ、未参照ファイル検出、Runbookでの確認手順が必要になる。
- 書籍削除時に即時物理削除しないため、論理削除済みデータと物理ファイルのライフサイクルを設計で明確にする必要がある。

## Alternatives

- 変換完了後に原本ファイルを削除する。
  - ストレージ使用量は抑えられるが、再変換、変換条件変更、生成物欠損時の回復ができなくなるため採用しない。
- 原本ファイルを一定期間だけ保持する。
  - ストレージと再変換可能性の折衷案だが、保持期限、期限切れ通知、期限後の再変換不可の扱いが増える。初期方針では保存し続ける。
- 原本ファイルを外部オブジェクトストレージへ移す。
  - 容量拡張性や耐久性を高められるが、初期の単一Linuxホスト上のDocker Compose構成では外部依存と運用が増えるため採用しない。
- WebPから必要に応じて派生物を再生成する。
  - サムネイル再生成には使える場合があるが、WebPは原本ではなく、画質やページ情報の完全な復元根拠にはならないため採用しない。

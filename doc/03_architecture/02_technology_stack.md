# 技術スタック初版

## 目的

このドキュメントは、自炊本閲覧Webアプリケーションの初期技術スタックと選定理由を整理する。

詳細なAPI契約、データモデル、ファイル保存、画像変換、検索、権限、運用手順は、個別の設計ドキュメントまたはADRで継続的に具体化する。

## 全体方針

本システムは、単一Linuxホスト上のDocker Compose構成で開始する。

初期開発では、Next.jsフロントエンド、Spring BootバックエンドAPI、Spring Boot変換ワーカー、PostgreSQL、Elasticsearch、RabbitMQ、書籍ファイル保存領域を組み合わせる。

アプリケーションは小から中規模を想定し、開発と運用の単純さを優先する。ただし、画像変換、検索、ファイル保存は負荷特性が異なるため、将来的にAPI、Worker、PostgreSQL、Elasticsearch、RabbitMQを分離できる構成を保つ。

## フロントエンド

フロントエンドにはNext.jsを採用する。

Next.jsは、一般ユーザ向けの本一覧、検索、ビューア、お気に入りと、管理ユーザ向けのアップロード、メタ情報編集、ジョブ状態確認、ユーザ管理を提供する。

フロントエンドの責務は画面表示、入力補助、API呼び出し、ユーザ体験上の状態管理に限定する。認証、権限、入力値、アップロードファイル、アーカイブ内容の最終検証はバックエンドAPIで行う。

パフォーマンス上は、次の方針を前提とする。

- 本一覧、検索結果、ページ一覧は大量件数を想定し、ページングまたは分割取得を利用する。
- ビューアでは原本ファイルではなく、変換済みWebP画像を表示する。
- サムネイルを利用し、一覧や検索結果で大きなページ画像を直接読み込まない。
- APIエラー、変換待ち、変換失敗、画像未生成状態を明示的に扱う。

## バックエンドAPI

バックエンドAPIにはSpring Boot 4.0.6を採用する。

バックエンドAPIは、HTTPリクエストを受け付け、認証、認可、入力検証、ユースケース実行、PostgreSQL更新、Elasticsearch検索、ファイル保存領域への参照、RabbitMQへのジョブ投入を担当する。

コントローラにはHTTP入出力の責務のみを置き、ビジネスロジックはアプリケーションサービスまたはドメイン層へ配置する。APIモデル、永続化エンティティ、ドメインモデルは概念的に分離する。

パフォーマンス上は、アーカイブ展開や画像変換を同期HTTPリクエスト内で実行しない。アップロード要求では、原本保存、メタ情報登録、変換ジョブ投入までを行い、重い処理は変換ワーカーへ渡す。

## 変換ワーカー

変換ワーカーにはSpring Boot 4.0.6を採用する。

変換ワーカーはRabbitMQからジョブを取得し、アーカイブ展開、画像変換、サムネイル生成、ジョブ状態更新、必要に応じた検索インデックス更新を行う。

変換ワーカーはAPIリクエストスコープに依存しない。長時間処理、外部プロセス呼び出し、一時ディスク使用、リソース制限、失敗時の状態更新を明示的に扱う。

初期設定の基準は次のとおり。

| 項目 | 初期値 / 方針 |
| --- | --- |
| 同時実行数 | 画像変換設計の安全上限を正本とする |
| 1ジョブのタイムアウト | 画像変換設計の安全上限を正本とする |
| 1ジョブの一時ディスク使用量 | 画像変換設計の安全上限を正本とする |
| 作業ディレクトリ | ジョブごとの専用作業ディレクトリ |
| メモリ / CPU制限 | OSまたはコンテナ側で制御 |
| 設定方法 | application.propertiesで変更可能にする |

## Java

Javaバージョンは25とする。

バックエンドAPIと変換ワーカーの実行基盤を揃えることで、言語仕様、ビルド、テスト、運用時のJVM設定を共通化する。

JavaとSpring Bootを採用することで、入力検証、認証、トランザクション、DBアクセス、監視、設定管理、テスト支援を一貫した仕組みで扱いやすくする。

## PostgreSQL

PostgreSQLは正本データの保存先とする。

PostgreSQLで管理する主なデータは次のとおり。

- ユーザ、管理ユーザ
- ロール、権限
- 書籍メタ情報
- 原本ファイル、変換済み画像、サムネイルの管理情報
- ページ情報
- 変換ジョブ状態
- 閲覧履歴
- お気に入り
- 検索更新Outbox

PostgreSQLを正本とすることで、書籍、著者、タグ、シリーズ、ユーザ、権限、ジョブ状態の関連と整合性をトランザクションで管理する。

Elasticsearchやファイル保存領域の状態と不整合が起きた場合も、PostgreSQLを基準に確認、再実行、再インデックスを行う。

## Elasticsearch

検索にはElasticsearchを採用する。日本語検索と表記揺れ正規化に必要なElasticsearch必須プラグインは「Elasticsearch必須プラグイン」を正本とする。

Elasticsearchはタイトル、著者、タグ、シリーズなどを高速に検索するための派生インデックスとする。PostgreSQLの正本データから再構築可能なものとして扱い、Elasticsearchだけにしか存在しない業務上重要な情報を持たせない。

初期方針は次のとおり。

- 日本語のタイトル、著者、タグ検索には日本語検索用のカスタムアナライザを使用する。
- 全角 / 半角、大小文字、互換文字などの表記揺れ対策としてnormalizerを利用する。
- Docker Compose、ローカル開発環境、本番運用環境では、Elasticsearch必須プラグインを同じ前提で導入する。
- バックエンドAPIの起動時またはインデックス作成前に必須プラグインの存在を確認し、未導入の場合はインデックス作成を失敗させる。
- 補完や部分一致が必要な項目には、別途edge n-gram系フィールドを追加する。
- インデックス更新に失敗した場合はPostgreSQLの検索更新Outboxを再試行待ちにし、RabbitMQで再試行を通知する。
- 書籍単位の再インデックスと全件再インデックスを可能にする。
- 検索結果の権限や表示可否が重要な場合は、必要に応じてPostgreSQLの正本データで確認する。

### Elasticsearch必須プラグイン

Elasticsearchの必須プラグインは、この節を正本とする。検索インデックス設計、環境構築手順、ローカル開発手順、Runbookでは、この節を参照し、必須プラグイン一覧を重複定義しない。

| プラグイン | 用途 |
| --- | --- |
| `analysis-kuromoji` | 日本語の形態素解析、基本形、品詞フィルタなどを利用する。 |
| `analysis-icu` | ICU normalizerで全角 / 半角、大小文字、互換文字などを正規化する。 |

Docker Compose、ローカル開発環境、本番運用環境では、上記の必須プラグインを同じ前提で導入する。バックエンドAPIの起動時またはインデックス作成前に必須プラグインの存在を確認し、未導入の場合はインデックス作成を失敗させる。

## ファイル保存

書籍ファイル保存領域には、原本ファイル、変換済みWebP画像、サムネイルを保存する。

初期方針は次のとおり。

| 種別 | 保存方針 |
| --- | --- |
| 原本ファイル | 保存し続ける。 |
| 変換済みWebP | 閲覧用画像として保存する。 |
| サムネイル | 一覧、検索結果、詳細画面用に保存する。 |

ファイルパスは内部実装の詳細として扱い、APIレスポンスへ不用意に露出しない。保存先、命名規則、削除タイミング、バックアップなし運用との関係は、ファイル保存設計で詳細化する。

アップロードファイル、アーカイブ内エントリ、展開先パス、生成ファイル名は、パストラバーサルや不正なファイル操作を防ぐためサーバ側で検証する。

業務上の保存容量上限と技術的な安全上限は分けて扱う。原本ファイルは通常運用では保存し続けるが、アップロード1ファイル、展開後総量、アーカイブエントリ数、画像ピクセル数、ジョブ一時領域、同時アップロード数には設定可能な安全上限を設ける。具体値は [画像変換設計のリソース制限と設定](../04_design/07_image_conversion_design.md#リソース制限と設定) を正本とする。

## 画像変換

画像変換では、原本アーカイブから取り出した画像をWeb閲覧向けのWebPへ変換する。

初期方針は次のとおり。

| 項目 | 方針 |
| --- | --- |
| 出力形式 | WebP |
| WebP品質値 | 80 |
| サムネイル | 生成する |
| 主な閲覧対象 | スマートフォンでの閲覧 |
| 拡大表示 | 初期想定には含めない |
| 1画像の最大ピクセル数 | 画像変換設計の安全上限を正本とする |
| 設定方法 | application.propertiesで変更可能にする |

WebP品質値80は、漫画や小説の本文、セリフの可読性を確保しつつ、保存容量と表示速度のバランスを取るための初期値とする。過度な高品質設定は初期方針としない。

最大幅、最大高さ、透過画像、縦長画像、変換後ファイルサイズ目安などは、画像変換設計で具体化する。

## アーカイブ展開

アーカイブ展開には7-Zip for Linux コンソール版を使用する。

zip / rar / 7zip 形式に対応するため、Java標準ライブラリやApache Commons Compressのrar対応には依存しない。Spring Boot変換ワーカーから外部プロセスとして呼び出す。

初期方針は次のとおり。

- 7-Zip for Linux コンソール版は変換ワーカーコンテナ内で利用する。
- 展開はジョブごとの専用作業ディレクトリで行う。
- 外部プロセスにはタイムアウトを設定する。
- 展開後総量とアーカイブエントリ数には、画像変換設計で定義する安全上限を適用する。
- 終了コード、標準出力、標準エラーを確認し、失敗時は変換ジョブ状態に反映する。
- アーカイブ内パスと展開先パスを検証し、パストラバーサルを防ぐ。
- 暗号化アーカイブ、破損アーカイブ、非対応形式の扱いは画像変換設計で具体化する。

## 非同期ジョブとRabbitMQ

非同期変換ジョブの専用キュー製品にはRabbitMQを採用する。

変換処理はCPU、メモリ、ディスクI/O、外部プロセス実行の負荷が大きく、HTTPリクエスト内で処理すると応答時間と安定性に影響する。そのため、バックエンドAPIはジョブ投入までを担当し、変換ワーカーがキューからジョブを取得して処理する。

RabbitMQを採用する理由は次のとおり。

- APIと変換ワーカーを疎結合にできる。
- Spring Boot / Spring AMQPとの連携がしやすい。
- ack、再配送、dead letter exchangeを利用して、ジョブ配送の失敗を扱いやすい。
- 変換ワーカーの同時実行数やリソース制限をAPIとは別に制御できる。
- 将来的にAPIとWorkerを別アプリケーション、別ホストへ分離しやすい。
- ジョブ滞留、失敗、再実行を運用上観測しやすくできる。
- DBをキューとして強く使う設計を避け、正本データ管理とジョブ配送の責務を分けられる。

RabbitMQはジョブ配送を担当し、業務上のジョブ状態はPostgreSQLの`conversion_job`を正本とする。配送保証はat-least-onceを前提とし、変換ワーカーは`conversionJobId`を受け取った後にPostgreSQLの状態を確認して冪等に処理する。

初期方針は次のとおり。

| 項目 | 方針 |
| --- | --- |
| キュー製品 | RabbitMQ |
| Spring連携 | Spring AMQPを使用する |
| メッセージ内容 | `conversionJobId`を中心に、必要に応じて要求日時や相関IDを含める |
| 配送保証 | at-least-once |
| 成功時 | PostgreSQL更新と生成物保存が完了した後にackする |
| 失敗時 | 失敗理由をPostgreSQLへ記録し、再試行上限後にdead letter queueへ送る |
| 重複配送 | PostgreSQLの状態遷移とジョブIDで冪等に扱う |

## Docker Compose

ローカル開発環境と本番運用環境はDocker Composeを前提とする。

本番運用は単一Linuxホスト上に、Next.js、Spring Boot API、Spring Boot変換ワーカー、PostgreSQL、Elasticsearch、RabbitMQ、書籍ファイル保存領域を配置する。

Docker Composeを採用する理由は次のとおり。

- 小から中規模の初期運用で構成を単純に保てる。
- 開発環境と本番運用環境の差異を減らせる。
- PostgreSQL、Elasticsearch、RabbitMQなどのミドルウェアをまとめて起動、停止しやすい。
- 将来的にコンポーネント単位で別ホストへ分離する前段階として、責務境界を保ちやすい。

## パフォーマンス上の選定理由

本システムでは、閲覧時の応答性と変換処理の安定性を分けて考える。

| 課題 | 技術選定による対応 |
| --- | --- |
| 画像変換が重くHTTP応答を阻害しやすい | RabbitMQと変換ワーカーで非同期処理にする。 |
| 大量ページ画像をそのまま配信すると表示が重い | WebP変換とサムネイル生成を行う。 |
| 日本語のタイトル、著者、タグ検索を高速化し、表記揺れを吸収したい | Elasticsearchを使用する。必須プラグインは「Elasticsearch必須プラグイン」を正本とする。 |
| メタ情報、権限、ジョブ状態の整合性を守りたい | PostgreSQLを正本とし、トランザクションで管理する。 |
| 初期運用を簡単にしつつ将来分離したい | Docker Composeとモジュラーモノリス方針を組み合わせる。 |
| rar / 7zip 対応を安定させたい | 7-Zip for Linux コンソール版を外部プロセスとして使用する。 |

特に、PostgreSQLを正本、Elasticsearchを再構築可能な派生データとすることで、検索性能とデータ整合性の責務を分ける。変換ワーカーをAPIから分離可能にすることで、変換負荷が閲覧や管理操作に与える影響を抑える。

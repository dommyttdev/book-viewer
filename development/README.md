# アジャイル開発成果物

このディレクトリは、自炊本閲覧Webアプリケーションのアジャイル開発成果物を管理する。

`doc/` 配下はプロダクト、バックログ、設計、テスト、運用に関する設計書として扱う。`development/` 配下は、スクラムで実装を進めるための計画、運用ルール、スプリント成果物、TDD実践記録の置き場とする。

## ディレクトリ構成

| ディレクトリ | 目的 |
| --- | --- |
| [scrum/](scrum/) | スクラム運用、プロダクトバックログ、リリース計画、スプリント計画を管理する。 |
| [tdd/](tdd/) | TDD方針、テスト観点、Red-Green-Refactorの進め方を管理する。 |
| [templates/](templates/) | スプリント計画、レビュー、レトロスペクティブ、ユーザーストーリー分解で使うテンプレートを管理する。 |

## 更新ルール

- スプリント開始時に、`scrum/04_sprint_plan.md` またはスプリント別ファイルを更新する。
- スプリントごとの計画、レビュー、レトロスペクティブ、テスト結果は `scrum/sprints/sprint-XX/` に保存する。
- PBIごとのTDD記録は `tdd/logs/pbi-XXX.md` に保存する。
- スプリント中に新しい実装判断、リスク、未完了事項が出た場合は、該当するスクラム成果物へ記録する。
- 仕様、設計、API、データモデル、テスト方針、運用手順に影響する変更は、`doc/` 配下の該当設計書も更新する。
- 実装タスクはTDDを基本とし、テストを書けない理由がある場合は、受入条件または手動確認観点を先に明確にする。

## `doc/` との関係

`development/` は実装計画と開発運用のための成果物であり、設計の正本ではない。

- プロダクト範囲、ユーザーストーリー、受入条件、API、データモデル、権限、運用手順は `doc/` を正とする。
- スプリントで扱う作業の優先順位、TDDの進め方、レビューとふりかえりの記録は `development/` を正とする。
- スプリント中に設計変更が確定した場合は、`development/` の記録だけで終わらせず、必ず `doc/` 側へ反映する。

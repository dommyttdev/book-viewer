# TDD実践方針

## 目的

このドキュメントは、自炊本閲覧Webアプリケーションの実装でTDDをどう使うかを定義する。

TDDはテスト数を増やすためだけではなく、仕様を小さく確認し、責務境界を保ち、リファクタリングしやすい設計へ導くために使う。

## 基本サイクル

1. Red: 期待する振る舞いを表す失敗するテストを書く。
2. Green: テストを通す最小実装を行う。
3. Refactor: 命名、責務、重複、例外処理、ログを整理する。
4. Document: 仕様や設計に影響があれば `doc/` を更新する。

このサイクルをPBI全体ではなく、なるべく小さな振る舞い単位で回す。

## テスト先行の単位

| 対象 | 最初に書くテスト |
| --- | --- |
| ドメインルール | 単体テスト |
| ユースケース | 単体テストまたは軽量な結合テスト |
| API入力検証 | API結合テスト |
| 認証、認可 | API結合テスト |
| PostgreSQL制約 | リポジトリまたは結合テスト |
| Elasticsearch検索 | 検索アダプタの結合テスト |
| ファイル保存 | ファイルシステム境界の単体または結合テスト |
| 7-Zip外部プロセス | アダプタ単体テストと手動または結合確認 |
| Next.js画面 | コンポーネントテスト、E2E、手動確認を組み合わせる |

## バックエンドAPIのTDD方針

優先してテストする。

- 入力検証
- 権限確認
- ユースケースの状態遷移
- PostgreSQLへの永続化
- エラー応答形式
- Elasticsearch更新指示
- ログに秘密情報を出さないこと

コントローラにビジネスロジックを置かず、ユースケースまたはドメインサービスを単体テストしやすい形にする。

## 変換ワーカーのTDD方針

優先してテストする。

- ジョブ状態遷移
- 作業ディレクトリの分離
- アーカイブ内パスの安全性
- 破損ファイル、暗号化ファイル、非対応画像の扱い
- 7-Zip終了コードとタイムアウト
- WebP品質値と設定値の反映
- 失敗時の状態更新

外部プロセスそのものはすべて単体テストで再現しない。コマンド組み立て、終了コード解釈、タイムアウト処理をテストし、実プロセス確認は小さなサンプルで結合または手動確認する。

## フロントエンドのTDD方針

優先してテストする。

- ログイン状態による表示切り替え
- 管理操作の非表示とAPI側エラー表示
- フォーム入力エラー
- 検索条件とAPIリクエスト
- ビューアのページ境界
- お気に入り状態の切り替え

画面の見た目や操作感は、コンポーネントテストだけで完結させず、必要に応じてE2Eと手動確認を組み合わせる。

## テストを書きにくい場合

次のような場合でも、実装前に確認観点を明確にする。

- ブラウザ表示の可読性
- 変換後画像の読みやすさ
- 7-Zip実行環境依存の挙動
- Elasticsearch analyzerの細かな検索結果
- Docker Compose上の複数コンポーネント連携

この場合は、受入条件、手動確認手順、結合確認メモのいずれかを先に書く。

## リファクタリング基準

Green後に次を確認する。

- テスト名が仕様として読める。
- テストが実装詳細に寄りすぎていない。
- ドメインルールがフレームワークやインフラに埋もれていない。
- モックが業務ルールを隠していない。
- エラー処理とログが責務に合っている。
- 同じ仕様が複数テストで過剰に重複していない。

## バグ修正時のTDD

不具合修正では、先に再現テストを書く。

1. 不具合を再現するテストを追加する。
2. テストが失敗することを確認する。
3. 最小修正でテストを通す。
4. 関連する回帰テストを確認する。
5. 必要に応じて `doc/06_testing/03_regression_tests.md` を更新する。

## スプリントレビューで確認すること

- どのテストが仕様を表現しているか。
- 先にテストを書けなかった箇所と理由。
- 次スプリントでテストしやすくするための改善点。
- 手動確認に残っているリスク。

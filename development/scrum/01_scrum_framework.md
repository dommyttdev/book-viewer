# スクラム運用方針

## 目的

このドキュメントは、自炊本閲覧WebアプリケーションをTDDとスクラムで実装するための運用方針を定義する。

スクラムは進捗管理だけでなく、プロダクト価値、技術リスク、品質、学習結果を短いサイクルで検査し、次の実装計画へ反映するために使う。

## 前提

- 1スプリントは1週間を基本とする。
- 各スプリントは、動作する小さなインクリメントを成果とする。
- TDDを基本とし、実装タスクは失敗するテスト、最小実装、リファクタリングの順で進める。
- `doc/` は設計書、`development/` はアジャイル開発成果物として分離する。
- PostgreSQLを正本、Elasticsearchを再構築可能な派生データとして扱う。
- API、変換ワーカー、フロントエンド、ミドルウェアの境界を意識して実装する。

## ロール

| ロール | 主な責務 |
| --- | --- |
| プロダクトオーナー | プロダクト価値、優先順位、受入可否を判断する。 |
| スクラムマスター | スクラムイベントを促進し、障害や過剰な作業範囲を取り除く。 |
| 開発者 | 設計、実装、テスト、ドキュメント更新、レビューを行う。 |
| ステークホルダー | スプリントレビューで成果物を確認し、フィードバックを提供する。 |

個人開発または少人数開発では、同一人物が複数ロールを兼ねてもよい。ただし、優先順位判断、実装判断、品質判断を区別して記録する。

## スクラムイベント

| イベント | 頻度 | 目的 | 成果物 |
| --- | --- | --- | --- |
| スプリントプランニング | 各スプリント開始時 | スプリントゴール、対象PBI、TDD方針、検証範囲を決める。 | スプリントバックログ |
| デイリースクラム | 作業日ごと | 進捗、障害、当日の検証対象を確認する。 | 障害メモ、調整済みタスク |
| バックログリファインメント | 週1回または必要時 | PBIを小さくし、受入条件とテスト観点を明確にする。 | 更新済みPBI |
| スプリントレビュー | 各スプリント終了時 | 動く成果物を確認し、受入可否と次の学びを整理する。 | レビュー記録 |
| レトロスペクティブ | 各スプリント終了時 | 開発プロセス、TDD、レビュー、品質の改善点を決める。 | 改善アクション |

## スプリントゴールの作り方

スプリントゴールは、実装タスクの羅列ではなく、利用者価値または技術リスク低減を表す。

良い例:

- 管理ユーザが自炊本アーカイブをアップロードし、変換ジョブ作成まで確認できる。
- 一般ユーザが検索結果から本を開き、ビューアでページ送りできる。
- 変換ワーカーの危険な入力を拒否し、失敗状態を追跡できる。

避ける例:

- APIを作る。
- DBを作る。
- フロントを進める。

## Definition of Ready

スプリントへ入れるPBIは、原則として次を満たす。

- 利用者または運用者にとっての価値が説明できる。
- 対応する設計書または受入条件が参照できる。
- TDDで最初に書くテスト観点が1つ以上ある。
- 正常系と主要な異常系が整理されている。
- 権限、入力検証、データ永続化、外部依存の影響が確認されている。
- 1スプリント内で完了できる大きさに分割されている。

## Definition of Done

完了判断は [doc/05_development/05_definition_of_done.md](../../doc/05_development/05_definition_of_done.md) を基準とする。

スクラム成果物としては、加えて次を確認する。

- スプリントゴールに対する貢献が説明できる。
- 失敗するテストを先に書いた、または先に書けなかった理由が記録されている。
- 実行したテスト、未実行のテスト、手動確認結果が記録されている。
- 設計変更がある場合、`doc/` 配下の該当文書が更新されている。
- 未完了作業は次スプリントの候補PBIとして分離されている。

## TDDとスクラムの接続

- PBIは、受入条件とテスト観点に分解してからスプリントへ入れる。
- タスクは、なるべく「テスト追加」「最小実装」「リファクタリング」「ドキュメント更新」に分ける。
- レビューでは、動作確認だけでなく、テストが仕様を表現できているかを確認する。
- レトロスペクティブでは、TDDが重すぎた箇所、先にテストを書けなかった箇所、次に小さく切れる箇所を振り返る。

## 指標

初期は見積もり精度よりも、完成した価値と品質の安定を重視する。

| 指標 | 使い方 |
| --- | --- |
| 完了PBI数 | スプリントごとの実装可能な粒度を把握する。 |
| 未完了PBI数 | スプリント計画の過大さを検知する。 |
| テスト先行率 | TDDが実践できているかを確認する。 |
| 不具合再発数 | 回帰テスト不足を検知する。 |
| レビュー差し戻し理由 | Definition of Ready / Doneの改善に使う。 |

ベロシティは3スプリント以上の実績が出てから参考値として扱う。

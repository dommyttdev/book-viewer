# Sprint S0 レトロスペクティブ

## スプリント

- Sprint: S0
- 実施日: 2026-05-15

## よかったこと

- issue #48がPBI-001の親Issueとして整理されており、#81から#88のsub-issueへ分割済みだった。
- #81で先に `apps/`、`libs/`、Gradle Wrapper、npm、API / Worker別プロセス方針を決めたことで、#82以降の作業入口が明確になった。
- `development/` 配下にスクラム運用、TDD方針、スプリント成果物の置き場が定義されていたため、実装前でも作業記録を残しやすかった。
- PBI-001の対象外がissue本文に明記されており、Sprint S0で作り込みすぎない境界を確認できた。
- フロントエンド、API、Worker、ミドルウェアの最小確認を `test-report.md` に集約でき、後続PBIの開始前確認として再利用しやすくなった。
- API / Workerの外部依存なしテストと、local profileでの外部依存疎通確認を分けたことで、普段の確認コマンドを軽く保てた。

## 問題だったこと

- 親Issue #48の受入条件が広く、sub-issueごとの完了条件と記録先を明確にしないと着手順序が曖昧になる。
- ローカル環境では `npm.ps1` の実行ポリシー、Docker設定ファイルのアクセス拒否、Docker daemon未起動、Gradle Wrapper配布物取得時のネットワーク制限など、実行環境差分の影響が出た。
- GitHub Projectの状態更新やissueチェックリスト更新は、リポジトリ内成果物の完了とは別作業として残りやすい。
- スプリント途中で実装済みになった後も、レビュー、レトロスペクティブ、計画文書に古い「未実施」前提が残りやすい。

## 学び

- Sprint S0は、#81で構成を確定してから、#82から#88を小さく完了させる進め方が有効だった。
- フロントエンド、API、Worker、ミドルウェアを並行に作る前に、ディレクトリ構成、ビルドツール、起動単位を固定する必要がある。
- Elasticsearch必須プラグイン、RabbitMQ配送詳細、7-Zip実行、WebP変換は、それぞれ後続PBIとの責務境界を明示して進める必要がある。
- 確認結果は `test-report.md` に集約しつつ、各issue文書のチェックリストも同時に更新しないと整合が崩れる。

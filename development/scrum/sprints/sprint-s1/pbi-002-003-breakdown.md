# PBI-002 / PBI-003 ユーザーストーリー分解

## ストーリー

| PBI | GitHub Issue | ユーザ | 目的 | 価値 |
| --- | --- | --- | --- | --- |
| PBI-002 | #49 | 一般ユーザ | メール確認後に会員登録を完了する。 | 登録メールアドレスの到達性を確認したうえで利用を開始できる。 |
| PBI-003 | #50 | 利用者 | メール2段階認証でログインし、安全なセッションを取得する。 | パスワードだけに依存せず、ログイン状態をサーバ側で失効できる。 |

## 前提

- #45で認証トークン、ログインチャレンジ、セッションの実装詳細を確定してから実装する。
- PBI-002とPBI-003はSprint S1の主導線として扱う。
- パスワードリセットはPBI-004 / Sprint S2で扱うが、トークン方針は#45で共通化する。
- 管理ユーザログインはPBI-003のセッション分離観点として考慮するが、管理ロール認可の本実装はSprint S2で扱う。

## PBI-002 受入条件ドラフト

- [ ] メールアドレス、パスワード、表示名を入力して一般ユーザ登録を要求できる。
- [ ] メールアドレス形式が不正な場合は登録できない。
- [ ] パスワードが方針を満たさない場合は登録できない。
- [ ] 有効な一般ユーザのメールアドレスと重複する場合は登録できない。
- [ ] 登録直後の一般ユーザはメール未確認状態で保存される。
- [ ] メール確認用トークンは平文保存されず、ハッシュ値、有効期限、試行回数、再送回数を持つ。
- [ ] メール確認成功時にユーザを利用可能状態へ遷移し、使用したトークンを使用済みにする。
- [ ] 期限切れ、使用済み、失効済み、試行回数超過のメール確認トークンは拒否される。
- [ ] 開発環境では送信予定のメール確認リンクまたはトークンを安全な方法で確認できる。
- [ ] レスポンス、ログ、エラーにパスワード、平文トークン、トークンハッシュを出力しない。

## PBI-003 受入条件ドラフト

- [ ] メール確認済みの一般ユーザはメールアドレスとパスワードでログイン開始できる。
- [ ] メール未確認、停止、退会済みの一般ユーザはログイン完了できない。
- [ ] パスワード認証成功後、ログイン2段階認証チャレンジを作成し、`challengeId`を返す。
- [ ] ワンタイムコードは平文保存されず、ハッシュ値、有効期限、試行回数、再送回数を持つ。
- [ ] 期限切れ、使用済み、失効済み、試行回数超過のチャレンジは拒否される。
- [ ] 2段階認証成功時に一般ユーザ用セッションを作成し、HTTP only Cookieを発行する。
- [ ] セッションCookieの平文値はDBへ保存せず、ハッシュ値で照合する。
- [ ] ログアウト時に現在のセッションを失効できる。
- [ ] 失効済み、期限切れ、主体が停止または退会済みのセッションでは認証できない。
- [ ] 一般ユーザ用セッションは管理ユーザ用セッションとして扱われない。
- [ ] レスポンス、ログ、エラーにパスワード、ワンタイムコード、セッションID、ハッシュ値を出力しない。

## TDD開始観点

| PBI | Red候補 | 期待する失敗 |
| --- | --- | --- |
| PBI-002 | 登録要求が有効なメールとパスワードを受け取り、未確認ユーザと確認トークンを作る。 | まだ永続化モデルやユースケースがなく失敗する。 |
| PBI-002 | 同じメールアドレスの有効ユーザがいる場合は登録できない。 | 重複判定が未実装で失敗する。 |
| PBI-002 | メール確認トークンは期限切れ、使用済み、試行回数超過を拒否する。 | トークン状態判定が未実装で失敗する。 |
| PBI-003 | メール未確認ユーザはログインチャレンジを作成できない。 | ユーザ状態判定が未実装で失敗する。 |
| PBI-003 | 正しいコード確認でセッションが発行され、チャレンジが使用済みになる。 | チャレンジ確認とセッション発行が未実装で失敗する。 |
| PBI-003 | 失効済みセッションでは認証できない。 | セッション検証が未実装で失敗する。 |

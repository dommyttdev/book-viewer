# リリース計画

## 目的

このドキュメントは、MVP、Beta、v1.0に向けたスクラム上のリリース計画を定義する。

日付固定の納期計画ではなく、各リリースで何が動けば価値があるか、どの技術リスクをいつ潰すかを明確にする。

## リリース方針

| リリース | ゴール | リリース判断 |
| --- | --- | --- |
| MVP | 管理ユーザが登録した本を一般ユーザが検索して読める。 | アップロード、変換、検索、閲覧、権限制御の主導線が通る。 |
| Beta | 日常利用で困る失敗確認、再実行、検索、閲覧操作を改善する。 | 主要な失敗状態を追跡でき、基本的な運用確認ができる。 |
| v1.0 | 単一Linuxホストで安定して使える正式版にする。 | 主要機能、テスト、Runbook、既知リスクがそろう。 |

## MVPリリーススライス

MVPは、次の順に縦切りで進める。

| 順序 | スライス | 成果 |
| --- | --- | --- |
| 1 | 開発基盤 | Next.js、Spring Boot API、Spring Boot Worker、DB、検索基盤の最小起動。 |
| 2 | 認証と権限 | 一般ユーザ、管理ユーザ、未認証のアクセス制御。 |
| 3 | 書籍アップロード | 管理ユーザがアーカイブをアップロードし、原本保存とジョブ作成ができる。 |
| 4 | 変換 | ワーカーがアーカイブを展開し、WebPとサムネイルを生成できる。 |
| 5 | メタ情報と検索 | 管理ユーザがメタ情報を整え、一般ユーザが検索できる。 |
| 6 | 閲覧 | 一般ユーザがビューアで読み、読みかけ位置とお気に入りを使える。 |
| 7 | 統合確認 | MVP主要フローのE2E、回帰、手動確認を行う。 |

## スプリント見通し

1スプリント1週間を基本に、初期見通しを次のように置く。実績に応じて毎スプリント見直す。

| スプリント | 主なゴール | 対象PBI |
| --- | --- | --- |
| Sprint 0 | 実装基盤とスクラム/TDD運用を整える。 | PBI-001 |
| Sprint 1 | 一般ユーザ登録、メール確認、ログイン2段階認証の最小スライスを作る。 | PBI-002, PBI-003 |
| Sprint 2 | パスワードリセット、初期管理ユーザ、管理ユーザ固定ロール認可を作る。 | PBI-004, PBI-005 |
| Sprint 3 | RabbitMQ基盤、書籍ドメイン、アップロード、原本保存を通す。 | PBI-006, PBI-007, PBI-008, PBI-009 |
| Sprint 4 | zipのみでアップロードからWebP変換までの縦切りを通す。 | PBI-010 |
| Sprint 5 | rar / 7zip、サムネイル、変換ジョブ状態確認を追加する。 | PBI-011, PBI-012 |
| Sprint 6 | メタ情報編集、検索Outbox、基本検索を通す。 | PBI-013, PBI-014, PBI-015 |
| Sprint 7 | 一覧、ビューア、読みかけ位置を通す。 | PBI-016, PBI-017, PBI-018 |

## MVP受入シナリオ

MVPリリース時に、最低限次を確認する。

1. 一般ユーザが登録、メール認証、ログイン時メール2段階認証、ログアウトできる。
2. 一般ユーザがパスワードリセットできる。
3. 初期 `super_admin` または管理ユーザがログインし、自炊本アーカイブをアップロードできる。
4. アップロード後に原本ファイルが保存され、PostgreSQL上の変換ジョブが作成され、RabbitMQへ配送される。
5. 変換ワーカーがWebP画像とサムネイルを生成する。
6. 管理ユーザがタイトル、著者、タグ、シリーズを編集できる。
7. Elasticsearchへ検索インデックスが反映され、PostgreSQLから再構築できる。
8. 一般ユーザが本一覧または検索から本を見つけられる。
9. 一般ユーザがビューアでページを表示し、ページ送りできる。
10. 一般ユーザの読みかけ位置が保存される。
11. 一般ユーザが管理操作とアップロードを実行できない。

## リスクと早期検証

| リスク | 早期検証方法 |
| --- | --- |
| 7-Zip外部プロセスの安全な呼び出し | Sprint 5で終了コード、タイムアウト、パストラバーサルをテストする。 |
| WebP変換品質とスマートフォン閲覧性 | Sprint 4から最小サンプルで手動確認を始める。 |
| ElasticsearchとPostgreSQLの整合性 | Sprint 6でOutbox、全件再インデックス、書籍単位再インデックスの検証を入れる。 |
| 権限漏れ | Sprint 1以降、API側の認可テストを各PBIに含める。 |
| 初期管理ユーザを作れず管理画面へ入れない | Sprint 2で初期 `super_admin` の安全な作成手順と重複防止を確認する。 |
| DBマイグレーション不備で環境差分が増える | Sprint 0からマイグレーション基盤と最小スキーマ確認を入れる。 |
| メール認証や2段階認証が開発環境で確認しづらい | Sprint 0でメール送信境界を抽象化し、Sprint 1から開発用メール確認手段を使う。 |
| E2Eが重くなりすぎる | Sprint 8までは単体、結合を厚くし、E2Eは主導線に絞る。 |

## リリース判定会

各リリース候補では、スプリントレビューとは別に次を確認する。

- 未完了PBIと既知不具合
- セキュリティ、権限、入力検証の未確認箇所
- テスト実行結果と未実行理由
- `doc/` 設計書の更新漏れ
- Runbook、障害時確認、ログ追跡性
- リリースノートに記録する変更点
